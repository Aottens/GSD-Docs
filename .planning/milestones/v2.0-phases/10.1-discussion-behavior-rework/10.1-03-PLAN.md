  <name>Task 1: Verbatim decision extractor service</name>
  <files>
    backend/app/services/decision_extractor.py

  <action>
    Create `backend/app/services/decision_extractor.py` with:

    1. `extract_verbatim_decision(user_message: str, current_topic: str, question_context: str) -> Optional[dict]`:
       - CRITICAL per user decision: "Verbatim/faithful extraction — use the engineer's exact words, trimmed to key sentences. No AI interpretation or distillation."
       - NO LLM in this extraction loop
       - Rule-based extraction:
         a. Strip leading/trailing whitespace
         b. Remove obvious filler words at sentence boundaries ONLY:
            - Start: "um", "uh", "like", "so", "well", "basically", "actually" (followed by comma or space)
            - End: "you know", "I think", "right", "I guess" (preceded by comma or space)
         c. Keep ALL technical content EXACTLY as stated
         d. If message is very short (< 5 words after stripping) and looks like a chip selection (matches a common option), still capture verbatim
       - Returns dict: `{"topic": str, "question": str, "decision": str, "confirmed": False, "timestamp": ISO string}`
       - Returns None if message appears to be a meta-conversation (e.g., "next topic", "meer vragen", "go on")

    2. `is_meta_message(message: str) -> bool`:
       - Detects messages that are conversation control, not answers:
         - "volgende onderwerp" / "next topic"
         - "meer vragen" / "more questions"
         - "ga door" / "go on" / "continue"
         - "bevestig" / "confirm"
         - "ja" / "nee" / "yes" / "no" (single word only — these are check-in responses)
       - Returns True for meta messages (no decision extraction needed)

    3. `trim_to_key_sentences(text: str, max_sentences: int = 5) -> str`:
       - If text has more than max_sentences, keep the ones with the most technical content
       - Technical content heuristic: contains numbers, units (°C, bar, L/min, mm, kW), specific terms (alarm, trip, interlock, valve, sensor, pump)
       - If all sentences seem equally important, keep first max_sentences
       - This preserves the engineer's words but removes excessive filler paragraphs

    This service must NEVER call the LLM. It is pure rule-based extraction to honor the "no interpretation" requirement.
  </action>

  <action>
    Rework `backend/app/services/discussion_engine.py`:

    1. Import and integrate ConversationState, StreamingXMLBuffer, decision_extractor:
       ```python
       from app.services.conversation_state import ConversationState, ConversationPhase, detect_foundation_phase
       from app.services.structured_output_parser import StreamingXMLBuffer, parse_question_card
       from app.services.decision_extractor import extract_verbatim_decision, is_meta_message
       ```

    2. Rework `start_discussion()`:
       - After creating conversation, detect if Foundation phase using `detect_foundation_phase()`
       - If Foundation: set ConversationState with `is_foundation=True`, skip topic selection, go directly to discussion phase
       - If Foundation: create initial assistant message with open-ended intake greeting (using FOUNDATION_INTAKE_PROMPT), NOT topic selection card
       - If NOT Foundation: keep current topic selection behavior (existing code is correct)
       - Store state in conversation.summary_data using `state.to_summary_data()`

    3. Rework `send_message()` to be state-aware:

       a. Load ConversationState from conversation.summary_data: `state = ConversationState.from_summary_data(conversation.summary_data)`

       b. **If state.phase == topic_selection:**
          - Parse user's topic selection from message content (they selected topics from TopicSelectionCard)
          - Call `state.confirm_topics(selected, discretion)`
          - Start first topic: `topic = state.next_topic(); state.start_topic(topic)`
          - Build LLM prompt with topic context and probe questions from gray areas
          - Yield `topic_boundary` event: `{"event": "topic_boundary", "data": {"topic": topic, "status": "starting"}}`

       c. **If state.phase == discussion:**
          - First, extract decision from user's answer using `extract_verbatim_decision()` (unless meta-message)
          - If decision extracted: yield `decision_captured` event with decision data, add to state.decisions
          - Increment question counter: `needs_checkin = state.increment_question()`
          - If needs_checkin (4 questions reached): inject check-in prompt into LLM context
          - Build LLM messages with current state context

       d. **If state.phase == check_in:**
          - Parse user's response: "more questions" -> stay on topic, "next topic" -> complete current, start next
          - If moving to next: yield `topic_boundary` event for current topic (complete), then next topic (starting)
          - If all topics complete: transition to completion

       e. **Stream LLM response with StreamingXMLBuffer:**
          ```python
          buffer = StreamingXMLBuffer()
          async for chunk in self.llm.stream_complete(messages, max_tokens=4096, temperature=0.7):
              buffer.feed(chunk)
              # Extract any structured events
              for event in buffer.extract_events():
                  yield event  # question_card, topic_boundary, completion_signal
              # Yield remaining text as message_delta
              text = buffer.get_text_content()
              if text:
                  yield {"event": "message_delta", "data": {"delta": text}}
          ```

       f. After stream complete: persist assistant message, save updated state to summary_data, commit

    4. Add `_build_state_aware_prompt(state: ConversationState, user_message: str) -> str`:
       - Builds additional context for LLM based on current state
       - Includes: current topic, remaining topics, questions asked so far, check-in instruction if applicable
       - For Foundation phase: includes reference file summaries if available (see step 5)
       - Appends to the conversation message history as a system message injection

    5. Add `_load_reference_files(project_id: int) -> Optional[str]`:
       - Per user decision: "AI reads reference files from the intake folder during discussion"
       - Load uploaded reference files for the project from the file system (using file_service/file_storage patterns from Phase 9)
       - Return a summary string of available files and their content previews (first 500 chars each)
       - Max 3000 chars total to avoid prompt bloat
       - If no files, return None
       - This content gets injected into Foundation phase prompts so AI can ask: "I see from [document] that... is this still current?"

    6. Add topic selection parsing logic for the initial response:
       - `_parse_topic_selection(content: str, available_topics: list[str]) -> tuple[list[str], list[str]]`
       - Parse which topics were selected vs discretion from user message or TopicSelectionCard callback
       - Handle both structured (JSON from TopicSelectionCard) and freeform ("I want to discuss topics 1, 3, 5")

    Update `backend/app/api/discussions.py`:

    7. Update `stream_message` endpoint to handle the new event types from the engine:
       - The engine now yields structured events (question_card, decision_captured, topic_boundary, etc.)
       - These should all be forwarded as SSE events in the same format
       - No changes needed to the SSE format itself — just ensure all event types pass through
