Use the existing SSE infrastructure and frontend components; focus on backend orchestration and LLM prompt engineering.

### Pattern 1: AI-Driven Topic Progression with State Machine

**What:** Conversation state machine tracks current_topic, completed_topics, selected_topics, and drives LLM prompts to maintain flow structure.

**When to use:** AI-led conversations where the LLM decides next questions but backend enforces workflow boundaries (topic selection → discussion → completion).

**Example:**
```python
# Source: AI Agent Orchestration patterns 2026
class ConversationState:
    phase: Literal["topic_selection", "discussion", "completion"]
    current_topic: Optional[str]
    completed_topics: list[str]
    selected_topics: list[str]
    decisions: list[dict]

    def next_prompt(self) -> str:
        if self.phase == "topic_selection":
            return build_topic_selection_prompt()
        elif self.phase == "discussion":
            if not self.current_topic:
                return build_next_topic_prompt(self.selected_topics, self.completed_topics)
            else:
                return build_topic_discussion_prompt(self.current_topic, self.decisions)
        elif self.phase == "completion":
            return build_completion_prompt(self.decisions)
```

**Why it works:** Backend maintains state machine, LLM generates content within current state. Prevents LLM from jumping topics or skipping workflow steps. Matches v1.0 flow: topic selection → 4 questions → check-in → next topic → completion.

### Pattern 2: Structured Output Parsing via XML Tags

**What:** LLM generates XML-tagged output, backend parses and emits SSE events (question_card, summary_card). Claude excels at XML format (Anthropic docs recommend tags like `<question>`, `<options>`, `<summary>`).

**When to use:** When LLM needs to generate structured UI elements (question cards) within a streaming conversation.

**Example:**
```python
# Source: Anthropic structured output docs + llm-xml-parser patterns
import re
from typing import Optional

def parse_question_card(llm_output: str) -> Optional[dict]:
    """Parse XML-tagged question from LLM output."""
    question_match = re.search(r'<question>(.*?)</question>', llm_output, re.DOTALL)
    options_match = re.search(r'<options>(.*?)</options>', llm_output, re.DOTALL)

    if not question_match:
        return None

    question = question_match.group(1).strip()
    options = []
    if options_match:
        # Parse <option>text</option> tags
        options = re.findall(r'<option>(.*?)</option>', options_match.group(1))

    return {
        "question": question,
        "options": [opt.strip() for opt in options]
    }

# System prompt instructs LLM:
# "When asking a question, format it as:
# <question>Question text here</question>
# <options>
#   <option>Option 1</option>
#   <option>Option 2</option>
# </options>
# If no options, omit <options> tag."
```

**Why it works:** Claude trained on XML patterns, reliable parsing even mid-stream. Frontend receives structured question_card events, renders QuestionCard component. Avoids brittle JSON parsing in streaming context.

### Pattern 3: Verbatim Decision Extraction (No Interpretation)

**What:** Extract engineer's exact words from conversation, trim to key sentences, NO summarization or rephrasing. User requirement: "interpretation is killing while writing technical documentation."

**When to use:** Technical documentation capture where exact terminology and phrasing is critical (tolerances, capacities, state names, interlock conditions).

**Example:**
```python
# Source: UX research verbatim transcription patterns 2026
def extract_verbatim_decision(
    user_message: str,
    question_context: str,
    topic: str
) -> dict:
    """
    Extract verbatim decision from user message.

    NO LLM summarization. Use exact words, trim only filler/fluff.

    Args:
        user_message: User's answer text
        question_context: The question that was asked
        topic: Current discussion topic

    Returns:
        Decision dict with verbatim text
    """
    # Simple rule-based extraction:
    # 1. Strip leading/trailing whitespace
    # 2. Remove obvious filler ("um", "uh", "like I said")
    # 3. Keep technical content EXACTLY as stated

    verbatim = user_message.strip()

    # Only remove non-content words at sentence boundaries
    filler_patterns = [r'^(um|uh|like|so|well),?\s+', r',?\s+(you know|I think)\s*$']
    for pattern in filler_patterns:
        verbatim = re.sub(pattern, '', verbatim, flags=re.IGNORECASE)

    return {
        "topic": topic,
        "question": question_context,
        "decision": verbatim,  # EXACT words from engineer
        "timestamp": datetime.utcnow().isoformat()
    }
```

**Why it works:** Preserves engineer's exact technical terminology. Critical for FDS where "85°C trip threshold" vs "approximately 85 degrees shutdown" changes implementation. Avoids AI hallucination/interpretation risk.

**When to use:** Initial project discovery phases where structured questions are premature — need to gather context first.

### Pattern 5: Context Window Management for Multi-Step Workflow

**What:** Clear conversation context between workflow steps (discuss → plan → write). User requirement: "Context rot is killing." New conversation/session for each workflow step.

**When to use:** Multi-step workflows where previous context (long discussion) shouldn't pollute next step (planning requires fresh LLM context with CONTEXT.md as single source).

**Example:**
```python
# Source: Context window management best practices 2026
class WorkflowStepTransition:
    """Manage context boundaries between workflow steps."""

    @staticmethod
    async def complete_discussion_step(conversation_id: int, db: AsyncSession) -> dict:
        """
        Complete discussion, generate CONTEXT.md, close conversation.

        Returns summary for next step — does NOT reuse conversation history.
        """
        # 1. Mark conversation as completed
        conversation = await get_conversation(conversation_id, db)
        conversation.status = ConversationStatus.completed

        # 2. Generate CONTEXT.md from decisions (NOT from full chat history)
        context_md = generate_context_md(conversation.summary_data)

        # 3. Save CONTEXT.md to filesystem
        save_context_file(conversation.project_id, conversation.phase_number, context_md)

        # 4. Return summary for next step
        # NOTE: Next step (planning) reads CONTEXT.md, NOT conversation history
        return {
            "context_file": f"phase-{conversation.phase_number}/CONTEXT.md",
            "decisions_count": len(conversation.summary_data["decisions"]),
            "next_step": "planning"
        }

    @staticmethod
    def should_start_new_session(from_step: str, to_step: str) -> bool:
        """Check if workflow step transition requires new conversation."""
        workflow_boundaries = [
            ("discuss", "plan"),
            ("plan", "write"),
            ("write", "verify")
        ]
        return (from_step, to_step) in workflow_boundaries
```

**Why it works:** Prevents 70-80% context capacity issues (best practice: summarize when hitting this threshold). For workflow transitions, CONTEXT.md is the compressed summary — new LLM session reads only CONTEXT.md, not 1000-line discussion history. Prevents "context rot" from old conversation bleeding into planning.

### Anti-Patterns to Avoid

- **LLM-driven decision summarization:** User explicitly forbids interpretation. Use verbatim extraction, not "LLM reads answer and generates decision summary."
- **Single conversation for all workflow steps:** Leads to context rot. Close discussion conversation, start fresh for planning with CONTEXT.md as input.
- **Brittle JSON parsing in streaming:** Use XML tags for Claude. JSON requires complete object before parsing; XML tags parseable mid-stream.
- **Generic questions:** v1.0 requires functional-spec depth. "Tell me about interlocks" is BAD. "When TT-101 exceeds 85°C, should system (a) alarm, (b) reduce power, (c) trip?" is GOOD.
- **Auto-advancing workflow steps:** User decision: only advance if previous step properly completed AND user confirms.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SSE streaming | Custom EventSource polyfill | Native EventSource API (browser) + sse-starlette (backend) | Auto-reconnect, event parsing, W3C spec compliance. Edge cases (connection drops, CORS) are handled. |
| XML parsing | Regex-only parser | lxml with fallback to regex | Nested tags, CDATA handling, malformed XML recovery. Regex works for simple tags but breaks on edge cases. |
| Conversation state persistence | In-memory state dict | SQLAlchemy Conversation.summary_data (JSON column) | Crash recovery, multi-instance backend support. User can refresh browser and resume. |
| Context window tracking | Manual token counting | LLM provider's token count API | Model-specific tokenization (Claude uses different tokenizer than GPT). Off-by-one errors cause failures. |
| Decision deduplication | String comparison | Semantic similarity (vector embeddings) if needed | "Flow rate 100 L/min" vs "100 liters per minute flow" are same decision. But for Phase 10.1: keep verbatim, let user confirm/edit in UI. |

**Key insight:** SSE and context management have critical edge cases (connection failures, token limit overflow) that production libraries handle correctly. Don't reimplement — use proven tools.

## Common Pitfalls

### Pitfall 1: Context Rot from Conversation History Reuse

**What goes wrong:** Discussion conversation accumulates 1000+ lines. Planning step reuses same conversation, passing all history to LLM. Context window fills with old Q&A, leaving little room for planning logic. Quality degrades.

**Why it happens:** Seems efficient to reuse conversation. Avoids "losing context." But LLM context != conversation history. CONTEXT.md is the distilled context.

**How to avoid:**
1. Close discussion conversation when complete (status = completed)
2. Generate CONTEXT.md from summary_data (NOT from message history)
3. Start NEW conversation for planning step
4. Planning prompt includes CONTEXT.md content, NOT discussion history

**Warning signs:**
- Planning tasks reference specific discussion questions (shouldn't see question text, only decisions)
- Context window errors during planning
- Planning quality decreases for phases with longer discussions

**Reference:** [Context Window Management Best Practices 2026](https://www.getmaxim.ai/articles/context-window-management-strategies-for-long-context-ai-agents-and-chatbots/)

### Pitfall 2: LLM Interpretation of Engineer Answers

**What goes wrong:** Engineer says "Temperature sensor trips at 85 degrees, we want alarm only not shutdown, existing system has 90 degree setpoint." LLM extracts: "Alarm at 85°C." Lost: existing 90° setpoint, no shutdown requirement.

**Why it happens:** LLM trained to summarize. Distillation feels natural. But technical specs need EXACT wording — "alarm only not shutdown" is a critical distinction.

**How to avoid:**
1. NO LLM in decision extraction loop for verbatim capture
2. Rule-based extraction: strip filler words ("um", "like"), keep technical content exactly
3. Show extracted decision immediately in UI for user confirmation
4. If user rejects, reopen question (don't edit inline — restart conversation flow)

**Warning signs:**
- Decisions shorter than user's original answer
- Technical terminology changed ("85°C" → "high temperature")
- Multiple constraints collapsed to one ("alarm not shutdown" → "alarm")

**Reference:** [UX Research Verbatim Transcription 2026](https://www.looppanel.com/blog/how-to-transcribe-interviews-for-ux-research)

### Pitfall 3: Foundation Phase as Structured Topics

**What goes wrong:** Foundation phase (Phase 1) shows topic selection card with "Scope boundaries", "Equipment grouping", etc. Engineer picks topics, gets structured questions. But Foundation needs open-ended intake — what IS the system, what docs exist, overall scope.

**Why it happens:** Code treats all phases identically. Foundation phase keyword "scope" matches generic content type detection → generates structured gray areas.

**How to avoid:**
1. Detect Foundation phase explicitly (phase_number == 1 OR phase_goal contains "foundation"/"intake")
2. Use different prompt template: open-ended Q&A, no topic cards
3. Skip topic selection step for Foundation
4. Build Foundation-specific UI component (IntakeConversation.tsx) for free-flowing discussion

**Warning signs:**
- Phase 1 shows topic selection when it should be open conversation
- Foundation discussion feels rigid/structured
- Engineer can't describe overall system before being asked specific questions

**Reference:** v1.0 discuss-phase.md Step 2.2 Foundation detection, CONTEXT.md user decision

### Pitfall 4: Chip Selection as Final Answer

**What goes wrong:** Question card shows chips for common answers. Engineer clicks "Yes" chip. System treats this as complete answer, moves to next question. But chip is a STARTING point — needs follow-up to deepen.

**Why it happens:** Chip click triggers onAnswer callback. Looks like a complete answer. But user decision: "Chip selections always trigger a follow-up question to deepen the answer."

**How to avoid:**
1. Chip click adds to message history but doesn't close question
2. LLM sees chip selection, generates follow-up: "You selected 'Yes'. Can you describe the specific conditions?"
3. Only mark question complete when freeform text input submitted OR LLM explicitly says "Moving to next topic"
4. Track question state: awaiting_initial_response → chip_selected_awaiting_details → completed

**Warning signs:**
- Single-chip-click discussions with shallow decisions
- No follow-up depth after option selection
- Summary panel shows decisions like "Yes", "Option A" without detail

### Pitfall 5: SSE Connection Leaks on Component Unmount

**What goes wrong:** User opens chat panel, starts discussion, SSE stream connects. User closes chat panel. Component unmounts but EventSource connection stays open. Memory leak, zombie connections.

**Why it happens:** React cleanup (useEffect return) not implemented. EventSource.close() never called.

**How to avoid:**
1. Store EventSource instance in useRef
2. useEffect cleanup function calls eventSource.close()
3. Abort ongoing fetch streams with AbortController
4. Test: open/close chat panel 10 times, check browser network tab for lingering connections

**Warning signs:**
- Network tab shows multiple EventSource connections for same conversation
- Backend SSE connection count increases without bound
- Memory usage grows when opening/closing chat repeatedly

**Reference:** [SSE Implementation Best Practices 2026](https://medium.com/@inandelibas/real-time-notifications-in-python-using-sse-with-fastapi-1c8c54746eb7)

### Pitfall 6: CONTEXT.md Generation Without Size Control

**What goes wrong:** Discussion captures 50 decisions across 8 topics. CONTEXT.md generator dumps all decisions verbatim. File grows to 300 lines. v1.0 requires <100 lines for usability.

**Why it happens:** Faithfulness to verbatim capture conflicts with size limit. No compression logic.

**How to avoid:**
1. Priority tiers for CONTEXT.md content (v1.0 Step 6.4):
   - Tier 1: Decisions that change what gets written (KEEP)
   - Tier 2: Specific technical values (temperatures, capacities) (KEEP)
   - Tier 3: General approach notes (COMPRESS or OMIT)
2. Use tables for parameter values (compact format)
3. Combine related decisions under single topic heading
4. Claude's Discretion section: bullet list, not full paragraphs
5. Validate line count before saving, compress if needed

**Warning signs:**
- CONTEXT.md exceeds 100 lines
- Decisions section has verbose explanations for every point
- Planning phase complains "too much context to parse"

**Reference:** v1.0 discuss-phase.md Step 6.4 size control

### Pitfall 7: Completion Without Explicit Confirmation

**What goes wrong:** LLM finishes last topic, immediately generates CONTEXT.md, saves file, marks discussion complete. Engineer wanted to add one more decision, but discussion is closed.

**Why it happens:** Auto-completion feels smooth. LLM detects "all topics covered" and proceeds.

**How to avoid:**
1. LLM posts completion_card event when topics covered
2. CompletionCard.tsx shows: "All selected topics discussed. Review decisions?"
3. Engineer actions: "Meer toevoegen" (opens freeform input) OR "Bevestig" (generates CONTEXT.md)
4. Only after "Bevestig" click does backend generate CONTEXT.md
5. Show CONTEXT.md preview for review/edit before final save

**Warning signs:**
- Engineers report "discussion closed too early"
- No way to add final thoughts after last structured topic
- CONTEXT.md appears without preview/confirmation step

**Reference:** User decision in CONTEXT.md — "AI determines when covered AND posts completion card for engineer confirmation"

### SSE Streaming with Structured Event Parsing

```python
# Source: backend/app/services/discussion_engine.py (Phase 10 implementation) + enhancements
from typing import AsyncGenerator
import re

async def send_message(
    self,
    conversation_id: int,
    content: str,
    attachments: Optional[list[str]] = None,
) -> AsyncGenerator[dict, None]:
    """
    Process user message and stream AI response with structured event parsing.

    Enhanced from Phase 10 to parse question cards and decisions.
    """
    # [... existing message persistence logic ...]

    # Build messages array for LLM
    messages = await self._build_message_history(conversation_id)

    # Stream LLM response
    full_response = ""
    question_card_buffer = ""
    in_question_tag = False

    try:
        async for chunk in self.llm.stream_complete(messages, max_tokens=4096):
            full_response += chunk

            # Check for question card tags
            if "<question>" in chunk:
                in_question_tag = True
                question_card_buffer = chunk
            elif in_question_tag:
                question_card_buffer += chunk
                if "</question>" in chunk or "</options>" in chunk:
                    # Parse and emit question card event
                    card_data = self._parse_question_card(question_card_buffer)
                    if card_data:
                        yield {
                            "event": "question_card",
                            "data": card_data
                        }
                    in_question_tag = False
                    question_card_buffer = ""
            else:
                # Regular text delta
                yield {
                    "event": "message_delta",
                    "data": {"delta": chunk}
                }

        # Extract decision if user answered a question
        decision = self._extract_decision_verbatim(
            user_message=content,
            conversation_state=conversation.summary_data
        )
        if decision:
            # Add to summary_data
            conversation.summary_data["decisions"].append(decision)
            # Emit decision event
            yield {
                "event": "decision_captured",
                "data": decision
            }

        # [... persist assistant message, commit, yield done ...]

    except Exception as e:
        yield {"event": "error", "data": {"error": str(e)}}
        raise

def _parse_question_card(self, llm_output: str) -> Optional[dict]:
    """Parse XML-tagged question card from LLM output."""
    question_match = re.search(r'<question>(.*?)</question>', llm_output, re.DOTALL)
    options_match = re.search(r'<options>(.*?)</options>', llm_output, re.DOTALL)

    if not question_match:
        return None

    question = question_match.group(1).strip()
    options = []
    if options_match:
        options = re.findall(r'<option>(.*?)</option>', options_match.group(1))

    return {
        "question": question,
        "options": [opt.strip() for opt in options]
    }

def _extract_decision_verbatim(self, user_message: str, conversation_state: dict) -> Optional[dict]:
    """
    Extract verbatim decision from user message.

    NO LLM summarization. Trim filler, keep exact technical content.
    """
    current_topic = conversation_state.get("current_topic")
    if not current_topic:
        return None

    # Strip filler at sentence boundaries only
    verbatim = user_message.strip()
    filler_patterns = [
        r'^(um|uh|like|so|well),?\s+',
        r',?\s+(you know|I think)\s*$'
    ]
    for pattern in filler_patterns:
        verbatim = re.sub(pattern, '', verbatim, flags=re.IGNORECASE)

    return {
        "topic": current_topic,
        "decision": verbatim,  # Engineer's EXACT words
        "timestamp": datetime.utcnow().isoformat(),
        "confirmed": False  # User must confirm in UI
    }
```

```typescript
// Source: frontend/src/features/discussions/hooks/useDiscussionStream.ts (Phase 10) + cleanup enhancements
import { useEffect, useRef, useCallback } from 'react'

export function useDiscussionStream() {
  const abortControllerRef = useRef<AbortController | null>(null)
  const [isStreaming, setIsStreaming] = useState(false)

  // Cleanup on unmount
  useEffect(() => {
    return () => {
      // Abort any ongoing streams
      if (abortControllerRef.current) {
        abortControllerRef.current.abort()
      }
    }
  }, [])

  const sendMessage = useCallback(async (content: string) => {
    // Cancel existing stream
    if (abortControllerRef.current) {
      abortControllerRef.current.abort()
    }
    abortControllerRef.current = new AbortController()

    setIsStreaming(true)

    try {
      const response = await fetch(`/api/.../messages/stream`, {
        method: 'POST',
        body: JSON.stringify({ content }),
        signal: abortControllerRef.current.signal,
      })

      const reader = response.body?.getReader()
      const decoder = new TextDecoder()
      let buffer = ''

      while (true) {
        const { done, value } = await reader.read()
        if (done) break

        buffer += decoder.decode(value, { stream: true })
        const lines = buffer.split('\n')
        buffer = lines.pop() || ''

        for (const line of lines) {
          if (line.startsWith('data: ')) {
            const data = line.slice(6)
            const event = JSON.parse(data)

            // Handle different event types
            if (event.event === 'question_card') {
              // Add question card to messages
            } else if (event.event === 'decision_captured') {
              // Update summary panel
            }
            // [... other event handlers ...]
          }
        }
      }
    } catch (err) {
      if (err.name === 'AbortError') {
        // Ignore - expected on cleanup
        return
      }
      throw err
    } finally {
      setIsStreaming(false)
    }
  }, [])

  return { sendMessage, isStreaming }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| WebSockets for chat | SSE for one-way streaming | 2025-2026 | Simpler implementation, auto-reconnect, lower overhead for LLM → client flow |
| JSON structured output | XML tags for Claude | 2025 (Anthropic guidance) | More reliable with Claude models, parseable mid-stream |
| Manual token counting | LLM provider token API | 2024+ | Model-specific tokenization (Claude ≠ GPT), prevents off-by-one errors |
| Prompt injection for tools | Native tool calling API | 2024+ (Claude, GPT-4, Gemini) | Structured tool calls, no ReAct prompt loop needed |
| Global conversation state | State machine orchestration | 2025-2026 | Clear workflow phases, prevents LLM jumping topics |
| Full conversation history reuse | Context window summarization | 2026 | 70-80% capacity trigger for summarization, prevents context rot |

**Deprecated/outdated:**
- Manual EventSource polyfills (native browser support stable)
- Regex-only XML parsing (lxml handles edge cases better)
- Single-conversation multi-step workflows (context rot proven issue)

## Open Questions

1. **Freeform input availability between question cards**
   - What we know: User wants to optimize for keeping LLM focused on current topic
   - What's unclear: Always-on text input vs card-only input (reveal after chip/question)
   - Recommendation: Card-only to start (matches v1.0 AskUserQuestion pattern). Always-on adds cognitive load. Test with users.

2. **CONTEXT.md preview location**
   - What we know: Must show preview before finalizing
   - What's unclear: In chat panel (CompletionCard component) vs main workspace area (separate panel)
   - Recommendation: Chat panel for preview, main workspace for edit mode (larger text area). Matches existing UI pattern (chat for flow, workspace for content).

3. **Chip follow-up depth**
   - What we know: Chips trigger follow-up questions
   - What's unclear: How many follow-up levels? (1 level: chip → detail, or 2+ levels: chip → detail → edge case?)
   - Recommendation: 1 level follow-up to start. If engineer's follow-up answer is shallow, LLM can ask another clarifying question, but don't hardcode multi-level. Let LLM judge depth.

4. **Reference file reading during discussion**
   - What we know: User wants AI to read intake folder files for informed questions
   - What's unclear: How to surface file content to LLM (embed in prompt? RAG retrieval? File path references?)
   - Recommendation: For Foundation phase, pre-load intake folder file summaries into system prompt. For other phases, reference CONTEXT.md from Phase 1 (contains extracted info). Defer full RAG to later phase if needed.
