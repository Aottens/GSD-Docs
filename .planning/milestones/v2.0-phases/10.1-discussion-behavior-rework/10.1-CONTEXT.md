# Phase 10.1: Discussion Behavior Rework - Context

**Gathered:** 2026-02-16
**Updated:** 2026-02-17 (UAT fix round)

<domain>
## Phase Boundary

Rework the discussion chat behavior so it follows the proven v1.0 discuss-phase flow: AI-driven topic progression, structured question cards, faithful decision capture, and clear completion with CONTEXT.md generation. Phase 10 built the infrastructure (chat panel, SSE streaming, DB models, components); this phase makes the actual discussion behavior work correctly.

**UAT update:** Plans 01-07 are code-complete. Human UAT on 2026-02-17 found 5 issues (2 blockers, 2 major, 1 minor). This context update captures decisions for fixing those issues.

</domain>

<decisions>
## Implementation Decisions

### Conversation flow
- AI-driven: AI leads the discussion, presents topics, asks questions, checks satisfaction, moves to next topic
- Triggered by "Bespreken" button — clicking it immediately starts the AI-driven discussion for that phase
- Structured topic progression matching v1.0: 4 questions per area, then check-in ("More questions or next topic?"), explicit boundaries between areas
- Topic selection step at start: AI presents detected gray areas as multi-select card, engineer picks which to discuss
- AI reads reference files from the intake folder during discussion to ask informed, context-aware questions

### Foundation conversation style (UAT fix — Issue 1C)
- Foundation phase uses the SAME question card + option chip format as regular topics (NOT bare freeform-only cards)
- LLM generates relevant option chips for Foundation questions — same interaction pattern as structured topics
- Freeform text input always available alongside chips for elaboration
- Keep the 5 fixed intake areas (system description, reference docs, scope, structure, special requirements)
- Mark each area as done when covered
- If gray areas surface during intake, note them and offer the chance to discuss before moving to the next area
- Foundation also follows the same check-in rhythm and decision summary pattern as regular topics

### Question & answer UX
- Question cards: styled cards with question text, clickable option chips for common answers, and text input for detailed/custom answers
- LLM generates questions in structured format (e.g., XML tags), backend parses and converts to question_card SSE events
- Chip selections always trigger a follow-up question to deepen the answer — chips are starting points, not final answers
- Freeform input approach: Claude's Discretion (optimize for keeping LLM focused on current topic)

### Decision summary timing (UAT fix — Issue 1B)
- Decisions accumulate SILENTLY during Q&A — no individual decision_captured events shown to the engineer
- At topic check-in: show full decision summary as a styled card in the chat AND populate the Beslissingen tab simultaneously
- Decisions from all completed topics accumulate in the Beslissingen tab (don't clear between topics)
- Confirm/reject UX for individual decisions at check-in: Claude's Discretion
- Verbatim/faithful extraction — use the engineer's exact words, trimmed to key sentences. No AI interpretation or distillation

### Loop prevention (UAT fix — Issue 1A)
- Deploy Plan 07 probe tracking changes (currently uncommitted) as baseline fix
- Anti-repetition strategy (LLM instruction + backend guardrail, or hard block): Claude's Discretion
- Foundation area detection fallback when keyword matching fails: Claude's Discretion
- Hard cap of 12 questions per topic as safety valve — force check-in at 12 regardless of probe state
- Probe exhaustion / check-in transition behavior: Claude's Discretion

### Conversation history & navigation (UAT fix — Issues 1D + 1E)
- Full chat history visible when reopening any conversation — every message, question card, and decision exactly as it appeared
- Gesprekken tab clicks load the conversation — navigation behavior: Claude's Discretion
- Three conversation states: Active (in progress), Paused (state persisted to files for clean resume), Completed (read-only)
- Pause saves structured state (decisions so far, topics covered, current position) to files — prevents content rot
- Resume rebuilds context from persisted files, not from replaying full conversation history — clean LLM context on resume
- Resume UX (full history vs compressed summary for the engineer): Claude's Discretion
- Note: Pause/resume pattern may need adjustment in later phases

### Completion & output
- AI determines when all selected topics are covered and posts a completion card for engineer confirmation
- Auto-generates CONTEXT.md from captured decisions, shown as preview for review/edit before finalizing
- After finalizing: suggest next workflow step (planning), do NOT auto-advance
- Clear context window between workflow steps (new conversation/session for planning) — prevents context rot
- Only advance if previous step is properly completed

### Claude's Discretion
- CONTEXT.md preview location (in chat panel vs workspace main area)
- Exact structured format the LLM uses for question generation (XML tags, markdown markers, etc.)
- Confirm/reject UX for individual decisions at check-in
- Anti-repetition strategy details (LLM + guardrail vs hard block)
- Foundation area detection fallback (question count vs LLM classification)
- Probe exhaustion transition (auto check-in vs open-ended follow-up)
- Gesprekken tab navigation behavior (load in panel vs navigate to phase)
- Resume UX for paused conversations (full history vs compressed summary)

</decisions>

<specifics>
## Specific Ideas

- V1.0 discuss-phase flow is the reference: same structured topic progression, same "4 questions then check" rhythm
- Foundation/intake phase conversation must be extensive — this is the project base, not a quick form
- "Interpretation is killing while writing technical documentation" — verbatim decision capture is a hard requirement
- "Context rot is killing" — fresh context window between workflow steps is a hard requirement
- AI must read intake folder reference files to ask informed questions (e.g., "I see from [document] that... is this still current?")
- Pause/resume inspired by GSD's `/gsd:pause-work` pattern — persist structured state to files, resume from files not from conversation replay
- Foundation questions should feel like a normal chat with option chips — "like we are chatting now, you provide some options and I can use free text when needed"

</specifics>
