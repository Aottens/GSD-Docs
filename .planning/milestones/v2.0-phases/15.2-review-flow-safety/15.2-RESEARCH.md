## Summary

Phase 15.2 closes three concrete gaps identified in the v2.0 milestone audit (2026-03-26). All three are pure frontend changes — no new backend routes, no schema changes, and no new npm packages required. The work touches three well-scoped areas.

**Gap 1 — Export safety banner (QUAL-04):** The `AssemblyPipeline` component already checks `readiness.unreviewed_phases` and shows a blocking `Alert` for `final` mode. The gap is in `draft` mode: sections with review status `afgewezen` or without any review entry can still be exported silently. The fix is a non-blocking warning banner in `AssemblyPipeline` that reads review state from localStorage (key pattern: `review-session-{projectId}-{phaseNumber}`) for all phases, computes the set of rejected/unreviewed sections, and presents a warning with "Doorgaan" / "Terug naar documenten" actions. Because `AssemblyPipeline` does not have direct access to `ReviewContext` (it lives in the export feature, outside `ReviewProvider`), it must read localStorage directly or receive summary data as a prop.

**Gap 2 — Multi-phase selection in DocumentsTab (QUAL-05):** `DocumentsTab` receives `activePhaseNumber` as a prop. `ProjectWorkspace` sets this to `activePhaseForReview`, which is always the highest-numbered phase with `has_verification=true`. This means an engineer cannot navigate to an older phase's review state in the GUI. The fix is a phase selector (Select dropdown or tab set) inside `DocumentsTab` that allows picking from all phases that have verification data. The phase list is already available from `usePhaseTimeline` (or can be derived from the `PhaseTimelineData` type already in scope). `DocumentsTab` needs to own a `selectedPhaseNumber` state and offer a dropdown when multiple verified phases exist.

**Gap 3 — Review persistence across navigation (QUAL-06):** `ReviewProvider` already persists to `localStorage` on every dispatch and initializes from localStorage in the `useReducer` initializer. This means persistence technically works today — but the gap is that `ReviewProvider` is re-mounted every time `DocumentsTab` unmounts/remounts (e.g., tab navigation). On re-mount it reads localStorage correctly. The issue to verify: whether the `phaseNumber` that `ReviewProvider` receives changes when the engineer returns to the Documents tab (because `activePhaseForReview` in `ProjectWorkspace` re-derives from polled data). If so, a different `storageKey` is generated and the previous session appears lost. The fix for QUAL-06 is to ensure `phaseNumber` passed to `ReviewProvider` is stable (not re-derived from polling on every render) and to add a visible persistence indicator so engineers can confirm their work is saved.

**Primary recommendation:** Three discrete frontend tasks, implemented in parallel. No backend changes. No new dependencies.

## Architecture Patterns

### Existing Patterns to Extend

#### Pattern 1: AssemblyPipeline Warning Banner

The `AssemblyPipeline` component already contains two `Alert` blocks (`hasNoContent`, `isFinalBlocked`). The review-safety warning fits the same pattern: compute a boolean flag from review state, render an `Alert` with `variant="default"` or `variant="destructive"` and two action buttons.

**Key design decision:** `AssemblyPipeline` lives outside `ReviewProvider` scope. To read per-section review state, it must:
- Option A: Read from `localStorage` directly using the key `review-session-{projectId}-{phaseNumber}`. The `phaseNumber` comes from the readiness data — `readiness.unreviewed_phases` already names phases with SUMMARY.md but no REVIEW.md.
- Option B: Lift `ReviewContext` higher (into `ProjectWorkspace` or `ExportTab`). This would require passing `projectId` + `phaseNumber` to a `ReviewProvider` at workspace level.

**Recommendation:** Option A — read localStorage directly in a small utility function inside `AssemblyPipeline`. This is the minimal-scope change. The export feature is intentionally decoupled from the review context; reading localStorage is a clean read-only access pattern.

**What "rejected or unreviewed" means for the banner:**
- `afgewezen` status sections: explicitly flagged by engineer
- No entry in `reviews` for a section with content: unreviewed

The banner does NOT need to enumerate every section. It needs a count: "X secties afgewezen, Y secties nog niet beoordeeld." It offers two buttons: "Doorgaan toch" (proceeds with export) and "Terug naar documenten" (calls an `onNavigateToDocs` callback prop).

**AssemblyPipeline already has `projectId` prop.** It needs `onNavigateToDocs?: () => void` added. `ExportTab` gets this prop from `ProjectWorkspace` which already has `setActiveSection`.

#### Pattern 2: Multi-Phase Selector in DocumentsTab

`DocumentsTab` currently receives `activePhaseNumber?: number` from `ProjectWorkspace`. The change:

1. `DocumentsTab` gains local state: `const [selectedPhase, setSelectedPhase] = useState(activePhaseNumber)`
2. When `activePhaseNumber` changes externally (prop update), `selectedPhase` syncs via `useEffect`.
3. A `Select` dropdown renders above the content area, showing all phases with `has_verification=true`. Uses `usePhaseTimeline(projectId)` (already wired into `ProjectWorkspace`; can be called again in `DocumentsTab` — TanStack Query deduplicates).
4. When exactly one verified phase exists, the dropdown is hidden (no visual noise).
5. When `selectedPhase` changes, `ReviewProvider` re-mounts with the new `phaseNumber`, loading that phase's localStorage session.

**Phase list shape** (from `PhaseTimelineData`):
```typescript
phases.filter(p => p.has_verification)
  .map(p => ({ value: p.number, label: `Fase ${p.number}: ${p.name}` }))
```

**Component location:** The Select lives at the top of `DocumentsTab`, above the split-panel layout, only when `verifiedPhases.length > 1`.

#### Pattern 3: Review Persistence Stability

The current `ReviewProvider` key is `review-session-{projectId}-{phaseNumber}`. The issue: if `ProjectWorkspace` re-derives `activePhaseForReview` from TanStack Query polling and this number changes mid-session, the engineer's work appears to vanish.

**Root cause check:** `activePhaseForReview` is `phaseTimelineData?.phases?.filter(p => p.has_verification)?.sort((a, b) => b.number - a.number)[0]?.number`. This is re-computed on every render when the query data reference changes. With `refetchInterval: 10000`, this could cause subtle re-mounts if a new phase gains `has_verification=true` while the engineer is reviewing.

**Fix:** Once `DocumentsTab` owns its `selectedPhase` state (from Pattern 2 above), the `ReviewProvider` key is driven by user selection, not polling. The fix for Pattern 3 is therefore a consequence of Pattern 2.

**Additional persistence indicator:** The `ReviewActionBar` already shows an "Opgeslagen" confirmation for 2 seconds after save. No additional indicator needed, but a static "Sessie opgeslagen in browser" note in the `ReviewSummary` component would close the confidence gap for engineers.

### Recommended Implementation Order

1. **Task 1** — Export warning banner (self-contained, touches only `AssemblyPipeline`, `ExportTab`, `ProjectWorkspace`)
2. **Task 2** — Multi-phase selector (touches `DocumentsTab`, trivially fixes Task 3 as a side effect)
3. **Task 3** — Persistence indicator (small addition to `ReviewSummary`, verifies localStorage is working)

### Anti-Patterns to Avoid

- **Don't lift ReviewProvider to ProjectWorkspace level** — it would couple export feature to review state, breaking the feature isolation established in Phase 12.
- **Don't block draft export** — the existing architecture explicitly allows draft export without review. The banner is a warning, not a hard block (hard block is already in place for `final` mode via `isFinalBlocked`).
- **Don't add a new polling endpoint for review state** — review state is local (localStorage), not server state. No backend involvement needed.
- **Don't re-implement phase filtering** — `usePhaseTimeline` is already called in `ProjectWorkspace`. Call it again in `DocumentsTab` — TanStack Query caches and deduplicates automatically.

## Common Pitfalls

### Pitfall 1: Banner reads stale localStorage on mount

**What goes wrong:** `AssemblyPipeline` reads localStorage once on mount. If the engineer switches to Documents tab, adds reviews, then switches back to Export tab, the component does not re-read unless it re-mounts.

**Why it happens:** `ExportTab` (and therefore `AssemblyPipeline`) stays mounted when the engineer navigates via `ProjectNavigation` — `ProjectWorkspace` renders all tabs conditionally via `{activeSection === 'export' && <ExportTab ... />}`, so `ExportTab` unmounts on tab switch and re-mounts on return. This means the banner WILL re-read localStorage on return.

**How to avoid:** No special handling needed. The unmount/re-mount cycle is already correct for this use case. Verify by checking `ProjectWorkspace` conditional rendering — confirmed: each tab is conditionally rendered, not hidden.

### Pitfall 2: phaseNumber instability causes ReviewProvider re-mount mid-session

**What goes wrong:** `ProjectWorkspace` derives `activePhaseForReview` from polling data. If a second phase gains `has_verification=true` while the engineer is reviewing phase N, the derived value jumps to phase N+1. `ReviewProvider` re-mounts with a new key, and the engineer's current in-flight changes (not yet committed to localStorage) are lost.

**Why it happens:** `useReducer` state is lost on unmount. Only the completed `localStorage.setItem` calls survive.

**How to avoid:** Pattern 2 (user-controlled `selectedPhase` state in `DocumentsTab`) eliminates this by decoupling phase selection from polling.

### Pitfall 3: "Unreviewed sections" count is inaccurate without knowing total section count

**What goes wrong:** The banner says "Y sections not reviewed." But which sections should be reviewed? The document outline has N sections total, some of which are `empty`. Counting unreviewed against all N overstates the problem.

**Why it happens:** The banner reads raw review entries (what the engineer has touched) vs. all outline sections.

**How to avoid:** Use a conservative heuristic for the banner: count only sections with `afgewezen` status (explicit rejection) for the hard warning. For "unreviewed," note only that the review session has fewer than the full outline count — or skip the count and just say "Sommige secties zijn nog niet beoordeeld." This is honest without requiring the banner to fetch the full outline.

### Pitfall 4: Select dropdown causes layout shift

**What goes wrong:** Adding a Select above the split-panel in `DocumentsTab` pushes the panel down and breaks the fixed-height layout (`h-full overflow-hidden`).

**Why it happens:** The outer `div ref={containerRef}` uses `flex h-full overflow-hidden`. Adding a sibling above it within the same flex container takes vertical space.

**How to avoid:** Wrap the Select + split-panel in a `flex flex-col h-full` container. The Select takes `shrink-0` height, the split-panel takes `flex-1 overflow-hidden`. This preserves the existing drag-resize behavior.

## Code Examples

### Reading localStorage for review state (utility function)

```typescript
// Proposed utility: frontend/src/features/documents/utils/reviewStorage.ts
import type { SectionReview } from '../types/verification'

export function readReviewSession(
  projectId: number,
  phaseNumber: number
): Record<string, SectionReview> {
  try {
    const key = `review-session-${projectId}-${phaseNumber}`
    const stored = localStorage.getItem(key)
    return stored ? JSON.parse(stored) : {}
  } catch {
    return {}
  }
}

export function countRejected(reviews: Record<string, SectionReview>): number {
  return Object.values(reviews).filter(r => r.status === 'afgewezen').length
}
```

## Open Questions

1. **Which phase number does the export banner use to read localStorage?**
   - What we know: `AssemblyPipeline` has `projectId`. `readiness.unreviewed_phases` is a list of phase dir names (strings like `"01-foundation"`), not integers.
   - What's unclear: To compute `rejectedCount`, we need the phaseNumber as integer to form the localStorage key. We could iterate all localStorage keys matching the pattern `review-session-{projectId}-*`, or pass `selectedPhaseNumber` down from `DocumentsTab` → `ProjectWorkspace` → `ExportTab` → `AssemblyPipeline`.
   - Recommendation: Iterate localStorage keys with `Object.keys(localStorage).filter(k => k.startsWith(\`review-session-${projectId}-\`))`, aggregate rejected counts across all phases. This is the most correct approach and requires no prop threading.

2. **Should the banner only appear for `draft` mode, or also `final` mode?**
   - What we know: `final` mode already has `isFinalBlocked` which is a hard block. Adding the rejected-sections banner on top creates redundant UI.
   - Recommendation: Show the rejected-sections banner for BOTH modes, positioned above the existing `isFinalBlocked` alert. They address different concerns: `isFinalBlocked` = missing REVIEW.md file; rejected-sections banner = GUI review state. An engineer could have a REVIEW.md (from CLI) but also have GUI rejections logged.
