## Summary

The bug is a missing `?project_id=` query parameter in a single `postForm` call inside `useOverrideFile` in `frontend/src/features/files/hooks/useFiles.ts` (line 116). The backend endpoint `POST /files/{file_id}/override` declares `project_id` as a FastAPI `Query(...)` parameter — meaning it is **required** in the URL query string, not in the request body. Because `postForm` sends a `multipart/form-data` body (for the file), the `project_id` has no place in the body and must be appended to the URL.

The current call sends `POST /api/files/{fileId}/override` with no query string. FastAPI cannot find the required `project_id` parameter and returns HTTP 422 Unprocessable Entity on every attempt. The fix appends `?project_id=${projectId}` directly to the URL string passed to `api.postForm()`. The `projectId` is already a parameter of `useOverrideFile(projectId: number)` and is available in scope — it just was never used in the URL construction.

No backend changes, no new hooks, no architectural changes are required. This is a one-line fix to a URL string.

**Primary recommendation:** Change `api.postForm<FileMetadata>('/files/${fileId}/override', formData)` to `api.postForm<FileMetadata>('/files/${fileId}/override?project_id=${projectId}', formData)` inside the `mutationFn` of `useOverrideFile`.

## Architecture Patterns

### Pattern 1: Query Parameter on multipart POST

**What:** FastAPI's `Query(...)` means the parameter MUST arrive in the URL query string, not the request body. For multipart requests the browser sets `Content-Type: multipart/form-data` and the body contains only the file. Query parameters must therefore be appended to the URL before calling `postForm`.

**Correct pattern already used elsewhere in this codebase** — `buildQueryParams` utility is available in `useFiles.ts` for this exact purpose, but is not needed here because `project_id` is a single fixed value:

```typescript
// Source: frontend/src/features/files/hooks/useFiles.ts (line 116) — AFTER fix
return api.postForm<FileMetadata>(`/files/${fileId}/override?project_id=${projectId}`, formData)
```

**Alternative — using buildQueryParams (also valid but unnecessary complexity):**
```typescript
const params = buildQueryParams({ project_id: projectId })
return api.postForm<FileMetadata>(`/files/${fileId}/override${params}`, formData)
```

The direct string interpolation is simpler and preferred for a single required parameter.

### Pattern 2: Cache Invalidation Scope

After a successful override the mutation currently invalidates:
- `fileKeys.projectFiles(projectId)` — project file list
- `fileKeys.projectFolders(projectId)` — project folder tree

This is correct — the override creates a new project-scoped file record. The shared library query cache does NOT need invalidation because shared files are not mutated.

### Recommended Project Structure (unchanged)

No structural changes needed. The fix is entirely contained within:

```
frontend/src/features/files/hooks/useFiles.ts  ← one-line fix at line 116
```

### Anti-Patterns to Avoid

- **Do NOT move `project_id` into the FormData body:** The backend reads it via `Query(...)`, not from the form body. Adding it to FormData would not fix the 422.
- **Do NOT add a new `postFormWithParams` method to `api.ts`:** The URL already accepts a query string — no new API client surface is needed.
- **Do NOT modify the backend endpoint:** The endpoint signature is correct; the contract is query-param-first for non-file data in multipart endpoints.

## Common Pitfalls

### Pitfall 1: Adding project_id to FormData instead of URL
**What goes wrong:** Override still returns HTTP 422 — FastAPI's `Query(...)` binding ignores multipart body fields.
**Why it happens:** Intuitive assumption that "everything in a form POST goes in the body."
**How to avoid:** Check the backend signature. `project_id: int = Query(...)` is explicit — it requires the URL query string.
**Warning signs:** 422 with detail `field required` pointing at `project_id`.

### Pitfall 2: Forgetting to reset the file input after failed mutation
**What goes wrong:** After a 422 error, the hidden `<input type="file">` retains the selected file. If the user tries again, `onChange` does not fire (same file selected), and `overrideMutation.mutate` is never called.
**Why it happens:** `SharedLibraryTab.tsx` already resets the input on success path — but the reset is inside `handleFileSelected` AFTER `mutate` is called, not in the `onError` callback. With the bug fixed this is a pre-existing correctness issue that does not need changing now (the input is reset synchronously in `handleFileSelected` regardless of outcome).
**Warning signs:** Second override attempt on same file silently does nothing.

### Pitfall 3: Cache invalidation not triggered after override
**What goes wrong:** Project file list does not update after successful override — engineer sees no feedback that the override worked.
**Why it happens:** Would only occur if the `onSuccess` path was broken. Current invalidation keys are correct.
**How to avoid:** Verify toast `'Bestand overschreven in projectbestanden'` appears and the project file list refreshes in the test plan.

export function useOverrideFile(projectId: number) {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: ({ fileId, newFile }: { fileId: number; newFile: File }) => {

    onSuccess: () => {

  })
}
