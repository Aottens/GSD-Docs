## Interaction Contract

### Override Flow (the affected interaction)

1. Engineer clicks the override action on a shared library file row.
2. `SharedLibraryTab.handleOverride` sets `selectedSharedFileId` and programmatically clicks the hidden `<input type="file">`.
3. Engineer selects a replacement file from the OS file picker.
4. `handleFileSelected` fires: calls `overrideMutation.mutate({ fileId, newFile })`.
5. **After fix:** `useOverrideFile` appends `?project_id=${projectId}` to the POST URL.
6. Backend returns HTTP 201 Created.
7. Success toast: "Bestand overschreven in projectbestanden" (sonner, default duration).
8. React Query invalidates `projectFiles` and `projectFolders` cache keys ŌĆö project files tab refreshes.

**Error path (HTTP 422 pre-fix, or any error post-fix):**
- Error toast: "Overschrijven mislukt ŌĆö probeer opnieuw" (sonner, default duration).
- File input is reset regardless (synchronous reset after `mutate` call in `handleFileSelected`).
- No modal, no inline error ŌĆö toast only.

### No New Interaction Patterns

This phase introduces no new modals, sheets, drawers, confirmation dialogs, or loading states. The pending/loading state during the mutation is handled by `overrideMutation.isPending` which the existing `FileList` component already threads through its `showOverrideAction` prop chain.
