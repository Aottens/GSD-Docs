# Phase 16 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase summary:** This phase corrects a display bug in `VerificationDetailPanel`: all phase-level truths currently appear on every leaf section regardless of relevance. The fix filters `truths` in `SectionBlock` before passing to `VerificationDetailPanel`, and adds an inline empty-state message ("Geen verificatiebevindingen voor deze sectie.") when no truths match. No new components, screens, layout patterns, or dependencies are introduced. All visual design is inherited from the established design system. The only new rendered element is a single `<p>` empty-state sentence using an existing text style class.

## Copywriting Contract

All copy in this phase is Dutch (project-wide convention established in Phase 8).

| Element | Copy |
|---------|------|
| Primary CTA | none — this phase introduces no new CTAs |
| Empty state (no matching truths) | "Geen verificatiebevindingen voor deze sectie." |
| Empty state context | Rendered as `<p className="text-xs text-muted-foreground mt-4">` — non-alarming, factual, no action required |
| Error state | none — filtering is a pure synchronous computation; no error path exists |
| Destructive confirmation | none — no destructive actions in this phase |

**Copywriting rationale for empty state:**

"Geen verificatiebevindingen voor deze sectie." is chosen over alternatives because:
- "Geen problemen gevonden" implies the section was positively verified — which is only true for `PASS` truths, not for sections simply not referenced in `evidence_files`.
- "Geen gegevens beschikbaar" is too generic and implies a loading failure.
- "Geen verificatiebevindingen" is neutral and accurate: there are no findings (positive or negative) linked to this section in the current verification run.

Source: Dutch copy convention from Phase 8 decision; pattern consistent with `EmptySectionCard` ("Nog geen inhoud beschikbaar").

## Interaction Contract

### Verification Display — Before and After Fix

**Before (current broken state):**
- Every leaf section in the document outline renders `VerificationDetailPanel` with all phase truths (e.g., 15 truths regardless of section relevance).
- Engineer clicks "Verificatieresultaten" on section 2.1 and sees truths about sections 2.4, 3.1, 5.2, etc.
- The collapsible always shows the trigger label regardless of whether any truths apply to the section.

**After (correct state):**
1. `SectionBlock` calls `filterTruthsForSection(verificationData.truths, node.id)` before the render block.
2. If `sectionTruths.length > 0`: renders `VerificationDetailPanel` with filtered truths only. Engineer sees only truths whose `evidence_files` reference the current section ID.
3. If `sectionTruths.length === 0`: renders `<p className="text-xs text-muted-foreground mt-4">Geen verificatiebevindingen voor deze sectie.</p>` in place of `VerificationDetailPanel`. No collapsible trigger. No misleading "Verificatieresultaten" label.
4. `ReviewActionBar` renders unconditionally in both cases — review controls are independent of verification findings.

**Filter logic (prescriptive — executor must implement exactly this):**
```typescript
function filterTruthsForSection(truths: TruthResult[], sectionId: string): TruthResult[] {
  return truths.filter(truth => {
    if (truth.evidence_files.length === 0) return false
    return truth.evidence_files.some(file => {
      const match = file.match(/(?:sectie|section)\s+([\d.]+)/i)
      return match ? match[1] === sectionId : false
    })
  })
}
```

**Exact string match rule:** Use `match[1] === sectionId` (strict equality). Do not use `includes()` — "2.1" would incorrectly match "2.10" and "2.11".

**Dual-language regex rule:** Match both `sectie` (Dutch, primary format) and `section` (English, fallback). If neither pattern matches an evidence file string, that evidence file does not count as a section reference — the truth is treated as phase-level and excluded from all sections.

### GEBLOKKEERD Alert Scope

The `isBlocked` alert (max cycles exceeded) is rendered inside `VerificationDetailPanel`. When `sectionTruths.length === 0`, `VerificationDetailPanel` is not rendered, so `isBlocked` is not displayed for that section. This is correct: `isBlocked` is phase-level information visible in the phase timeline; showing it on every section without findings would be alarming noise.

### ReviewActionBar Remains Unconditional

`ReviewActionBar` is always rendered for leaf sections when the phase has verification, regardless of `sectionTruths.length`. Review controls (approve / reject / request changes) are independent of whether verification found issues in this section.

### No New Loading States

`filterTruthsForSection` is a synchronous filter on already-fetched data. No loading spinner, skeleton, or intermediate state is needed.

### No New Interaction Patterns

This phase introduces no modals, sheets, drawers, confirmation dialogs, or new toasts. The existing `useVerificationData` hook, `Collapsible`, `Badge`, `Alert`, and `ReviewActionBar` components are unchanged in structure — only the data passed to `VerificationDetailPanel` is narrowed.
