## Summary

Phase 15.1 is a targeted cleanup phase with five discrete, well-scoped tasks identified in the v2.0 milestone audit (2026-03-26). The codebase is already in good shape — this phase closes cosmetic and UX friction gaps rather than building new features.

The most technically interesting task is change-notification toasts. The application already polls three data sources (setup-state at 5s, phase timeline at 10s, document outline at 15s) using TanStack Query's `refetchInterval`. The challenge is detecting *when polled data changes* and triggering a `sonner` toast — without spamming on every poll. The correct pattern is comparing previous data with current data using TanStack Query's `select` or a `useRef`-based previous-value comparison in a `useEffect`. The `sonner` library (v2.0.7) is already installed and used throughout the project for file operations.

The four remaining tasks are straightforward surgical edits: (1) remove "Discussie starten" button from ProjectOverview, (2) wire "Referenties uploaden" to call `onNavigate('references')`, (3) replace the two private `CliCommandBlock` copies in `EmptySectionCard.tsx` and `PhasePopover.tsx` with imports of the shared `@/features/timeline/components/CliCommandBlock`, and (4) delete `ChatContextPanel.tsx` from `frontend/src/features/projects/components/`.

**Primary recommendation:** Implement change notifications as a custom hook `usePollingChangeNotifier` that wraps TanStack Query's `select` + `useRef` for previous-data comparison, then fire a `toast.info()` from `sonner`. Apply to the three existing polling hooks. All other tasks are one-to-five line edits.

### Pattern 1: Change Detection via useRef Previous-Value Comparison

**What:** Track previous query data in a `useRef`; compare with current data in a `useEffect`; fire toast when a meaningful field changes.

**When to use:** When `refetchInterval` is set and you want to notify the user that background data changed, without modifying the polling hook's core logic.

**Example:**
```typescript
// Applied to useSetupState — detect when has_scaffolding or file counts change
import { useRef, useEffect } from 'react'
import { useSetupState } from './useSetupState'
import { toast } from 'sonner'

export function useSetupStateWithNotification(projectId: number) {
  const query = useSetupState(projectId)
  const prevDataRef = useRef<typeof query.data>(undefined)

  useEffect(() => {
    if (!query.data || !prevDataRef.current) {
      prevDataRef.current = query.data
      return
    }
    // Compare a stable fingerprint — e.g. JSON of doc_types status
    const prev = JSON.stringify(prevDataRef.current.doc_types?.map(d => d.status))
    const curr = JSON.stringify(query.data.doc_types?.map(d => d.status))
    if (prev !== curr) {
      toast.info('Setup status bijgewerkt', {
        description: 'CLI heeft wijzigingen aangebracht in de projectconfiguratie.',
      })
    }
    prevDataRef.current = query.data
  }, [query.data])

  return query
}
```

**When to use this approach vs. alternatives:**
- Use `useRef` + `useEffect` (this pattern) when you need to compare previous vs current data inside a React component lifecycle.
- Do NOT put toast calls inside the `queryFn` — that runs outside React, and sonner's `toast()` needs to run in a React context (or after mount).
- Do NOT use `onSuccess` (removed from TanStack Query v5).

### Anti-Patterns to Avoid

- **Toast on every poll interval:** Comparing data in `useEffect` with `[query.data]` dependency fires only when data changes, not every N seconds. Do NOT set up a `setInterval` independently.
- **Toast in queryFn:** `queryFn` is not a React context. Sonner's `toast()` function works outside React but the Toaster component must be mounted. Use the effect pattern instead for safety.
- **Modifying useSetupState/usePhaseTimeline/useDocumentOutline directly:** Keep the polling hooks pure (no side effects). Create wrapper hooks or add the notification logic in the component that already mounts these hooks.
- **Spamming toasts:** Only fire when the fingerprint changes. The `useRef` approach prevents re-firing on re-renders.

## Common Pitfalls

### Pitfall 1: Toast fires on initial data load
**What goes wrong:** `useEffect` fires when `query.data` first becomes defined (transitions from `undefined` to data). This triggers a spurious "data changed" toast on page load.
**Why it happens:** The `useRef` starts as `undefined`, so the first comparison `prev !== curr` will always be true.
**How to avoid:** Guard with `if (!prevDataRef.current) { prevDataRef.current = query.data; return }`. Set the ref to initial data silently, only compare on subsequent changes.
**Warning signs:** Toast appears immediately on page load without any CLI activity.

### Pitfall 2: Stale ref on fast data changes
**What goes wrong:** If data changes twice before the effect runs, only the second change triggers notification.
**Why it happens:** React batches state updates; `useEffect` runs after render.
**How to avoid:** Not a real problem for this use case — polling intervals are 5-15 seconds, React renders are much faster. Accept the behavior.

### Pitfall 3: MessageSquare import left dangling
**What goes wrong:** After removing the "Discussie starten" button, `MessageSquare` import from lucide-react may be unused, causing a lint warning.
**How to avoid:** Check if `MessageSquare` appears anywhere else in `ProjectOverview.tsx`. If not, remove it from the import statement.

### Pitfall 4: Grid layout breaks after removing one button
**What goes wrong:** `grid-cols-3` with only 2 items looks visually unbalanced.
**How to avoid:** Change grid class to `sm:grid-cols-2` when the third button is removed.

### Pitfall 5: ChatContextPanel.tsx re-added by future AI tooling
**What goes wrong:** An AI agent might regenerate the file since it looks like valid scaffolding.
**How to avoid:** The deletion is sufficient. The audit document already notes the file as orphaned.

### Pitfall 6: PhasePopover private CliCommandBlock has different styling
**What goes wrong:** The private copy uses `text-xs` while EmptySectionCard private copy uses `text-sm`. The shared component uses `text-xs`.
**How to avoid:** Accept `text-xs` as the canonical size (consistent with FaseringTab and PhasePopover). The change in EmptySectionCard is intentional and documented.

## Code Examples

### Change Notification Hook (setup-state)
```typescript
// frontend/src/features/projects/hooks/useSetupStateWithNotification.ts
import { useRef, useEffect } from 'react'
import { toast } from 'sonner'
import { useSetupState } from './useSetupState'
import type { SetupStateResponse } from '@/types/setupState'

function fingerprint(data: SetupStateResponse): string {
  return data.doc_types?.map(d => `${d.id}:${d.status}:${d.file_count}`).join(',') ?? ''
}

export function useSetupStateWithNotification(projectId: number) {
  const query = useSetupState(projectId)
  const prevRef = useRef<string | undefined>(undefined)

  useEffect(() => {
    if (!query.data) return
    const curr = fingerprint(query.data)
    if (prevRef.current === undefined) {
      prevRef.current = curr
      return
    }
    if (prevRef.current !== curr) {
      toast.info('Setup status bijgewerkt', {
        description: 'CLI heeft wijzigingen aangebracht in de projectconfiguratie.',
      })
      prevRef.current = curr
    }
  }, [query.data])

  return query
}
```

### Change Notification Hook (document outline)
```typescript
// frontend/src/features/documents/hooks/useDocumentOutlineWithNotification.ts
import { useRef, useEffect } from 'react'
import { toast } from 'sonner'
import { useDocumentOutline } from './useDocumentOutline'
import type { DocumentOutlineResponse } from '../types/document'

function fingerprint(data: DocumentOutlineResponse): string {
  // Count sections with content and total section count
  function countWithContent(nodes: typeof data.sections): number {
    return nodes.reduce((acc, n) => {
      const self = n.status === 'complete' || n.status === 'written' ? 1 : 0
      return acc + self + countWithContent(n.children ?? [])
    }, 0)
  }
  return `${data.sections.length}:${countWithContent(data.sections)}`
}

export function useDocumentOutlineWithNotification(projectId: number) {
  const query = useDocumentOutline(projectId)
  const prevRef = useRef<string | undefined>(undefined)

  useEffect(() => {
    if (!query.data) return
    const curr = fingerprint(query.data)
    if (prevRef.current === undefined) {
      prevRef.current = curr
      return
    }
    if (prevRef.current !== curr) {
      toast.info('Documentstructuur bijgewerkt', {
        description: 'Nieuwe secties beschikbaar na CLI schrijffase.',
      })
      prevRef.current = curr
    }
  }, [query.data])

  return query
}
```

## Open Questions

1. **PhasePopover "Bekijk documenten" — should it filter to phase-specific sections?**
   - What we know: DocumentsTab accepts `activePhaseNumber` which enables review mode for that phase. The outline shows all sections regardless of phase.
   - What's unclear: The success criteria says "navigates to Documenten tab filtered to that phase's sections." The current DocumentsTab has no phase-based section filtering — it shows the full outline.
   - Recommendation: Interpret "filtered" as "with that phase's sections visible and review context active." The simplest implementation is to set `activePhaseForReview` to the phase number. Since `activePhaseForReview` already controls which phase's verification data appears, navigating to Documenten + setting the active phase number achieves the intent. No new filtering UI is needed.

2. **usePhaseTimeline is called in both ProjectWorkspace and PhaseTimeline — where to add notification?**
   - What we know: `ProjectWorkspace` calls `usePhaseTimeline` to find `activePhaseForReview`. `PhaseTimeline` calls it to render the bar. Both share the same query key so they use the same cache entry.
   - Recommendation: Add the notification wrapper in `ProjectWorkspace` only (it's the top-level consumer). `PhaseTimeline` continues using `usePhaseTimeline` directly. Prevents duplicate toasts.

3. **DocumentsTab calls useDocumentOutline — should notification wrapper replace it?**
   - Recommendation: Use `useDocumentOutlineWithNotification` in `DocumentsTab` instead of `useDocumentOutline`. Simple drop-in replacement since the wrapper returns the same query object.
