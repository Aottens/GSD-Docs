# Requirements: GSD-Docs Industrial

**Defined:** 2026-02-06
**Core Value:** Engineers can go from project brief to complete, verified FDS document through a structured, AI-assisted workflow

## v1 Requirements

### Project Initialization

- [x] **INIT-01**: `/doc:new-fds` classifies project as Type A, B, C, or D based on user responses
- [x] **INIT-02**: `/doc:new-fds` generates appropriate ROADMAP.md based on project type (6 phases Type A, 4-5 Type B, 3-4 Type C, 2 Type D)
- [x] **INIT-03**: `/doc:new-fds` creates PROJECT.md with project config (name, type, client, standards settings, language)
- [x] **INIT-04**: `/doc:new-fds` creates REQUIREMENTS.md with functional requirements derived from project scope
- [x] **INIT-05**: `/doc:new-fds` creates STATE.md initialized at Phase 1
- [x] **INIT-06**: `/doc:new-fds` scaffolds folder structure (.planning/, output/, diagrams/, export/)
- [x] **INIT-07**: For Type C/D projects, `/doc:new-fds` generates BASELINE.md capturing existing system as immutable reference
- [x] **INIT-08**: Dynamic ROADMAP evolution after System Overview phase — when >5 units identified, propose expanded phase grouping (3-5 units per phase, max 7)
- [x] **INIT-09**: User can accept, adjust, or reject proposed ROADMAP expansion

### Phase Workflow — Discuss

- [x] **DISC-01**: `/doc:discuss-phase N` reads ROADMAP.md to identify phase goals and scope
- [x] **DISC-02**: `/doc:discuss-phase N` identifies gray areas specific to the phase content type
- [x] **DISC-03**: `/doc:discuss-phase N` presents gray areas grouped by topic with structured questions
- [x] **DISC-04**: `/doc:discuss-phase N` captures all decisions in phase-N/CONTEXT.md
- [x] **DISC-05**: `/doc:discuss-phase N` updates RATIONALE.md with decision rationale for each significant choice
- [x] **DISC-06**: Items explicitly marked as "Claude's Discretion" are documented but not asked

### Phase Workflow — Plan

- [x] **PLAN-01**: `/doc:plan-phase N` reads CONTEXT.md decisions and ROADMAP.md phase goals
- [x] **PLAN-02**: `/doc:plan-phase N` generates one PLAN.md per section (NN-MM-PLAN.md format)
- [x] **PLAN-03**: Each PLAN.md includes: goal, sections, context, standards, verification checklist
- [x] **PLAN-04**: `/doc:plan-phase N` assigns wave numbers for parallel execution based on dependency analysis
- [x] **PLAN-05**: `/doc:plan-phase N` self-verifies plans before completing
- [x] **PLAN-06**: `/doc:plan-phase N --gaps` generates targeted fix plans from VERIFICATION.md gaps

### Phase Workflow — Write

- [x] **WRIT-01**: `/doc:write-phase N` discovers all PLANs in phase and groups by wave
- [x] **WRIT-02**: Each writer subagent loads only: PROJECT.md + phase CONTEXT.md + its own PLAN.md + standards (if enabled)
- [x] **WRIT-03**: Each writer produces CONTENT.md with substantive documentation (not stubs)
- [x] **WRIT-04**: Each writer produces SUMMARY.md (max 150 words, facts only)
- [x] **WRIT-05**: Writers in the same wave execute in parallel via subagent spawning
- [x] **WRIT-06**: STATE.md updated before and after each wave (checkpoint for recovery)
- [x] **WRIT-07**: Edge cases and failure modes detected during writing are captured in EDGE-CASES.md
- [x] **WRIT-08**: Cross-references created during writing are logged in CROSS-REFS.md

### Phase Workflow — Verify

- [x] **VERF-01**: `/doc:verify-phase N` reads phase goals from ROADMAP.md and derives must-have truths
- [x] **VERF-02**: Verification checks 5 levels: exists, substantive, complete, consistent, standards-compliant
- [x] **VERF-03**: Verification produces VERIFICATION.md with pass/gap status per truth and evidence
- [x] **VERF-04**: Cross-references checked with warn-only for targets not yet written
- [x] **VERF-05**: PASS result offered with optional Fresh Eyes review
- [x] **VERF-06**: GAPS_FOUND result routes to `/doc:plan-phase N --gaps`
- [x] **VERF-07**: Verification loop terminates after max 2 gap-closure cycles, then escalates to human_needed
- [x] **VERF-08**: After System Overview phase verify PASS, trigger ROADMAP evolution check

### Phase Workflow — Review

- [x] **REVW-01**: `/doc:review-phase N` presents CONTENT per section for client/engineer review
- [x] **REVW-02**: Feedback captured in phase-N/REVIEW.md
- [x] **REVW-03**: Issues from review route to `/doc:plan-phase N --gaps`

### Content Templates

- [x] **TMPL-01**: Equipment module template with structured tables
- [x] **TMPL-02**: State machine template with Mermaid stateDiagram-v2 + transition tables
- [x] **TMPL-03**: Interface template with overview table, signal list, protocol details
- [x] **TMPL-04**: Templates support configurable language (Dutch/English)

### State Management & Recovery

- [x] **STAT-01**: STATE.md tracks: current phase, current plan, operation status, completed items, decisions, blockers
- [x] **STAT-02**: STATE.md updated before and after each operation (checkpoint for crash recovery)
- [x] **STAT-03**: `/doc:status` reads STATE.md + ROADMAP.md and displays progress table with per-phase status
- [x] **STAT-04**: `/doc:resume` detects incomplete operations from STATE.md and offers: resume, view status, start other operation
- [x] **STAT-05**: Partial write detection: CONTENT.md < 200 chars, [TO BE COMPLETED] marker, or abrupt ending
- [x] **STAT-06**: Forward-only recovery -- completed work is never rolled back, only incomplete items are retried

### Standards Integration

- [x] **STND-01**: PackML state model reference loaded only when `standards.packml.enabled: true` in PROJECT.md
- [x] **STND-02**: ISA-88 equipment hierarchy reference loaded only when `standards.isa88.enabled: true` in PROJECT.md
- [x] **STND-03**: When PackML enabled: exact state names enforced, transitions validated, modes checked
- [x] **STND-04**: When ISA-88 enabled: terminology enforced (Unit, Equipment Module, Control Module), hierarchy depth validated
- [x] **STND-05**: Standards compliance checked as verification level in `/doc:verify-phase`
- [x] **STND-06**: Standards never pushed -- disabled by default, only enabled via explicit PROJECT.md config

### Document Assembly & Versioning

- [x] **ASBL-01**: `/doc:complete-fds` verifies all phases PASS before assembly
- [x] **ASBL-02**: `/doc:complete-fds` concatenates all CONTENT.md files with proper section numbering
- [x] **ASBL-03**: `/doc:complete-fds` resolves cross-references (symbolic to final section numbers)
- [x] **ASBL-04**: `/doc:complete-fds` performs final cross-reference validation -- broken refs block completion (unless --force)
- [x] **ASBL-05**: `/doc:complete-fds` detects orphan sections (not referenced by any other section)
- [x] **ASBL-06**: `/doc:complete-fds --force` generates with [BROKEN REF] placeholders and DRAFT suffix
- [x] **ASBL-07**: `/doc:complete-fds` generates ENGINEER-TODO.md for diagrams exceeding Mermaid complexity limits
- [x] **ASBL-08**: `/doc:complete-fds` archives phase files to `.planning/archive/vN.M/`
- [x] **ASBL-09**: Version management: v0.x internal drafts, vN.0 client releases
- [x] **ASBL-10**: `/doc:release --type client` promotes draft to client release (v0.x to v1.0)
- [x] **ASBL-11**: `/doc:release --type internal` bumps internal version (v1.2 to v1.3)
- [x] **ASBL-12**: FDS and SDS versioned independently; SDS references source FDS version on frontpage

### Knowledge Transfer

- [x] **KNOW-01**: RATIONALE.md updated automatically during `/doc:discuss-phase` with decision + why + alternatives + reference
- [x] **KNOW-02**: EDGE-CASES.md updated automatically during `/doc:write-phase` with situation + trigger + system behavior + recovery
- [x] **KNOW-03**: Fresh Eyes review offered after `/doc:verify-phase` PASS -- simulates new engineer reading documentation, flags ambiguities
- [x] **KNOW-04**: ENGINEER-TODO.md lists diagrams too complex for Mermaid with section reference, type, description, priority

### SDS Generation

- [x] **SDS-01**: `/doc:generate-sds` reads completed FDS and transforms to SDS structure
- [x] **SDS-02**: Equipment modules matched against CATALOG.json typicals library
- [x] **SDS-03**: Unmatched equipment flagged as "NEW TYPICAL NEEDED" (not hallucinated)
- [x] **SDS-04**: TRACEABILITY.md generated linking FDS requirements to SDS implementation
- [x] **SDS-05**: SDS has independent version number with "Based on: FDS vX.Y" on frontpage

### Export

- [x] **EXPT-01**: `/doc:export` renders Mermaid diagrams to PNG
- [x] **EXPT-02**: `/doc:export` converts FDS/SDS markdown to DOCX via Pandoc with huisstijl.docx template
- [x] **EXPT-03**: Exported DOCX includes: heading styles, header with logo, footer with page numbers, table formatting
- [x] **EXPT-04**: `--draft` flag available for work-in-progress exports
- [x] **EXPT-05**: `--skip-diagrams` flag available when mermaid-cli not installed
- [x] **EXPT-06**: Diagrams exceeding complexity limits to ENGINEER-TODO.md (not broken renders)

### Plugin Infrastructure

- [x] **PLUG-01**: All commands registered as .md files in `~/.claude/commands/doc/` with proper frontmatter
- [x] **PLUG-02**: Supporting files (references, templates) in `~/.claude/gsd-docs-industrial/`
- [x] **PLUG-03**: Subagent definitions for doc-writer, doc-verifier
- [x] **PLUG-04**: @-reference pattern for context injection from templates and references
- [x] **PLUG-05**: Configurable output language (Dutch/English) via PROJECT.md setting
- [x] **PLUG-06**: GSD-Docs runs alongside GSD without interference (separate /doc: namespace)
