## Cross-Phase Integration Issues

### 1. Timeline → Document Navigation Gap
**Phases:** 10 → 11 | **Requirements:** WORK-01, WORK-03

Phase timeline shows phase nodes with CLI commands. Document outline shows section tree. But there is no direct navigation link from clicking a phase in the timeline to seeing that phase's generated sections in the document view. Engineer must manually switch from Fases tab to Documenten tab.

**Impact:** UX friction — context switching between "where am I in the process" and "what content exists" requires manual tab navigation.

### 2. Review → Export Safety Gap
**Phases:** 12 → 13 | **Requirements:** QUAL-04, OUTP-02

No explicit gate prevents triggering FDS assembly while sections have "rejected" review status. An engineer could export a document with un-addressed review feedback.

**Impact:** Process safety — accidental export of unreviewed or rejected content.

## UX Flow Gaps (CLI ↔ GUI Handoff Points)

These are the specific friction points where the engineer switches between CLI and web:

### 1. No GUI-to-CLI Context Passing
When GUI shows "Volgende stap: `/doc:discuss-phase 3`", the engineer must manually copy this command and switch to their terminal. There's no integration with the terminal (no deep-link, no clipboard auto-copy on click).

**Mitigation already in place:** CliCommandBlock has a copy button with toast confirmation.

### 2. Silent GUI Updates After CLI Work
After the CLI writes results to disk (e.g., after `/doc:write-phase`), the GUI's 5-second polling picks up changes silently. There's no toast, no badge, no visual indicator that new content arrived.

**Risk:** Engineer finishes CLI work, switches to browser, and doesn't realize the content has already updated — or hasn't updated yet and thinks something is wrong.

### 3. No "CLI is Working" Indicator
While the CLI is running a long operation (e.g., `/doc:write-phase` with LLM calls), the GUI has no awareness. The engineer sees stale data with no indication that fresh content is on the way.

### 4. Review Decision → CLI Re-write Handoff
After rejecting sections in the review interface, the GUI shows the CLI command for re-writing. But the review feedback (text comments, approve/reject status) needs to be in a format the CLI can consume. The JSON export exists but the handoff UX (how the engineer gets their feedback into the CLI workflow) is unclear.
