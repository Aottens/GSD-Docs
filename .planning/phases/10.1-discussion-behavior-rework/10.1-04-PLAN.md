    Enhance `backend/app/services/context_generator.py`:

    1. Rework `generate()` to work with the new verbatim decision format:
       - Decisions are now dicts with: `{"topic": str, "question": str, "decision": str, "confirmed": bool, "notes": str?, "timestamp": str}`
       - Group decisions by topic
       - Only include confirmed decisions (confirmed == True) in the main decisions section
       - Unconfirmed decisions get a "## Pending Review" section at the bottom
       - Use engineer's EXACT decision text (verbatim — do NOT summarize or rephrase)
       - If notes exist on a decision, append as sub-bullet: `  - _Note: {notes}_`

    2. Implement the compression logic (replace the TODO truncation):
       - Priority 1: Decisions that change what gets written (KEEP — look for keywords: "should", "must", "will", "not", "instead")
       - Priority 2: Specific technical values — numbers, units, temperatures, capacities (KEEP)
       - Priority 3: General approach notes without specific values (COMPRESS — combine into single bullet per topic)
       - Use tables for parameter values when multiple numeric decisions exist for same topic
       - Target: under 100 lines

    3. Add `generate_preview(conversation_summary_data: dict, project_type: str) -> str`:
       - Same as generate() but returns content string without saving to disk
       - Used by the preview endpoint

    Update `backend/app/api/discussions.py`:

    4. Add `POST /{conversation_id}/preview-context` endpoint:
       - Generates CONTEXT.md preview from current conversation decisions
       - Returns `{"content": str, "line_count": int, "decisions_count": int, "unconfirmed_count": int}`
       - Does NOT save to disk or change conversation status

    5. Add `POST /{conversation_id}/finalize` endpoint:
       - Validates all decisions are confirmed (or warns about unconfirmed)
       - Generates final CONTEXT.md via context_generator
       - Saves CONTEXT.md to project's phase directory
       - Marks conversation as completed (status = "completed")
       - Returns `{"success": True, "context_file": str, "next_step": "planning", "message": "Discussion complete. Run /gsd:plan-phase to plan this phase."}`
       - Per user decision: "suggest next workflow step (planning), do NOT auto-advance"
       - Per user decision: "Clear context window between workflow steps" — conversation marked completed, new session for planning

    6. Update the existing `complete_discussion` endpoint to redirect to `finalize` (backwards compatibility)

    Update `backend/app/services/discussion_engine.py`:

    7. Add completion detection in `send_message()`:
       - When state.all_topics_complete() returns True after completing last topic:
         - Yield a `completion_card` event: `{"event": "completion_card", "data": {"message": "Alle geselecteerde onderwerpen zijn besproken.", "decisions_count": len(state.decisions), "topics_covered": state.completed_topics}}`
         - Transition state to completion phase
       - When in completion phase and user sends a message:
         - If message is "meer toevoegen" / "add more": transition back to discussion, allow freeform questions
         - If message is "bevestig" / "confirm": the frontend will call the preview-context endpoint

    8. Handle Foundation phase completion differently:
       - No topic tracking, so completion is not automatic
       - AI determines when intake is thorough enough and posts completion card
       - Parse completion_signal from LLM output to trigger completion card
