# Phase 10.1 Plan 03: Backend Discussion Engine Integration

**One-liner:** State-aware discussion orchestration with verbatim decision extraction and Foundation intake flow

## Overview

Integrated the conversation state machine and structured output parser (from Plan 01) into the discussion engine to implement AI-driven topic progression, verbatim decision capture, and Foundation phase detection. The existing discussion_engine.py was a simple pass-through (user message → LLM → stream response). This plan transformed it into a state-aware orchestrator that manages topic progression, extracts decisions faithfully, and emits structured SSE events.

**Context:** Phase 10.1-01 built the state machine and XML parser. This plan makes them operational by integrating them into the discussion flow.

## What Was Built

### 1. Verbatim Decision Extractor (decision_extractor.py)

**CRITICAL DESIGN DECISION:** This service NEVER calls the LLM. All extraction is rule-based to honor the "no interpretation" requirement from v1.0 discuss-phase.md.

**Core functions:**

- **`extract_verbatim_decision(user_message, current_topic, question_context)`**
  - Captures engineer's exact words with minimal cleanup
  - Removes filler words ONLY at sentence boundaries (leading "Um, basically", trailing "you know")
  - Keeps ALL technical content EXACTLY as stated
  - Returns decision dict: `{topic, question, decision, confirmed: False, timestamp}`
  - Returns None for meta-messages (conversation control)

- **`is_meta_message(message)`**
  - Detects conversation control messages that shouldn't be extracted as decisions
  - Recognizes: "next topic", "volgende onderwerp", "more questions", "meer vragen", "ga door", "continue"
  - Single-word affirmatives: "ja", "nee", "yes", "no" (check-in responses)

- **`trim_to_key_sentences(text, max_sentences=5)`**
  - If text has more than 5 sentences, keeps the ones with most technical content
  - Technical content scoring: counts units (°C, bar, L/min), technical terms (alarm, trip, valve, sensor)
  - Preserves original sentence order after selection
  - Removes excessive filler paragraphs while keeping engineer's words intact

**Example extraction:**
```
Input: "Well, the temperature sensor trips at 85 degrees, we want alarm only not shutdown, existing system has 90 degree setpoint"
Output: "the temperature sensor trips at 85 degrees, we want alarm only not shutdown, existing system has 90 degree setpoint"
```

Filler "Well" removed, all technical content ("85 degrees", "alarm only not shutdown", "90 degree setpoint") preserved verbatim.

### 2. Enhanced Discussion Engine (discussion_engine.py)

**State-aware orchestration throughout the conversation lifecycle:**

#### start_discussion() enhancements:

1. **Foundation phase detection:**
   - Uses `detect_foundation_phase(phase_number, phase_goal)` from Plan 01
   - If Foundation: initializes state with `is_foundation=True`, sets phase to `discussion`, skips topic selection
   - If NOT Foundation: initializes state to `topic_selection` phase

2. **Initial message creation:**
   - **Foundation:** Creates open-ended intake greeting: "Welkom bij de Foundation bespreking..." (Dutch) / "Welcome to the Foundation discussion..." (English)
   - **Regular phases:** Creates topic selection card with gray areas and discretion option (existing behavior)

3. **State persistence:**
   - Stores `state.to_summary_data()` in `conversation.summary_data` JSON column
   - State includes: phase, current_topic, questions_in_current_topic, selected_topics, decisions, etc.

#### send_message() complete rewrite:

**Phase-aware processing:**

1. **Load state:** `ConversationState.from_summary_data(conversation.summary_data)`

2. **topic_selection phase:**
   - Parse user's topic selection (which topics to discuss, which to delegate to discretion)
   - Transition to discussion phase: `state.confirm_topics(selected, discretion)`
   - Start first topic: `state.start_topic(first_topic)`
   - Yield `topic_boundary` event: `{"topic": first_topic, "status": "starting"}`

3. **discussion phase:**
   - Extract decision from user's answer using `extract_verbatim_decision()` (unless meta-message)
   - If decision extracted: yield `decision_captured` event with verbatim text, add to `state.decisions`
   - Increment question counter: `needs_checkin = state.increment_question()`
   - If >= 4 questions: transition to `check_in` phase

4. **check_in phase:**
   - Parse user's response: "next topic" → complete current topic, start next
   - "more questions" → stay on current topic, return to discussion phase
   - Yield `topic_boundary` events for completed and starting topics
   - If all topics complete: transition to `completion` phase

**Streaming with XML parsing:**

```python
buffer = StreamingXMLBuffer()
async for chunk in self.llm.stream_complete(messages, ...):
    buffer.feed(chunk)
    # Extract structured events (question_card, topic_boundary, completion_signal)
    for event in buffer.extract_events():
        yield event
    # Yield text content as message_delta
    yield {"event": "message_delta", "data": {"delta": chunk}}
```

**State persistence:**
- After stream complete: `conversation.summary_data.update(state.to_summary_data())`
- All decisions, topic progression, question counts persisted to DB

#### New helper methods:

- **`_build_state_aware_prompt(state, user_message, project_id)`**
  - Injects context into LLM prompts based on current state
  - Includes: current topic, questions asked, remaining topics, check-in instructions
  - For Foundation phase: includes reference file summaries (placeholder for now)

- **`_parse_topic_selection(content, available_topics)`**
  - Parses user's topic selection from message content
  - Simple heuristic: if "discretion" mentioned, split topics; otherwise all selected
  - Can be refined when frontend format is finalized

- **`_get_last_question(conversation_id)`**
  - Retrieves last assistant question for decision extraction context
  - Tries to parse `<question>` tag first, falls back to full content
  - Used by `extract_verbatim_decision()` to know what question the decision answers

- **`_load_reference_files(project_id)`**

## Implementation Notes

**Verbatim extraction design:**
- Rule-based processing honors "Interpretation is killing while writing technical documentation" principle
- Filler removal is minimal and conservative: only obvious sentence boundary fillers
- Technical content scoring uses regex patterns for units and common industrial terms
- No LLM call in decision extraction loop ensures faithful capture of engineer's words

**State machine orchestration:**
- State transitions are explicit: `topic_selection` → `discussion` → `check_in` → `discussion` (repeat) → `completion`
- 4-question rhythm is enforced but flexible (LLM can check-in earlier if topic exhausted)
- State persists in `conversation.summary_data` JSON column for durability across requests

**Foundation vs Regular phase:**
- Foundation phase uses `is_foundation` flag to bypass topic selection
- Foundation gets open-ended intake greeting instead of structured topic card
- Reference file loading (placeholder) will inject file summaries into Foundation prompts
- Regular phases follow structured topic progression as before

**Streaming architecture:**
- StreamingXMLBuffer accumulates chunks and extracts complete XML tags incrementally
- Structured events (question_card, topic_boundary) emitted as soon as tags are complete
- Text content with tags stripped continues to flow as message_delta events
- Frontend receives both structured events AND clean text simultaneously

**Decision capture timing:**
- Decisions extracted after EVERY user answer in discussion phase
- Meta-messages ("next topic", "more questions") skip decision extraction
- Each decision gets: topic context, question context, verbatim answer, timestamp
- Decisions accumulate in state.decisions and persist to DB
