# Phase 10.1 Plan 04: Backend Completion Workflow

**CONTEXT.md generation from verbatim decisions with priority-based 100-line compression, preview/finalize endpoints, and completion detection for structured and Foundation phases**

## Overview

Implemented the backend completion workflow that transforms verbatim decisions into structured CONTEXT.md output. The existing context_generator had basic decision formatting but no verbatim decision handling, no compression logic, and no preview capability. The discussion engine had state orchestration but no completion detection. This plan adds the full completion machinery: verbatim decision grouping with confirmation tracking, priority-based compression to meet 100-line limits, preview/finalize API endpoints, and completion detection for both structured topic-based phases and open-ended Foundation phases.

**Context:** Phase 10.1-03 implemented verbatim decision extraction and state-aware orchestration. This plan completes the backend by adding the output generation and completion workflow.

## What Was Built

### 1. Enhanced Context Generator (context_generator.py)

**Verbatim decision formatting:**

Reworked `_format_decisions()` to handle new decision format from Plan 10.1-03:
- Decision dict structure: `{topic, question, decision, confirmed, notes, timestamp}`
- Groups decisions by topic with topic headers (`### Operating parameters`)
- Separates confirmed and unconfirmed decisions
- Confirmed decisions in main section with verbatim text
- Unconfirmed decisions in "## Pending Review" section at bottom
- Notes appended as sub-bullets: `  - _Note: {notes}_`
- Legacy string format handled (treated as confirmed)

**Priority-based compression:**

Implemented `_compress_to_100_lines()` with 3-tier priority system:

**Priority 1: Prescriptive decisions (KEEP ALL)**
- Keywords: should, must, will, shall, not, instead, never, always, required, mandatory
- Regex: `\b(should|must|will|shall|not|instead|never|always|required|mandatory)\b`
- These decisions change what gets written — critical for engineering spec

**Priority 2: Numeric values (KEEP ALL)**
- Technical values with units: 85°C, 100 L/min, 5 bar, 120 mm, etc.
- Pattern: `\b\d+\.?\d*\s*(°C|°F|K|bar|psi|MPa|L/min|m³/h|kg/h|mm|cm|m|%|A|V|W|Hz|rpm)\b`
- Exact technical parameters — cannot be omitted

**Priority 3: General approach notes (COMPRESS)**
- No prescriptive keywords, no numeric values
- Combined into single bullet per topic: "General approach: {note1}; {note2}; {note3}"
- Reduces line count while preserving information

**Table formatting for dense numeric data:**
- If 3+ numeric values for same topic, convert to table:
  ```
  | Parameter | Value |
  | --- | --- |
  | Temperature setpoint | 85°C |
  | Flow rate nominal | 100 L/min |
  | Flow rate max | 120 L/min |
  ```
- More compact than bullet list for parameter-heavy topics

**generate_preview() method:**
- Same logic as `generate()` but returns content string without saving to disk
- Used by preview endpoint
- No side effects — conversation status unchanged

### 2. Preview/Finalize API Endpoints (discussions.py)

**POST /{conversation_id}/preview-context:**
- Generates CONTEXT.md preview from current conversation decisions
- Returns JSON:
  ```json
  {
    "content": "...",
    "line_count": 47,
    "decisions_count": 12,
    "unconfirmed_count": 2
  }
  ```
- No side effects: does not save to disk, does not change conversation status
- Engineer can see what CONTEXT.md will contain before finalizing

**POST /{conversation_id}/finalize:**
- Validates decisions (counts unconfirmed, adds warnings if any)
- Generates final CONTEXT.md via `context_generator.generate()`
- Saves CONTEXT.md to project's phase directory via `context_generator.save()`
- Marks conversation as completed (`status = "completed"`)
- Returns next step guidance:
  ```json
  {
    "success": true,
    "context_file": "projects/123/.planning/phases/03-equipment-modules/03-CONTEXT.md",
    "next_step": "planning",
    "message": "Discussion complete. Run /gsd:plan-phase to plan this phase.",
    "warnings": ["2 unconfirmed decision(s) will be marked as 'Pending Review' in CONTEXT.md"]
  }
  ```
- Per user decision: "suggest next workflow step (planning), do NOT auto-advance"
- Per user decision: "Clear context window between workflow steps" — conversation marked completed, new session for planning

**POST /{conversation_id}/complete (deprecated):**
- Redirects to `/finalize` for backwards compatibility
- Existing code calling this endpoint continues to work

### 3. Completion Detection in Discussion Engine (discussion_engine.py)

**Structured phase completion (topic-based):**

In `check_in` phase, when user says "next topic":
- Check if `state.all_topics_complete()` returns True (all selected topics in completed_topics)
- If True:
  - Transition to `completion` phase: `state.start_completion()`
  - Yield `completion_card` event:
    ```json
    {
      "event": "completion_card",
      "data": {
        "message": "Alle geselecteerde onderwerpen zijn besproken.",
        "decisions_count": 15,
        "topics_covered": ["Operating parameters", "Interlocks", "Safety"]
      }
    }
    ```
- Frontend receives event and displays completion card with preview/finalize buttons

**Foundation phase completion (LLM-driven):**

Foundation phases have no topic list, so completion is not automatic. AI determines when intake is thorough enough:

- LLM instructed to emit `<completion_signal>...</completion_signal>` XML tag when ready
- During streaming, `StreamingXMLBuffer.extract_events()` detects `completion_signal` event
- Set flag: `completion_signal_detected = True`
- After stream complete, if `state.is_foundation` and `completion_signal_detected`:
  - Transition to `completion` phase
  - Yield `completion_card` event:
    ```json
    {
      "event": "completion_card",
      "data": {
        "message": "Foundation bespreking is afgerond. De AI heeft genoeg informatie verzameld.",
        "decisions_count": 8,
        "topics_covered": ["Foundation"]
      }
    }
    ```

**Completion phase interactions:**

Added new phase handling for `ConversationPhase.completion`:
- User message: "meer toevoegen" / "add more" → transition back to `discussion` phase, set `current_topic = "Additional Topics"`
- User message: "bevestig" / "confirm" → frontend calls `preview-context` endpoint (no backend action needed)

## Decisions Made

**Verbatim decision format:**
- Decisions are dicts with: `topic, question, decision, confirmed, notes, timestamp`
- Confirmed decisions in main section, unconfirmed in "Pending Review"
- Notes appended as sub-bullets when present
- Preserves engineer's exact words (no AI summarization)

**Compression strategy:**
- Priority 1 (prescriptive keywords) > Priority 2 (numeric values) > Priority 3 (general notes)
- Table format for 3+ numeric values in same topic
- Target: under 100 lines (v1.0 Pitfall 7 mitigation)

- `preview-context` endpoint returns content-only response
- No file saved, no conversation status changed
- Engineer can review multiple times before finalizing

**Completion workflow:**
- Finalize saves CONTEXT.md, marks conversation completed, suggests planning
- Does NOT auto-advance to planning (per user decision)
- Clear context window between workflow steps (new session for planning)

**Completion detection:**
- Structured phases: `all_topics_complete()` triggers `completion_card` event
- Foundation phases: LLM emits `<completion_signal>` when intake is thorough
- Both emit same event type for consistent frontend handling

## Implementation Notes

**Compression logic design:**
- Regex-based classification into 3 priority tiers
- Prescriptive keywords (should, must, will, not) identify decisions that change behavior
- Numeric patterns with units (85°C, 100 L/min) capture exact technical specs
- General notes without keywords or numbers get compressed
- Table formatting reduces line count for parameter-heavy topics

**Preview/finalize separation:**
- Preview is read-only: returns content, no DB writes
- Finalize is write-once: saves file, updates status, returns next step guidance
- Frontend can call preview multiple times, finalize only once

**Completion detection for Foundation:**
- Foundation phases have no topic list, so automatic completion not possible
- LLM instructed to emit XML tag when intake is thorough
- XML tag parsed by existing StreamingXMLBuffer from Plan 10.1-01
- Consistent event format for frontend regardless of phase type

**Next step guidance:**
- Finalize endpoint returns `next_step: "planning"` with message
- Does NOT trigger planning automatically (per user decision: "suggest, do NOT auto-advance")
- Engineer manually starts planning when ready
- Clear context window between workflow steps (new conversation for planning)
