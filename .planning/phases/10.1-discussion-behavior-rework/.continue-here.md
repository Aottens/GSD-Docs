<architectural_problem>

## The Core Issue: "LLM drives" vs "Code drives"

### How v1.0 works (correct pattern):
In v1.0, `discuss-phase.md` is a workflow that Claude Code executes PROCEDURALLY:
- Step 5.1: Claude generates ONE thing — topic announcement
- Step 5.2: Claude generates ONE thing — 3-5 questions (then stops, waits via AskUserQuestion)
- Step 5.3: Claude generates ONE thing — decision summary
Each step is a SCOPED output. The workflow controls WHEN to ask, WHEN to summarize, WHEN to move on.

### How v2.0 works (broken pattern):
`discussion_engine.py:send_message()` does this every turn:
1. Update state (count questions, check phase)
2. Build FULL message history
3. Inject small state context hint
4. Call `stream_complete()` with ALL context → stream whatever LLM generates

The LLM is expected to drive the conversation based on system prompt instructions. The state machine OBSERVES but does not CONTROL the LLM output. Behavior depends entirely on which model is used and how well it follows instructions.

### Why this fails:
- Different models interpret prompts differently (7b was verbose, 14b is better, Claude would be different again)
- The system prompt says "ask ONE question" but nothing enforces it
- max_tokens and temperature are band-aids, not architecture
- Over long conversations, any model can drift from prompt instructions
- Foundation phase completion relies on LLM emitting `<completion_signal>` XML tag — fragile

### What we need (v1.0-faithful pattern):
**Backend drives, LLM generates scoped content.** Instead of one big `stream_complete()` per turn, make SMALL SCOPED LLM calls:
- "Generate ONE follow-up question about {topic} based on: {user_answer}" → returns one question
- "Summarize these 4 decisions into bullet points" → returns summary
- "Has the engineer provided enough info about {topic}?" → returns yes/no signal

The state machine already has the right phases. It needs to DRIVE the LLM, not ADVISE it.

## What needs to CHANGE:

### `discussion_engine.py:send_message()` — the orchestration core
Current: builds full history → one big LLM call → hopes for good output
Needed: state machine drives → small scoped LLM calls → deterministic flow

### `discuss_phase.py` prompts
Current: DISCUSSION_SYSTEM_PROMPT is a big "be a good discussion partner" prompt
Needed: Small scoped prompt templates:
- GENERATE_QUESTION_PROMPT: "Given topic {X} and answer {Y}, generate ONE follow-up question"
- SUMMARIZE_TOPIC_PROMPT: "Summarize these decisions for topic {X}: {decisions}"
- CHECK_SUFFICIENCY_PROMPT: "Has enough been captured about {topic}? Decisions so far: {list}"
- FOUNDATION_QUESTION_PROMPT: "Generate ONE intake question about {area}. Previous answers: {context}"

### Check-in behavior
Current: state machine sets phase=check_in, then injects hint to LLM prompt
Needed: backend directly emits check_in SSE event with "Continue or next topic?" — no LLM call needed

### Topic boundaries
Current: LLM expected to emit `<topic_boundary>` XML tag
Needed: backend emits topic_boundary event when state.complete_topic() is called — deterministic

### Completion detection
Current: Structured phases use state machine (good). Foundation uses LLM `<completion_signal>` (fragile).
Needed: Foundation should use backend heuristic (question count + coverage of 5 intake areas) OR a small scoped LLM call "is intake sufficient?"
