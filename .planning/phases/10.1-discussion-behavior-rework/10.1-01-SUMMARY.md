# Phase 10.1 Plan 01: Conversation State Machine & Structured Output Parser

**One-liner:** State machine orchestration with XML-tagged question card parsing and Foundation phase intake flow

## Overview

Built the conversation state machine and structured output parser that form the orchestration backbone for AI-driven discussion behavior. The existing discussion_engine.py streams LLM responses but has no state tracking (topic progression, completion detection) or structured output parsing. This plan adds the state machine that enforces workflow boundaries (topic_selection → discussion → completion) and the XML parser that converts LLM output into question_card SSE events.

**Context:** Phase 10 implemented the infrastructure (SSE streaming, Conversation/Message models, LLM abstraction). Phase 10.1-01 adds the discussion flow logic.

## What Was Built

### 1. Conversation State Machine (conversation_state.py)

**ConversationState dataclass** tracks discussion flow with enforced transitions:
- **Phases:** `topic_selection`, `discussion`, `check_in`, `completion`
- **Topic tracking:** `selected_topics`, `completed_topics`, `current_topic`
- **Question rhythm:** `questions_in_current_topic` counter, `increment_question()` returns True at 4 questions for check-in
- **Decision accumulation:** `decisions` list (verbatim captures), `deferred_ideas` for scope creep
- **Serialization:** `to_summary_data()` / `from_summary_data()` for DB persistence in Conversation.summary_data JSON column

**State transition methods:**
- `start_topic_selection()` → sets phase to topic_selection
- `confirm_topics(selected, discretion)` → transitions to discussion
- `start_topic(topic)` → sets current_topic, resets question counter
- `increment_question()` → bumps counter, returns True if >= 4 (check-in time)
- `complete_topic()` → moves current_topic to completed_topics
- `start_completion()` → transitions to completion phase
- `next_topic()` → returns next unprocessed topic from selected_topics
- `all_topics_complete()` → True if all selected_topics are in completed_topics

**Foundation phase detection:**
- `detect_foundation_phase(phase_number, phase_goal)` function
- Returns True if `phase_number == 1` OR phase_goal contains "foundation", "intake", "scope" keywords
- Per user decision and v1.0 discuss-phase.md Step 2.2

### 2. Structured Output Parser (structured_output_parser.py)

**XML tag parsers:**
- `parse_question_card(text)` → extracts `<question>...</question>` and `<options><option>...</option></options>`
- `parse_topic_boundary(text)` → extracts `<topic_boundary topic="..." status="..." />`
- `parse_completion_signal(text)` → extracts `<completion_signal>...</completion_signal>`
- `strip_xml_tags(text)` → removes all recognized tags for clean message_delta events

**StreamingXMLBuffer class** for incremental SSE parsing:
- `feed(chunk)` → accumulates text across chunk boundaries (handles `<quest` in one chunk, `ion>...</question>` in next)
- `extract_events()` → returns complete structured events as dicts: `{"type": "question_card", "data": {...}}`
- Tracks `extracted_positions` to avoid duplicate event emission
- `get_text_content()` → returns clean text without XML tags for message_delta events

### 3. Foundation Phase Prompt Templates (discuss_phase.py)

**FOUNDATION_INTAKE_PROMPT:**
- Open-ended intake for Phase 1 (not structured topic cards)
- Asks about: system description, reference docs, scope boundaries, equipment structure, terminology
- "Ask 2-3 questions at a time, then wait" pattern
- Language-aware (`{project_language}`)
- Instructs LLM: "Do NOT use topic cards or structured question format for Foundation phase"

**TOPIC_CHECK_IN_PROMPT:**
- After ~4 questions: "We have covered {n} questions about {topic}. Would you like to continue or move to next topic?"

**Updated DISCUSSION_SYSTEM_PROMPT:**
- Added XML output instructions:
  ```
  <question>Question text here</question>
  <options>
    <option>Option 1</option>
    <option>Option 2</option>
  </options>
  ```
- Topic boundary markers: `<topic_boundary topic="..." status="complete" />`
- Completion signal: `<completion_signal>All selected topics covered.</completion_signal>`

**Updated build_system_prompt():**
- Detects Foundation phase (phase_number == 1 OR phase_goal contains keywords)
- Returns FOUNDATION_INTAKE_PROMPT for Foundation, DISCUSSION_SYSTEM_PROMPT otherwise
- Foundation prompt skips gray area generation and topic selection

### 4. Model Updates (conversation.py, schemas.py)

**MessageType enum additions:**
- `completion_card` → CONTEXT.md preview + confirm
- `topic_boundary` → topic completion markers
- `decision_captured` → real-time decision capture events
- `check_in` → 4-question rhythm check-ins

**ConversationPhase enum** (added to models for type consistency):
- Matches state machine phases: `topic_selection`, `discussion`, `check_in`, `completion`

**Pydantic schemas for SSE events:**
- `QuestionCardEvent(question, options)`
- `DecisionCapturedEvent(topic, question, decision, confirmed, timestamp)`
- `TopicBoundaryEvent(topic, status)`
- `CompletionSignalEvent(message)`
- `CheckInEvent(topic, questions_asked)`

## Implementation Notes

**State machine design:**
- Enforces workflow boundaries: cannot skip topic_selection, cannot jump to completion without completing all selected topics
- 4-question rhythm is a soft guideline (LLM can check-in earlier if topic is exhausted)
- Foundation phase uses `is_foundation` flag to skip topic selection in discussion_engine integration

**XML parsing approach:**
- Regex-based for simple tags (no full XML parser needed per research)
- StreamingXMLBuffer handles partial tags across chunks by accumulating and tracking extraction positions
- `extracted_positions` list prevents duplicate events when buffer is fed multiple times

**Foundation vs Regular phase:**
- Foundation: open-ended Q&A, no topic cards, no gray area generation
- Regular: structured topic selection → discussion → check-in → completion
- Detection is automatic in `build_system_prompt()` — discussion_engine doesn't need special handling

**Dutch language support:**
- All prompt templates support `{project_language}` variable
- FOUNDATION_INTAKE_PROMPT and TOPIC_CHECK_IN_PROMPT will render in Dutch when project_language == "nl"
- LLM instructed to use project language for all questions
