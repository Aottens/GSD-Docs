## Accomplishments
- `increment_question()` refactored to return string with three tiers: 'continue' / 'check_in' (at 4) / 'force_check_in' (hard cap at 12)
- `detect_covered_area_with_fallback()` added: keyword match first, falls back to cycling uncovered areas after 2+ questions
- `decision_captured` SSE events removed from `_handle_discussion()` — decisions now accumulate silently in `state.decisions`
- Check-in event payload expanded to include `decisions` (topic-filtered) and `all_decisions` (full list) for Beslissingen tab
- `GENERATE_FOUNDATION_QUESTION_PROMPT` updated with anti-repetition rules and `"options"` field in JSON response format
- `GENERATE_QUESTION_PROMPT` updated with anti-repetition rules including graceful exit when out of new questions

## Decisions Made
- `increment_question()` now returns a string enum ('continue'/'check_in'/'force_check_in') rather than bool — enables distinct handling of normal check-in vs forced check-in from hard cap
- Foundation area detection uses keyword match + question-count cycling fallback — ensures area progress even when real Q&A text doesn't match keyword lists
- Decision accumulation is fully silent during discussion phase — no interruptions to conversation flow; decisions revealed as a bundle at check-in
- Check-in SSE event carries `decisions` and `all_decisions` so frontend can update the Beslissingen tab at the natural check-in boundary
- Foundation questions now use the same `{"question": ..., "options": [...]}` JSON response format as regular topic questions — no special-case parsing required
