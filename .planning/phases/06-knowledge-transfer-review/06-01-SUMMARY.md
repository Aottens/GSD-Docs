# Phase 06 Plan 01: Knowledge Transfer Templates + Discuss-Phase Enhancement

**One-liner:** Created 4 knowledge transfer templates (RATIONALE.md for decision logging, EDGE-CASES.md for per-phase edge case documentation, FRESH-EYES.md for multi-perspective review findings, REVIEW.md for structured feedback) and enhanced discuss-phase workflow with automatic RATIONALE.md capture after each discussion topic

## What Was Built

### Task 1: 4 Knowledge Transfer Templates

Created 4 template files following the established HTML comment doc block + markdown structure pattern used by verification.md, cross-refs.md, and summary.md.

**rationale.md template:**
- Project-wide single file in .planning/RATIONALE.md (not per phase)
- Entries organized by FDS section reference (§X.Y) for easy lookup
- 5-field structure: Decision (1-2 sentences), Reasoning (2-3 sentences), Alternatives (what was ruled out and why), Date (YYYY-MM-DD), Phase (N)
- Multi-section decision pattern: single entry under most relevant section with "Also applies to:" cross-reference
- Completeness over curation: log every significant decision from discuss-phase
- Example entries showing equipment module decisions (E-stop behavior, protocol selection)

**edge-cases.md template:**
- Per-phase file in .planning/phases/{NN}-{name}/{NN}-EDGE-CASES.md
- 3 severity sections: CRITICAL (safety/equipment damage), WARNING (operational impact), INFO (notable behavior)
- 5-column table format: Situation, System Behavior, Recovery Steps, Design Reason, Section
- Design Reason column is REQUIRED (not optional) per user decision - edge cases without design reason are incomplete
- CRITICAL edge cases stand out visually with blockquote warning box format in addition to table entry
- Cross-reference note: edge cases referencing equipment from other phases trigger automatic CROSS-REFS.md entry

**fresh-eyes.md template:**
- Per-phase file generated by fresh-eyes subagent after verify-phase PASS
- Summary table: Category (Comprehension Gaps / Completeness Gaps) × Severity (MUST-FIX / SHOULD-FIX / CONSIDER)
- Separate sections per perspective: Engineer Perspective, Customer Perspective, Operator Perspective (only present if that perspective was selected)
- Each perspective has Comprehension Gaps and Completeness Gaps subsections
- Finding format: Section-Ref: SEVERITY - Brief Description, then What's unclear/missing, Context, Why it matters
- Routing to Gap Closure section: actionable flag, findings routed count, fix plan IDs
- Severity calibration guidance: stricter for safety-critical sections, lighter for overview

**review.md template:**
- Per-phase file generated by review-phase workflow (interactive section-by-section review)
- Summary table: Status (Approved / Comment / Flag) × Count
- Main feedback table: Section, Status, Finding, Suggested Action
- Detailed "Flagged Issues (Detail)" section with Severity, Finding, Context, Suggested Action, Routed to gaps
- Review Progress section for multi-session resume support: "Sections reviewed: X of Y. Next: section-ref"
- Resolution approach section: default manual, --route-gaps flag sends findings to gap closure pipeline

All templates follow the exact pattern of existing templates: HTML comment at top explaining who creates/consumes, placeholder variables in {curly braces}, clear structural guidance.

### Task 2: Enhance Discuss-Phase Workflow with RATIONALE.md Capture

Added a new Step 5.1.4a to the discuss-phase workflow, inserted AFTER each discussion topic completes (not after all topics). This step runs within Step 5.1 "Discussion Flow Per Topic", after item 4 ("After each question batch, summarize decisions").

**New substep 5.1.4a: Update RATIONALE.md**

After summarizing decisions for a topic, extract and log to RATIONALE.md:

1. **Check if RATIONALE.md exists:**
   - If not: create from template at `~/.claude/gsd-docs-industrial/templates/rationale.md`, write to `.planning/RATIONALE.md`
   - Per user decision: single project-wide file, not per-phase
   - Replace placeholder variables (Project Name, Date)

2. **Extract decision context from the just-discussed topic:**
   - Decision statement (what was decided)
   - Reasoning (why this choice - engineer's explanation)
   - Alternatives considered (what was ruled out and why)
   - FDS section reference (which section this decision applies to, derived from discussion context)

3. **Check for existing section entry in RATIONALE.md:**
   - If section heading already exists (from previous phase's discuss-phase): append under existing section heading as new sub-entry
   - If new section: create section heading + entry
   - Per user decision: entries organized by FDS section, not by phase or chronology

4. **Append structured entry to RATIONALE.md:**
   Format per template:
   ```
   ### Decision Title
   **Decision:** [1-2 sentences]
   **Reasoning:** [2-3 sentences]
   **Alternatives:** [What was ruled out, 1-2 sentences]
   **Date:** YYYY-MM-DD
   **Phase:** N
   ```

5. **Entry boundary guidance (Claude's discretion):**
   - If one discussion answer covers a single topic: one entry
   - If one answer covers multiple distinct decisions: split into separate entries under same section
   - If decision spans multiple sections: single entry under most relevant section with "Also applies to: [other sections]"

6. **Display confirmation:**
   ```
   📝 RATIONALE.md updated:
      [Section Ref] Decision Title
      -- Decision, reasoning, and alternatives logged
   ```

7. **Continue to next discussion topic.**

**Also added to Step 7.3 (Completion):**
In Step 7.3 "Display Summary", added a line for RATIONALE entries:
```
RATIONALE entries: {count} logged to .planning/RATIONALE.md
```

**Also added to Workflow Rules section:**
Rule 13: "RATIONALE.md capture: After each discussion topic completes, log decisions to .planning/RATIONALE.md. Every significant decision is logged (completeness over curation). RATIONALE.md is project-wide, organized by FDS section reference."

**IMPORTANT:** Did NOT restructure or rewrite the existing discuss-phase workflow. Inserted the new step cleanly into the existing structure. The existing 7-step flow remains intact - this adds a substep within Step 5 and minor additions to Steps 7 and Workflow Rules.
