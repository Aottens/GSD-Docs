Create the review-phase workflow file with comprehensive interactive review logic. Target: ~400-500 lines following the established workflow pattern (XML workflow wrapper, numbered steps, clear substeps).

**Workflow structure:**

### Step 1: Parse Arguments and Validate Phase

1.1 Parse phase number and flags from $ARGUMENTS:
- Phase number (required)
- --route-gaps flag (optional: route flagged issues to gap closure)
- --resume flag (optional: resume partial review from REVIEW.md)

1.2 Read ROADMAP.md to find phase entry. Extract phase name and goal. Error if phase not found.

1.3 Read PROJECT.md for language setting.

1.4 Check phase has content:
- Look for *-CONTENT.md files in phase directory
- If none found: error "Phase {N} has no content to review"
- Optionally check for VERIFICATION.md with PASS status (warn if not verified, but allow review of unverified content)

1.5 Display banner:
```
DOC > REVIEWING PHASE {N}
```

### Step 2: Load Phase Content

2.1 Discover all CONTENT.md and SUMMARY.md files in phase directory.

2.2 Build section list in order (sorted by plan number).

2.3 For each section, load:
- CONTENT.md file path and size
- SUMMARY.md file path
- Cross-references from CROSS-REFS.md where this section is source or target

2.4 Count total sections.

2.5 Display overview:
```
Review: {section_count} sections in Phase {N}: {Phase Name}
Type: Engineer handover / Client walkthrough
```

### Step 3: Check for Resume State

3.1 If --resume flag provided OR REVIEW.md already exists:
- Read REVIEW.md
- Parse "Review Progress" section for last reviewed section
- Calculate which sections are already reviewed
- Display:
  ```
  Resuming review from section {next_section} ({reviewed}/{total} already reviewed)
  ```
- Set starting index to next unreviewed section

3.2 If no --resume and REVIEW.md exists:
- Ask engineer: "Previous review found. Resume or start fresh?"
- If resume: same as 3.1
- If fresh: initialize new REVIEW.md from template

3.3 If no REVIEW.md: initialize from template at `~/.claude/gsd-docs-industrial/templates/review.md`

### Step 4: Interactive Section-by-Section Review

For each section (starting from resume point or first section):

4.1 Display section header:
```
Section {current}/{total}: {section_name}
```

4.2 Display SUMMARY.md content:
Read and display SUMMARY.md for this section. Show all 4 sections (Facts, Key Decisions, Dependencies, Cross-refs).

4.3 Display cross-references:
Show cross-references where this section is source or target. Format:
```
CROSS-REFERENCES:
  Depends on: {target sections}
  Referenced by: {source sections}
```

4.4 Display CONTENT.md (paginated):
- If content < 60 lines: show full content
- If content >= 60 lines: show first 40 lines + "... ({total} lines total)"
- Offer "View full content" option in feedback selection

4.5 Collect feedback via AskUserQuestion:
```
Use AskUserQuestion:
  header: "Section {current}/{total}: {section_name}"
  question: "Review status?"
  options:
    - "Approved (no issues)"
    - "Comment (minor note)"
    - "Flag (needs revision)"
    - "View full content"
    - "Skip to next"
```

4.6 Handle response:

**If "Approved":**
- Record in REVIEW.md: | {section_id} | Approved | - | - |
- Display: "Section approved"
- Continue to next section

**If "Comment":**
- Follow up with AskUserQuestion for comment text (free text input via inline conversation)
- Record in REVIEW.md: | {section_id} | Comment | {comment text} | - |
- Display: "Comment logged"
- Continue to next section

**If "Flag":**
- Follow up with inline conversation for:
  - Issue description (what needs revision)
  - Severity selection via AskUserQuestion: "High (blocks approval)" / "Medium (should fix)" / "Low (nice to have)"
  - Suggested action (optional, inline conversation)
- Record in REVIEW.md feedback table: | {section_id} | Flag | {issue} | {suggested_action} |
- Record in REVIEW.md Flagged Issues (Detail) section with full context
- Display: "Issue flagged ({severity})"
- Continue to next section

**If "View full content":**
- Display full CONTENT.md
- Return to feedback selection (re-present options)

**If "Skip":**
- Do not record in REVIEW.md
- Continue to next section

4.7 Review fatigue check (per research pitfall 4):
After every 10 sections, if more sections remain:
```
Use AskUserQuestion:
  header: "Review Progress"
  question: "You have reviewed {reviewed} of {total} sections. Continue?"
  options:
    - "Continue reviewing"
    - "Save progress and stop"
```

If "Save progress": Update REVIEW.md with progress section and exit to Step 6 with partial results.

4.8 Update REVIEW.md "Review Progress" section after each section:
```
Sections reviewed: {N} of {total}. Next: {next_section_id}
```

### Step 5: Route to Gap Closure (Optional)

5.1 Count flagged issues.

5.2 If --route-gaps flag AND flagged issues exist:
- Display preview of all flagged issues:
  ```
  Review complete. {flag_count} issues flagged.

  Flagged issues:
  1. [{severity}] {section}: {brief description}
  2. [{severity}] {section}: {brief description}
  ...
  ```

- Per research pitfall 5: always preview before routing. Ask confirmation:
  ```
  Use AskUserQuestion:
    header: "Route to Gap Closure"
    question: "Generate fix plans for these issues?"
    options:
      - "All flagged issues"
      - "Select specific issues"
      - "Skip -- resolve manually"
  ```

- If "Select specific": present numbered list, let engineer select which to route
- If "All" or "Select": display instruction:
  ```
  Run: /doc:plan-phase {N} --gaps --source review
  Then: /doc:write-phase {N} to execute fixes
  Then: /doc:verify-phase {N} to re-verify
  ```

5.3 If no --route-gaps flag AND flagged issues exist:
```
{flag_count} issues flagged. Resolve manually or re-run with --route-gaps to generate fix plans.
```

### Step 6: Review Summary and Completion

6.1 Display completion banner:
```
DOC > PHASE {N} REVIEWED
```

6.2 Display summary:
```
Sections reviewed: {reviewed_count}
  Approved: {approved_count}
  Comments: {comment_count}
  Flagged: {flag_count}
  Skipped: {skipped_count}

Feedback: {REVIEW.md path}
```

6.3 If partial review (stopped early):
```
Review Progress: {reviewed}/{total} sections
Resume: /doc:review-phase {N} --resume
```

6.4 Next Up block:
```
## > Next Up

**Phase {N+1}: {Next Phase Name}** -- {goal}

`/doc:discuss-phase {N+1}`

Also available:
- `/doc:verify-phase {N} --perspective engineer` -- Fresh Eyes review
- `/doc:review-phase {N} --resume` -- continue review
- `/doc:status` -- view overall progress
```

### Workflow Rules section:
1. Language consistency: all user-facing text matches project language
2. DOC > prefix on all banners
3. Interactive mode: AskUserQuestion for selections, inline conversation for free-text feedback
4. REVIEW.md is per-phase (in phase directory)
5. Review does not change verification status -- review is supplementary to verify-phase
6. Partial review: engineer can stop and resume at any point
7. Fatigue check: offer pause every 10 sections
8. Gap closure routing: optional via --route-gaps, always with preview before routing
9. Unverified content: warn but allow review (review can happen before or after verify-phase)
