  <action>
Create 4 template files following the established template pattern (HTML comment doc block + markdown structure) used by verification.md, cross-refs.md, and summary.md.

**rationale.md template:**
- HTML comment doc block: generated by discuss-phase workflow, consumed by plan-phase and write-phase, project-wide single file in .planning/
- Header: Phase {N}: {Name} with date
- Entries organized by FDS section reference (e.g., ## Section Reference heading, ### Decision Title subheading)
- Each entry has 5 fields: **Decision:** (1-2 sentences), **Reasoning:** (2-3 sentences), **Alternatives:** (what was ruled out and why, 1-2 sentences), **Date:** (YYYY-MM-DD), **Phase:** (N)
- Per user decision: completeness over curation -- log every decision from discuss-phase
- Include "Also applies to:" pattern for multi-section decisions (single entry under most relevant section, cross-reference to others)
- Example entries showing equipment module decisions (from research examples)

**edge-cases.md template:**
- HTML comment doc block: generated by doc-writer subagents via temporary files, aggregated by write-phase orchestrator, per-phase file in phase directory
- Header: Phase {N}: {Name} - Edge Cases with created/updated dates
- 3 severity sections with clear headings: ## CRITICAL (Safety / Equipment Damage), ## WARNING (Operational Impact), ## INFO (Notable Behavior)
- Per user decision: CRITICAL edge cases stand out visually -- use blockquote warning box format for CRITICAL entries in addition to table
- Table format per severity section: | Situation | System Behavior | Recovery Steps | Design Reason | Section |
- Include "Design Reason" column as REQUIRED (not optional) -- per user decision, edge cases without design reason are incomplete
- Include note about cross-reference: edge cases referencing equipment from other phases trigger automatic CROSS-REFS.md entry

**fresh-eyes.md template:**
- HTML comment doc block: generated by fresh-eyes subagent after verify-phase PASS, per-phase file
- Header: Phase {N}: {Name} - Fresh Eyes Review with date, perspective, reviewer fields
- Summary table: Category | MUST-FIX | SHOULD-FIX | CONSIDER | Total (rows: Comprehension Gaps, Completeness Gaps, Total)
- Separate sections per perspective: ## Engineer Perspective, ## Customer Perspective, ## Operator Perspective (present only if that perspective was selected)
- Each perspective has: ### Comprehension Gaps and ### Completeness Gaps subsections
- Finding format: #### Section-Ref: SEVERITY - Brief Description, then **What's unclear/missing:**, **Context:**, **Why it matters:**
- Footer: ## Routing to Gap Closure section (actionable flag, findings routed count, fix plan IDs)

**review.md template:**
- HTML comment doc block: generated by review-phase workflow, per-phase file
- Header: Phase {N}: {Name} - Review Feedback with date, reviewer, type fields
- Summary table: | Status | Count | (rows: Approved, Comment, Flag, Total Sections)
- Main feedback table: | Section | Status | Finding | Suggested Action |
- Detailed section: ## Flagged Issues (Detail) with per-issue entries (Severity, Finding, Context, Suggested Action, Routed to gaps)
- Include "Review Progress" section for multi-session resume support: "Sections reviewed: X of Y. Next: section-ref"

All templates follow the exact pattern of existing templates: HTML comment at top explaining who creates/consumes, placeholder variables in {curly braces}, clear structural guidance.

**Integration point:** Inside Step 5.1 "Discussion Flow Per Topic", after item 4 ("After each question batch, summarize decisions"), add a substep that updates RATIONALE.md.

**New substep 5.1.4a: Update RATIONALE.md**

After summarizing decisions for a topic, extract and log to RATIONALE.md:

1. **Check if RATIONALE.md exists:**
   - If not: create from template at `~/.claude/gsd-docs-industrial/templates/rationale.md`, write to `.planning/RATIONALE.md`
   - Per user decision: single project-wide file, not per-phase

2. **Extract decision context from the just-discussed topic:**
   - Decision statement (what was decided)
   - Reasoning (why this choice -- engineer's explanation)
   - Alternatives considered (what was ruled out and why)
   - FDS section reference (which section this decision applies to, derived from discussion context)

3. **Check for existing section entry in RATIONALE.md:**
   - If section heading already exists (from previous phase's discuss-phase): append under existing section heading as new sub-entry
   - If new section: create section heading + entry
   - Per user decision: entries organized by FDS section, not by phase or chronology

4. **Append structured entry to RATIONALE.md:**
   Format per template:
   ```
   ### Decision Title
   **Decision:** [1-2 sentences]
   **Reasoning:** [2-3 sentences]
   **Alternatives:** [What was ruled out, 1-2 sentences]
   **Date:** YYYY-MM-DD
   **Phase:** N
   ```

5. **Entry boundary guidance (Claude's discretion):**
   - If one discussion answer covers a single topic: one entry
   - If one answer covers multiple distinct decisions: split into separate entries under same section
   - If decision spans multiple sections: single entry under most relevant section with "Also applies to: [other sections]"

6. **Display confirmation:**
   ```
   RATIONALE.md updated:
      [Section Ref] Decision Title
      -- Decision, reasoning, and alternatives logged
   ```

7. **Continue to next discussion topic.**

**Also add to Step 7 (Completion):**
In Step 7.3 "Display Summary", add a line for RATIONALE entries:
```
RATIONALE entries: {count} logged to .planning/RATIONALE.md
```

**Also add to Workflow Rules section:**
Rule 13: "RATIONALE.md capture: After each discussion topic completes, log decisions to .planning/RATIONALE.md. Every significant decision is logged (completeness over curation). RATIONALE.md is project-wide, organized by FDS section reference."

**IMPORTANT:** Do NOT restructure or rewrite the existing discuss-phase workflow. Insert the new step cleanly into the existing structure. The existing 7-step flow remains intact -- this adds a substep within Step 5 and minor additions to Steps 7 and Workflow Rules.
