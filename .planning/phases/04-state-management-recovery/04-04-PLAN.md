<objective>
Create the /doc:expand-roadmap command for dynamic ROADMAP expansion and add auto-trigger logic to verify-phase for System Overview completion.

Purpose: Real projects discover complexity during writing. When System Overview identifies >5 equipment units, the original ROADMAP phases become too large for single write cycles. This command lets engineers (or the auto-trigger) propose, review, and insert new decimal-numbered phases (4.1, 4.2, etc.) that break down the work into manageable chunks of 3-5 units per phase.

Output:
- commands/doc/expand-roadmap.md (slash command entry point)
- gsd-docs-industrial/commands/expand-roadmap.md (version-tracked copy)
- gsd-docs-industrial/workflows/expand-roadmap.md (interactive expansion workflow)
- Enhanced verify-phase.md workflow (auto-trigger after System Overview PASS)
</objective>

**Step 1: Detect Units and Determine Insertion Point**

Two entry paths:
1. **Manual invocation:** `/doc:expand-roadmap [after-phase]`
   - If after-phase provided: use that phase number
   - If not provided: scan ROADMAP.md for the first phase with status "Written" or "Verified" that has equipment unit content, use that phase
2. **Auto-trigger:** Called from verify-phase after System Overview PASS (receives unit list and after-phase via context)

Read the phase's CONTENT.md files and SUMMARY.md files to extract identified units:
- Parse equipment module references: look for patterns like "EM-NNN", "Equipment Module", unit names
- Build a unit list with names and descriptions
- Count total units identified

If unit count <= 5:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

{unit_count} units identified (threshold: >5)

No expansion needed. Current ROADMAP phases can accommodate this scope.
```

STOP execution.

If unit count > 5: proceed to Step 2.

**Step 2: Propose Phase Groupings**

Claude's discretion on grouping strategy (per context decision). Available strategies:
- **Process area:** Group units by their function (mixing, filling, transport, packaging, etc.)
- **Dependency:** Group units that reference each other into the same phase
- **Complexity:** Balance phases by estimated writing effort
- **Mixed:** Combine strategies as appropriate for the project

Grouping constraints (locked decisions):
- 3-5 units per phase group
- Maximum 7 new phases total
- Each unit must appear in exactly one group

Generate suggested groupings:
```
groups = [
  { number: "{after_phase}.1", name: "{descriptive name}", units: [...], rationale: "..." },
  { number: "{after_phase}.2", name: "{descriptive name}", units: [...], rationale: "..." },
  ...
]
```

Display overview:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```

**Step 3: Interactive Approval Loop**

For each proposed group (locked decision: "system suggests groupings one by one, engineer approves/modifies each group"):

Display:
```
───────────────────────────────────────────────────────────────
Proposed Phase {number}: {name}

  Units: {unit list}
  Rationale: {rationale}

  Options:
  1. Approve (use as-is)
  2. Modify (change name or units)
  3. Skip (merge units into next group)
───────────────────────────────────────────────────────────────
```

Use AskUserQuestion for each group. Process responses:
- **Approve:** Add group to approved list as-is
- **Modify:** Ask for new name (default: current name). Ask for units (comma-separated, default: current units). Add modified group to approved list.
- **Skip:** Move this group's units into the next group. If this is the last group, warn: "These units have no group. Add them to the previous group? (Y/n)"

Re-number approved groups sequentially: {after_phase}.1, {after_phase}.2, etc.

**Step 4: Final Confirmation**

Display approved expansion:

```
Phase {N}.1: {name} ({unit_count} units)
Phase {N}.2: {name} ({unit_count} units)
Phase {N}.3: {name} ({unit_count} units)

Total: {total_units} units across {phase_count} phases

Proceed with ROADMAP expansion? (Y/n)
───────────────────────────────────────────────────────────────
```

New subsection: "### ROADMAP Evolution Check (VERF-08)"

Logic:
1. Check if the just-verified phase is the System Overview phase (or the phase that identifies equipment units):
   - Read ROADMAP.md phase goal for current phase
   - Check if goal mentions "System Overview" OR "equipment" OR "unit identification" OR if this is a Type A/B project's first content phase after project setup

2. If this IS a System Overview phase verification PASS:
   - Read all CONTENT.md files in the phase directory
   - Parse for equipment module identifiers (EM-NNN patterns, equipment module headings, unit names)
   - Count unique units identified
   - If count > 5:
     ```
     ───────────────────────────────────────────────────────────────
     ⚠ ROADMAP EVOLUTION TRIGGERED

     System Overview identified {count} units (threshold: 5).
     The current ROADMAP may need expansion to handle this scope.

     Running /doc:expand-roadmap automatically...
     ───────────────────────────────────────────────────────────────
     ```
     - Then provide instructions to invoke the expand-roadmap workflow logic (delegate to the workflow's logic — either by referencing the workflow file or by providing the unit list and insertion point)
     - After expansion completes (or is declined), continue with the normal PASS Next Up display

   - If count <= 5:
     ```
     ROADMAP check: {count} units identified (threshold: 5). No expansion needed.
     ```

3. If this is NOT a System Overview phase: skip this check entirely (no output).

**Important implementation notes:**
- This auto-trigger implements VERF-08: "After System Overview phase verify PASS, trigger ROADMAP evolution check"
- The trigger should work with BOTH manual /doc:expand-roadmap AND this auto-trigger (locked decision)
- If the engineer declines the expansion during the interactive loop, the PASS result still stands
- The auto-trigger does NOT re-verify after expansion — the new phases are empty and will go through their own discuss/plan/write/verify cycle

Add a workflow rule under "State management":
```
ROADMAP evolution: After System Overview phase verification PASS, check unit count.
If >5 units, auto-trigger expand-roadmap workflow. Engineer can accept, modify, or
reject expansion. PASS result is unaffected by expansion decision.
```
