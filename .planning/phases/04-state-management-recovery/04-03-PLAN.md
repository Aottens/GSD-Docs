**Command file (commands/doc/resume.md):**

Create lean command file (~60-70 lines) following established pattern.

Frontmatter:
```yaml
---
name: doc:resume
description: Detect and resume interrupted operations with forward-only recovery
allowed-tools:
  - Read
  - Write
  - Bash
  - Glob
  - Grep
  - Task
---
```

Note: Task tool needed because resume may need to re-spawn writer subagents. Write tool needed because resume updates STATE.md. No argument-hint (takes no arguments — it auto-detects what to resume).

Execution context references: resume workflow, ui-brand, CLAUDE-CONTEXT.md.
Context: STATE.md, ROADMAP.md, PROJECT.md.

**Workflow file (gsd-docs-industrial/workflows/resume.md, ~250-350 lines):**

Wrap in `<workflow>` tags.

**Step 1: Detect Interrupted Operations**

Read `.planning/STATE.md` and extract Current Operation section.
- Parse: command, phase, wave, wave_total, plans_done, plans_pending, status, started

If status is not IN_PROGRESS:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 DOC > RESUME
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

No interrupted operation found.

To start work, use /doc:status to see your project state.
```
STOP execution.

If status is IN_PROGRESS, proceed.

**Step 2: Verify Completion Proofs**

For each plan_id in plans_done:
- Check file existence: `{phase_dir}/{plan_id}-CONTENT.md` AND `{phase_dir}/{plan_id}-SUMMARY.md`
- If BOTH exist: verified complete
- If either missing: move from plans_done to incomplete list (STATE.md lied — crash corrupted checkpoint)

For each plan_id in plans_pending:
- Check if actually complete (SUMMARY.md exists but STATE.md didn't record it — checkpoint happened before file write)
- If both CONTENT.md and SUMMARY.md exist: move to verified complete list

Build three lists:
- `verified_complete`: Plans confirmed done by file existence
- `incomplete`: Plans that need re-execution
- `partial`: Plans detected as partial writes (CONTENT.md exists but no SUMMARY.md, or < 200 chars, etc.)

**Step 3: Display Context Summary**

Show what was interrupted and current state:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 DOC > RESUME
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Interrupted operation: /doc:{command} {phase}
  Started: {started}
  Wave: {wave}/{wave_total}

Completed: {verified_count} plans
  {list of verified plan IDs}

Remaining: {incomplete_count} plans
  {list of incomplete + partial plan IDs}

{If any partials:}
Partial writes detected:
  {plan_id}: {reason}
```

**Step 4: Smart Default Decision**

Apply the locked user decision: "Smart default: auto-resume if only one thing was interrupted; show choices if multiple things are incomplete."

**Single incomplete operation (smart default):**
- Ask confirmation: "Resume /doc:{command} {phase} from wave {wave}? (Y/n)"
- Default: Y (auto-resume)
- If confirmed: proceed to Step 5

**Multiple options needed (rare — e.g., phase has both partial writes AND a different command was interrupted):**
- Present choices:
  ```
  Resume options:
  1. Resume /doc:{command} {phase} (complete {remaining_count} plans)
  2. View detailed status (/doc:status)
  3. Start different operation (keeps completed work)
  ```
- Wait for selection

If user chooses "view status": Display message "Run /doc:status for full project state." and STOP.
If user chooses "start different operation": Update STATE.md to clear Current Operation (set status to ABANDONED, preserve plans_done for reference). STOP — user will run their desired command.

**Step 5: Execute Resume**

Re-execute the interrupted operation using forward-only recovery:

If command was `write-phase`:
- Determine which wave to resume from:
  - Find the lowest wave number among incomplete plans
  - All plans in earlier waves should already be verified complete
- Re-run write-phase execution from that wave forward:
  - Follow write-phase workflow Steps 4-7 (wave execution through completion)
  - Spawn doc-writer subagents for incomplete plans only
  - Checkpoint STATE.md per wave (same pattern as normal execution)
  - Skip completed plans (NEVER re-execute plans with verified SUMMARY.md)

If command was `verify-phase`:
- Re-spawn doc-verifier subagent
- Follow verify-phase workflow from Step 3 onwards

If command was anything else:
- Display the command that was interrupted and suggest re-running it:
  ```
  To resume, run: /doc:{command} {phase}
  The command will auto-detect the interrupted state.
  ```

**Step 6: Completion**

After successful resume:
- Update STATE.md: status → COMPLETE
- Display completion summary (delegate to the command's normal completion flow)

**Workflow Rules:**
- Forward-only recovery: NEVER re-execute plans with verified SUMMARY.md + CONTENT.md
- SUMMARY.md existence is the ONLY completion proof
- STATE.md is a hint — always verify against filesystem
- If filesystem and STATE.md disagree: trust the filesystem
- DOC > prefix on all banners

**Write-phase auto-detect (gsd-docs-industrial/workflows/write-phase.md):**

Enhance Step 3 (Check for Resume State) to implement the locked user decision: "Both standalone /doc:resume command AND auto-detect when running the same command that was interrupted."

The existing Step 3 already checks for resume state. Enhance it to:

1. After detecting resume state (command == "write-phase" AND status == IN_PROGRESS AND phase matches):
   - Display context summary: what was running, what completed, what's next
   - Show confirmation: "Previous write-phase was interrupted. Resume from wave {N}? (Completed: {done_count} plans, Remaining: {remaining_count} plans)"
   - Default: Yes (auto-resume)
   - If confirmed: filter execution plan to exclude completed plans and proceed with wave execution
   - If declined: ask "Start fresh? This will re-execute ALL plans including previously completed ones." Default: No.

2. Add a new check at the TOP of Step 3 for command conflict (locked user decision: "Running a different command than what was interrupted triggers warn + confirm"):
   - Read STATE.md Current Operation
   - If status is IN_PROGRESS AND command is NOT "write-phase" (or phase does not match):
     - Display warning:
       ```
       ⚠ WARNING: An interrupted operation was detected.

       /doc:{interrupted_command} phase {interrupted_phase} was interrupted during wave {wave}.

       Proceeding with /doc:write-phase {current_phase} may leave the interrupted
       operation in an inconsistent state.

       Options:
       1. Continue with write-phase {current_phase} anyway
       2. Cancel and run /doc:resume to complete the interrupted operation
       ```
     - Wait for selection
     - If continue: proceed normally (interrupted operation remains in STATE.md but won't be overwritten until new operation starts its own checkpoint)
     - If cancel: display "Run /doc:resume to complete the interrupted operation." and STOP
