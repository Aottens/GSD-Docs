## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding (e.g. stage connector dots, status icon gap) |
| sm | 8px | Compact element spacing (e.g. pipeline stage internal button gap, table cell padding) |
| md | 16px | Default element spacing (e.g. pipeline card internal padding, export options gap) |
| lg | 24px | Section padding (e.g. ExportTab section spacing, version history table margins) |
| xl | 32px | Layout gaps between pipeline section and version history section |
| 2xl | 48px | Major section breaks (e.g. FDS pipeline block and SDS section separator) |
| 3xl | 64px | Page-level spacing (not used in this phase) |

Exceptions:
- Pipeline stage card minimum height: 120px — wide enough to show icon + label + status + action button without crowding
- SDS table row minimum touch target: 36px height — carries over from Phase 11/12 OutlineNode pattern
- Cancel button minimum touch target: 44px height — safety-critical abort action

## Interaction Contracts

### 1. Pipeline Stage Layout

The Export tab renders a three-stage horizontal pipeline: **Assembleren → Exporteren → Downloaden**, laid out left-to-right with connector arrows between cards.

- Each stage is a Card-like surface (`bg-card border rounded-lg p-4`) showing: stage icon (top), stage name (label, 14px semibold), status badge (idle/running/done/error), and a primary action button
- Connector between stages: a right-pointing chevron icon (`ChevronRight`, `text-muted-foreground`) centered vertically between cards
- Stage cards have equal width (each `flex-1`), pipeline container uses `flex flex-row gap-4 items-stretch`
- Source: CONTEXT.md — "Pipeline view layout: three-stage visual flow left to right"

### 2. Export Options (above pipeline)

- Render two controls above the pipeline in a horizontal row: Mode selector + Language override
- Mode selector (`Select`): options "Concept (met watermerk)" and "Definitief" — default to "Concept"
- Language override (`Select`): options "Nederlands" and "English" — default from `project.language`
- Final mode: if selected and any phase has no REVIEW.md, the Assembleren button is disabled and an `Alert` (variant="default") appears: "Definitief exporteren vereist dat alle fases zijn gereviewd."
- Source: CONTEXT.md — "Export options: mode (Draft with watermark / Final) and language (Dutch/English from project setting, overridable per export)"

### 3. SSE Progress Bar (named steps)

- When assembly/export starts, a `Progress` bar replaces the static pipeline stage content with named steps
- Named steps rendered as a horizontal list of step labels below the bar: "Cross-referenties oplossen" → "Secties samenvoegen" → "DOCX genereren" → "Diagrammen renderen"
- Each step label shows status icon (Circle/Loader2/CheckCircle2/XCircle) with the step name at 14px
- Active step: `text-blue-500` Loader2 spinner + step name at `font-medium`
- Completed step: `text-green-500` CheckCircle2 + step name at `text-muted-foreground`
- Failed step: `text-destructive` XCircle + step name — card border changes to `border-destructive`
- Progress bar fills proportionally: each completed step = 25% fill (4 steps total)
- Source: CONTEXT.md — "Step-by-step progress bar with named steps; each step ticks as it completes"

### 4. Cancel Button

- "Annuleren" button (Button variant="destructive", outline style) renders during active operation only — below the progress bar
- On click: EventSource is closed (frontend disconnect), backend detects disconnect via `request.is_disconnected()`, cleans up partial files, pipeline returns to pre-run state
- Minimum height 44px — safety-critical abort action
- Button disappears when operation completes or errors
- Source: CONTEXT.md — "Cancel button available during operations — aborts gracefully"

### 5. Inline Error Display

- On pipeline step failure: the failed step card shows an expandable detail block (Collapsible, collapsed by default)
- Collapsed state: error icon + "Stap mislukt — klik voor details" in `text-sm text-destructive`
- Expanded state: error message text at 14px + fix hint (e.g., "Pandoc niet gevonden — installeer via `brew install pandoc`") in monospace `bg-muted` block
- Pipeline stays visible on error — other completed steps retain their done/checkmark state
- Source: CONTEXT.md — "Errors shown inline in the pipeline step that failed: expandable detail block with what went wrong and fix hint"

### 6. Pandoc Not Installed Detection

- If export is triggered and Pandoc is not found: skip SSE streaming entirely, return immediate 422 with error payload
- Frontend: show `Alert` variant="destructive" above the pipeline: heading "Pandoc niet gevonden", body "Installeer Pandoc om DOCX te exporteren:", CLI hint `brew install pandoc` in monospace block
- Alert renders below export options, above the pipeline stages
- Source: RESEARCH.md — "Check lazily when first export is triggered, return clear inline error in failed pipeline step" + Claude's Discretion

### 7. Version History Table

- Renders as `Table` component below the pipeline block, separated by `Separator`
- Columns: Versie | Type | Taal | Datum | Grootte | Downloaden
- Most recent row at top
- Type column: Badge variant="outline" — "Concept" (amber tint) or "Definitief" (green tint)
- Language column: "NL" or "EN" plain text
- Downloaden column: Button variant="ghost" size="sm" with `Download` lucide icon — triggers `GET /api/projects/{id}/export/download/{filename}`
- Empty state: single row spanning all columns — "Nog geen exports. Gebruik de pijplijn hierboven om uw eerste document te exporteren."
- Loading state: 3 `Skeleton` rows, height 36px each
- Source: CONTEXT.md — "Version history as simple table below pipeline: most recent at top"

### 8. SDS Tab Layout

- SDS is its own workspace tab (nav label "SDS", icon `Layers`)
- Tab content: heading "SDS Opbouw" (text-2xl semibold), descriptive paragraph (text-sm muted-foreground), then "SDS Opbouwen" trigger button, then results table below
- Results table only renders when SDS scaffolding has been run (results file exists); otherwise shows empty state
- Table has filter input (text input, placeholder "Filter op module...") and sort controls above it
- Source: CONTEXT.md — "SDS gets its own workspace tab; Phase 13 scope: display scaffolding results with typicals matching only"

### 9. SDS TypicalsMatchTable

- Columns: Equipment Module | Gekoppeld Typical | Overeenkomst | Status
- Overeenkomst (Confidence Score) column: percentage displayed as `{N}%` at 14px semibold — color-coded: ≥70% green, 40–69% amber, <40% red
- Status column: Badge — "Gekoppeld" (green), "Lage overeenkomst" (amber), "NIEUW TYPICAL NODIG" (red/destructive)
- Rows are sortable by clicking column headers (client-side sort, no refetch)
- Filter input above table filters on Equipment Module column (case-insensitive substring match)
- Source: CONTEXT.md — "Equipment-to-typical mapping table: Equipment Module | Matched Typical | Confidence Score | Status; sortable and filterable"

### 10. SDS Match Detail Expansion

- Each SDS table row is expandable via `Collapsible` (click anywhere on row or expand chevron at far right)
- Collapsed: row shows icon + name columns only in normal density
- Expanded: a detail panel opens below the row (not a modal) showing:
  - Why matched/failed: short paragraph at 14px muted
  - Closest match (if unmatched): module name + confidence score
  - CLI hint: `npx doc:generate-sds --refine {module_name}` in monospace `bg-muted` block
- Source: CONTEXT.md — "Click to expand: shows why matching failed, closest match, CLI hint for /doc:generate-sds to refine"

### 11. Navigation Tab Order

Navigation items in `ProjectNavigation.tsx` extend the existing array with two new entries:

```
Overzicht | Fases | Documenten | Referenties | Exporteren | SDS | Instellingen
```

- "Exporteren" uses `Download` lucide icon
- "SDS" uses `Layers` lucide icon
- Both tabs append after "Referenties" (index 4 and 5) and before any existing "Instellingen" entry
- Source: RESEARCH.md Pattern 6 — Navigation Extension

### 12. Loading and Error States (general)

- SDS results loading: 5 `Skeleton` rows at 36px height each
- Version history loading: 3 `Skeleton` rows at 36px height each
- SDS scaffold trigger loading: Button shows `Loader2` spinner icon + disabled state during POST request
- Network error (version history or SDS results fetch): `text-sm text-muted-foreground` paragraph — "Gegevens konden niet worden geladen. Ververs de pagina of controleer de verbinding."
- Source: RESEARCH.md — Claude's Discretion on loading/error states

### 13. Animation / Transition

- Pipeline stage status change (idle → running): `transition-all duration-150` on status badge swap
- Progress bar fill: CSS `transition-all duration-300` on width change per step completion
- Collapsible expansion (SDS detail, inline error): shadcn Collapsible default animation (already handles transition)
- No custom spring animations — Motion library not used in this phase
- Source: RESEARCH.md — Claude's Discretion on animation

## Export State Architecture Contract

This is an interaction contract between components — the executor must follow this exactly.

| Decision | Value | Source |
|----------|-------|--------|
| Pipeline state location | Local component state in `ExportTab` — not in Context | Claude's Discretion — pipeline is single-tab, no cross-tab sharing needed |
| SSE consumer | `useAssemblyStream` hook in `features/export/hooks/` — opens `EventSource`, parses named step events, exposes `{ stages, isRunning, start, cancel }` | RESEARCH.md Pattern 2 |
| Version history fetch | `useVersionHistory(projectId)` React Query hook, 0s refetch interval (manual invalidation on export complete) | RESEARCH.md Standard Stack |
| SDS results fetch | `useSdsResults(projectId)` React Query hook, no polling (results are static until next scaffold run) | RESEARCH.md Pattern |
| Backend write for review | None — SDS tab is read-only display; no REVIEW.md or catalog writes from GUI | CONTEXT.md — "display scaffolding results only; no library browser, no inline editing" |
| Typicals matching auto-apply | Never — display only; engineer acts via CLI `/doc:generate-sds` | RESEARCH.md Anti-Patterns |
| Cancel mechanism | `eventSourceRef.current.close()` on frontend; backend detects via `await request.is_disconnected()` — no separate cancel endpoint | RESEARCH.md Pattern 3 pitfall note |
| Toast notifications | Sonner `toast.success()` on export complete; `toast.error()` on unrecoverable error | CONTEXT.md — "Sonner for toasts" established pattern |
