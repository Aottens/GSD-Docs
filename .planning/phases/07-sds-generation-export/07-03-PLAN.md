  <action>
Create comprehensive SDS generation workflow following established workflow patterns (see new-fds.md workflow for scaffolding reference, complete-fds.md for document analysis reference).

**SDS Generation Pipeline Steps (12 steps):**

**Step 1: Validate FDS Prerequisites**
- Scan `output/` for assembled FDS documents (FDS-*.md files).
- If multiple found: use most recent by modification time, or ask user to specify.
- Read FDS YAML frontmatter: extract project name, version, date, language, type.
- Verify FDS has content (not just frontmatter — check file size > 1KB).
- Read STATE.md for current FDS version and project configuration.
- Log: `DOC > SDS Generation: Reading FDS v{version} for {project-name}`

**Step 2: Load Typicals Library**
Two modes per user decision:
a. **External reference:** If `--typicals [path]` provided or PROJECT.md has `typicals.path`, resolve the path (absolute or relative to project root), look for CATALOG.json at that location.
b. **Imported copy:** If `--import` flag, copy CATALOG.json (and any .md documentation files) from external path to `references/typicals/`, then load from there.
c. **No typicals:** If no path configured and no --typicals flag, log warning: "No typicals library configured. All equipment modules will be flagged as NEW TYPICAL NEEDED." Proceed with empty catalog.

Validate loaded CATALOG.json against CATALOG-SCHEMA.json (from Plan 07-01): check schema_version, library metadata, typicals array structure. Warn on validation issues but don't block.

Log: `DOC > Typicals: Loaded {N} typicals from {library-name} v{version} ({mode})` or `DOC > Typicals: No library configured — skeleton mode`

**Step 3: Extract FDS Equipment Modules**
Parse the assembled FDS document to extract equipment module profiles:
- Find all equipment module sections (section 4.x in FDS structure).
- For each module, extract:
  - Module ID (e.g., EM-200)
  - Module name (e.g., Dosing System)
  - I/O list (from I/O table: count of DI, DO, AI, AO, data types)
  - Operating states (from state table: state names, count)
  - Parameters (from parameter table: names, ranges, units)
  - Interlocks (if present: count and types)
  - Keywords (derived from description text for matching)
- Store as structured module profiles array.
- Log: `DOC > FDS Analysis: Found {N} equipment modules`

**Step 4: Match Equipment Modules Against Typicals** (skip if no catalog)
For each FDS equipment module profile, attempt matching against loaded typicals catalog:

**Matching algorithm (suggest + confirm per user decision):**
a. For each typical in catalog, calculate match score:
   - I/O interface match (40% weight): compare input/output counts and data types
   - Keyword/use-case match (30% weight): Jaccard similarity between module keywords and typical use_cases
   - State complexity match (20% weight): compare state count similarity
   - Category match (10% weight): exact category match
b. Filter matches with confidence >= 30%
c. Rank by confidence descending, take top 3 suggestions per module
d. Classify: HIGH (>=80%), MEDIUM (60-79%), LOW (40-59%), VERY LOW (<40%)

**Output matching report** — For each equipment module, generate a comparison table (per research Example 3):
- FDS requirements summary (I/O count, states, parameters)
- Top suggested typical with confidence score
- Interface match analysis (exact/partial/mismatch)
- Recommendation: accept match, review interfaces, or NEW TYPICAL NEEDED

**This is a report for the engineer** — the engineer confirms or overrides matches during the SDS discuss-phase cycle (matching is NEVER auto-applied per user decision).

For modules with NO match above 30%: mark as "NEW TYPICAL NEEDED" immediately.

Log: `DOC > Matching: {matched} matched, {unmatched} unmatched, {review} need review`

**Step 5: Generate SDS Equipment Module Seeds**
For each equipment module, create initial SDS content:

a. **Matched modules (any confidence):** Generate SDS section using section-equipment-software.md template. Pre-fill:
   - Typical Assignment: matched typical ID, confidence, library reference
   - FB Composition: infer from typical's interfaces (placeholder for engineer to refine)
   - Parameter Configuration: merge FDS parameters with typical defaults
   - State Machine Implementation: map FDS states to typical states where names align
   - Data Flow: placeholder with I/O signals from FDS
   Mark as "NEEDS ENGINEER REVIEW" — this is a starting point, not final content.

b. **Unmatched modules:** Generate structured skeleton from FDS per user decision (NOT a stub — a structured starting point):
   - Typical Assignment: "## NEW TYPICAL NEEDED: FB_{ModuleName}" with I/O profile from FDS
   - FB Composition: empty template with module I/O as requirements
   - Parameter Configuration: all FDS parameters listed with "Source: FDS" and no typical defaults
   - State Machine Implementation: all FDS states listed with "FB State: [TO BE DESIGNED]"
   - Data Flow: all FDS I/O signals listed with "Destination: [TO BE DESIGNED]"
   Mark as "NEW TYPICAL NEEDED" with structured requirements from FDS.

Store seeds as individual section files for the SDS write-phase cycle.

**Step 6: Generate TRACEABILITY.md**
Using the traceability.md template (from Plan 07-01):
- Parse FDS for all functional requirements (from REQUIREMENTS.md or inline requirements in FDS sections).
- For each FDS requirement, create a traceability row:
  - FDS Req ID, FDS Section (with number), SDS Section (mapped from equipment module), Implementation (placeholder), Status (Pending)
- Generate Coverage Analysis: total requirements, mapped count, not-applicable count.
- Write to `.planning/sds/TRACEABILITY.md`
- Log: `DOC > Traceability: {N} FDS requirements mapped to SDS sections`

**Step 7: Scaffold SDS Project Structure**
Create the SDS project directory structure (parallel to FDS .planning/):

```
.planning/sds/
├── PROJECT.md          # SDS-specific config, inherits from FDS
├── ROADMAP.md          # SDS phases (3-phase structure)
├── STATE.md            # SDS progress tracking
├── REQUIREMENTS.md     # SDS-specific requirements derived from FDS
├── TRACEABILITY.md     # FDS → SDS requirement mapping (from Step 6)
└── phases/
    ├── sds-01-foundation/    # Typicals matching confirmation + software architecture
    ├── sds-02-equipment/     # Equipment module SDS sections
    └── sds-03-integration/   # Interfaces, HMI, final assembly
```

**SDS PROJECT.md:** Copy language, client, project name from FDS PROJECT.md. Add:
- `type: SDS`
- `version: 0.1`
- `based_on: { document: FDS, version: {fds-version}, date: {fds-date} }`
- `typicals: { mode: external|imported, path: {path}, catalog: {catalog-path} }`
- `sds_structure: equipment-first` (or software-first if --structure flag used)

**SDS ROADMAP.md:** 3-phase SDS structure:
- SDS Phase 1: Foundation + Typicals Matching Confirmation (software architecture overview, confirm/override typical matches, library setup)
- SDS Phase 2: Equipment Module Details (write SDS sections per equipment module using seeds from Step 5)
- SDS Phase 3: Integration + Assembly (interfaces, HMI, final SDS assembly, TRACEABILITY verification)

**SDS STATE.md:** Initialize at SDS Phase 1, pending.

**SDS REQUIREMENTS.md:** Derive from FDS requirements, focused on software implementation.

**Step 8: Generate Typicals Matching Report**
Write a human-readable matching report to `.planning/sds/MATCHING-REPORT.md`:
- Summary table: Equipment Module ID | Module Name | Suggested Typical | Confidence | Status
- Detailed comparison for each module (I/O, states, parameters alignment)
- Unmatched modules section with structured requirements
- Instructions: "Review matches during `/doc:discuss-phase 1 --sds`. Confirm, override, or mark as NEW TYPICAL."

**Step 9: Update PROJECT.md with SDS Configuration**
Add SDS section to the main PROJECT.md (not .planning/sds/PROJECT.md — the main one):
- `sds: { version: 0.1, based_on_fds: v{version}, structure: equipment-first, typicals: { mode: ..., path: ... } }`
This enables other commands (/doc:export, /doc:status) to detect SDS project existence.

- This workflow creates a SCAFFOLDED SDS project, not a complete SDS document. The engineer then runs the full discuss-plan-write-verify cycle for SDS phases.
- Typical matches are SUGGESTIONS only — never auto-applied. Engineer confirms during SDS discuss-phase.
- The workflow file is MARKDOWN with step-by-step instructions for Claude to follow, NOT a script.
- Use the same lean-command + detailed-workflow separation pattern.
- Target ~1200-1600 lines for the workflow (most complex workflow in the project — FDS analysis + typicals matching + scaffolding).
- The --sds flag on existing commands (discuss-phase, plan-phase, write-phase, verify-phase) tells those commands to look in .planning/sds/ instead of .planning/. This flag support is DOCUMENTED in the workflow but NOT implemented here — it requires minimal changes to existing commands that the engineer can make when first running the SDS cycle.
