Create sds-structure.json following the EXACT same JSON pattern as fds-structure.json but for SDS documents. The SDS has a **hybrid structure** per user decision: equipment modules as primary sections, software structure (FB hierarchy, instantiation, data flow) within each.

**SDS sections (canonical ordering):**
1. Introduction (Purpose/Scope, Document Conventions, References, Abbreviations) — same pattern as FDS section 1
2. Software Architecture Overview (Program structure, FB hierarchy overview, Communication architecture)
3. Typicals Library Reference (typicals catalog summary, library metadata, version info)
4. Equipment Modules — **dynamic expansion** (one 4.x per equipment module, each containing: Typical Assignment, FB Composition, Instantiation, Parameter Configuration, Data Flow, State Machine Implementation)
5. Interfaces (PLC-SCADA, PLC-PLC, Field bus, External systems)
6. HMI/SCADA Integration (Screen structure, Faceplate mapping, Alarm configuration)
7. Appendices (Signal list, Parameter summary, FB instance list)

Each section needs bilingual titles (en/nl), depth, required flag, source_type (like FDS template), and children array. Equipment modules section (4) must have dynamic expansion pattern matching FDS section 4.

Include equipment module subsections:
- 4.x.1: Typical Assignment — which typical(s) matched, confidence, "NEW TYPICAL NEEDED" flag if unmatched
- 4.x.2: FB Composition — which FBs compose this module, hierarchy diagram
- 4.x.3: Instantiation — DB instances, instance names, PLC mapping
- 4.x.4: Parameter Configuration — parameter values derived from FDS, typical defaults
- 4.x.5: Data Flow — inputs/outputs, interconnections between FBs
- 4.x.6: State Machine Implementation — mapping from FDS states to FB states

Include `based_on` metadata field for SDS version cross-reference to FDS version.
Include `sds_structure` field supporting "equipment-first" (default) and "software-first" presets per Pitfall 4 mitigation.

**CATALOG-SCHEMA.json:**
Create a JSON schema (not a catalog itself) defining the expected structure of a project-specific typicals catalog. Include:
- `schema_version: "1.0"` for future evolution
- `library` object: name, version, platform (TIA Portal V18+), updated date
- `typicals` array where each typical has: id (e.g., "FB_AnalogIn"), type ("Control Module"), category, description, interfaces (inputs/outputs arrays with name+dataType), use_cases array, states array (optional), documentation path (optional)
- Validation notes in comments/description fields explaining each field's purpose

This is a SCHEMA file that validates project-specific CATALOG.json files. Projects create their own CATALOG.json matching this schema. Include an `example` field in the schema showing a complete FB_AnalogIn typical entry.

**Do NOT create an actual CATALOG.json** — that is project-specific and created per-project during /doc:generate-sds.

**section-equipment-software.md:**
Create an SDS equipment module template following the SAME frontmatter+comment pattern as fds/section-equipment-module.md. This template is used by doc-writer subagents during SDS write-phase.

Frontmatter: type: equipment-software, language: "{LANGUAGE}", subsections with required/optional lists.

Required subsections (6):
1. Typical Assignment — table: Equipment Module ID, Matched Typical, Confidence (HIGH/MEDIUM/LOW), Status (Matched/NEW TYPICAL NEEDED), Library Reference. Include "NEW TYPICAL NEEDED" marker pattern for unmatched modules.
2. FB Composition — table: FB Name, Type (from library or custom), Purpose, Parent FB. Include Mermaid diagram placeholder for FB hierarchy. Focus on SOFTWARE STRUCTURE (which FBs compose this module) per user decision.
3. Instantiation — table: Instance Name, FB Type, DB Number, PLC/CPU, Comment. This maps FBs to actual PLC program instances.
4. Parameter Configuration — table: Parameter, Value, Source (FDS/Typical Default/Engineer), Unit, Description. Cross-references FDS parameter table.
5. Data Flow — table: Signal, Source FB.Output, Destination FB.Input, Data Type, Description. Include Mermaid flowchart placeholder for data flow diagram.
6. State Machine Implementation — table: FDS State, FB State, Trigger, Action, Notes. Maps FDS operating states to FB implementation states.

When referencing typicals: use **summary + reference** pattern per user decision — brief description of purpose and key interfaces, with library reference for full details. Do NOT document typical internals.

Include bilingual column headers (English / Nederlands pattern from FDS templates).
Include HTML comment documentation block (who creates, who consumes, placeholder rules).

**section-typicals-overview.md:**
Template for the SDS Typicals Library Reference section (section 3 in sds-structure.json).
- Library metadata table: Library Name, Version, Platform, Last Updated, Load Mode (External/Imported)
- Typicals summary table: Typical ID, Category, Description, Used By (equipment module IDs), Interfaces Summary
- Unmatched equipment section: lists all equipment modules that need NEW TYPICAL development
- Library reference path (from PROJECT.md typicals config)

**traceability.md:**
Template for TRACEABILITY.md — an internal quality check (NOT client deliverable per user decision).
- Header: project name, FDS version, SDS version, generation date
- Main table: FDS Req ID, FDS Section, SDS Section, Implementation (FB/DB reference), Status (Complete/Partial/Not Applicable)
- Coverage Analysis section: Total FDS requirements, Mapped to SDS count/percentage, Not Applicable count, Missing Implementation count
- Quality gate: 100% coverage required before SDS can be released (or --force override)
- Include HTML comment explaining this is consumed by /doc:generate-sds workflow for traceability generation
