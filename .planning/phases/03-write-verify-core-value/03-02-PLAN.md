<task type="auto">
  <name>Task 2: Create write-phase workflow file</name>
  <files>
    gsd-docs-industrial/workflows/write-phase.md
  </files>
  <action>
    Create `gsd-docs-industrial/workflows/write-phase.md` -- the detailed execution logic for parallel FDS content generation. This is the most complex workflow in Phase 3 because it orchestrates subagent spawning with strict context isolation and crash recovery.

    Target: 500-700 lines. Wrap ALL content in `<workflow>` tags. Use numbered `## Step N:` headings.

    **Step 1: Parse Arguments and Validate Phase**
    - Parse $ARGUMENTS to get phase number
    - Read `.planning/ROADMAP.md` to find the phase entry
    - Extract: phase name, phase goal, dependencies
    - Verify dependencies met (check for SUMMARY.md files in dependent phase directories)
    - Read `.planning/PROJECT.md` for project config (language, standards settings)
    - If phase not found or dependencies unmet: show error box (from ui-brand.md) and abort
    - Display banner: `DOC > WRITING PHASE {N}`

    **Step 2: Discover and Classify Plans**
    - Glob for all `{NN}-*-PLAN.md` files in the phase directory
    - Read each plan's YAML frontmatter to extract: plan number, name, wave, depends_on, type
    - Group plans by wave number
    - Count waves and plans per wave
    - Display wave structure table:
      ```
      Wave Structure:
      | Wave | Plans | Sections |
      |------|-------|----------|
      | 1    | 01, 02 | EM-100, EM-200 |
      | 2    | 03 | Phase Overview |
      ```

    **Step 3: Check for Resume State**
    - Read `.planning/STATE.md` to check if a previous write-phase was interrupted
    - If STATE.md shows `current_operation.command: write-phase` and `current_operation.status: IN_PROGRESS`:
      - Extract `plans_done` list -- these plans are already complete
      - Filter them out of the execution plan
      - Display resume message: `DOC > Resuming from Wave {N} ({count} plans already complete)`
      - Verify completed plans by checking CONTENT.md + SUMMARY.md existence
      - If completed plan files are missing: add back to execution plan (forward-only recovery)
    - If no resume state or fresh start: proceed with all plans

    **Step 4: Execute Waves (Sequential)**
    For each wave (in order):

    **4a. Pre-wave checkpoint:**
    Update STATE.md with current operation state:
    ```markdown
    ## Current Operation
    - command: write-phase
    - phase: {N}
    - wave: {current_wave}
    - wave_total: {total_waves}
    - plans_done: [{completed plan IDs}]
    - plans_pending: [{remaining plan IDs}]
    - status: IN_PROGRESS
    - started: {timestamp}
    ```

    Display: `DOC > Wave {W}/{total}: Writing {count} sections in parallel`

    **4b. Spawn writers (Parallel within wave):**
    For EACH plan in the current wave, spawn a doc-writer subagent using the Task tool:

    CRITICAL CONTEXT ISOLATION: The orchestrator builds the EXACT file list for each writer. Never pass directory paths. Never pass other plans' files.

    For each plan_id in wave:
    ```
    Spawn subagent: doc-writer
    Context files (explicit list):
      - .planning/PROJECT.md
      - .planning/phases/{NN}-{name}/{NN}-CONTEXT.md
      - .planning/phases/{NN}-{name}/{plan_id}-PLAN.md
      - (if standards.packml.enabled): gsd-docs-industrial/references/standards/packml/
      - (if standards.isa88.enabled): gsd-docs-industrial/references/standards/isa-88/
    Message: "Write section {plan_id} per plan. Output files to .planning/phases/{NN}-{name}/. Produce {plan_id}-CONTENT.md and {plan_id}-SUMMARY.md."
    ```

    The orchestrator NEVER passes to writers:
    - Other PLAN.md files
    - Other CONTENT.md files
    - Other SUMMARY.md files
    - Main session conversation history

    All writers in a wave spawn simultaneously (parallel). Wait for all to complete before proceeding.

    **4c. Post-wave validation:**
    After all writers in wave complete:
    - For each plan in wave: verify both CONTENT.md and SUMMARY.md exist
    - If any missing: log error but continue (don't block other waves)
    - Display per-writer results:
      ```
      Wave {W} Results:
      - {plan_id}: CONTENT.md ({size}), SUMMARY.md ({words} words), [VERIFY]: {count}
      - {plan_id}: CONTENT.md ({size}), SUMMARY.md ({words} words), [VERIFY]: {count}
      ```

    **4d. Post-wave checkpoint:**
    Update STATE.md: add completed plans to plans_done, remove from plans_pending.

    **Step 5: Aggregate Cross-References**
    After all waves complete:
    - Read all CROSS-REFS.md entries created by writers
    - If multiple writers created/appended to the same file: deduplicate entries
    - Update status column: check if target sections now exist (resolved) or not (pending)
    - Write consolidated CROSS-REFS.md to phase directory: `.planning/phases/{NN}-{name}/{NN}-CROSS-REFS.md`

    **Step 6: Final STATE.md Update**
    Update STATE.md:
    ```markdown
    ## Current Operation
    - command: write-phase
    - phase: {N}
    - status: COMPLETE
    - completed: {timestamp}
    - plans_written: [{all plan IDs}]
    - verify_markers: {total [VERIFY] count}
    - cross_refs: {total cross-reference count}
    ```

    **Step 7: Completion Summary**
    Display banner: `DOC > PHASE {N} WRITTEN`

    Show summary:
    ```
    Sections written: {count}
    Waves executed: {wave_count}
    CONTENT.md files: {count} ({total size})
    SUMMARY.md files: {count}
    [VERIFY] markers: {count} (engineer review needed)
    Cross-references: {count} ({resolved} resolved, {pending} pending)
    ```

    Show Next Up block:
    ```
    ## Next Up
    /doc:verify-phase {N}
    ```

    If [VERIFY] markers > 0:
    ```
    Note: {count} [VERIFY] markers in content require engineer review.
    These are inferred values that should be confirmed for accuracy.
    ```

    **WORKFLOW RULES:**
    - All user-facing text matches the project language (from PROJECT.md)
    - Use DOC > prefix on all banners (never GSD >)
    - NEVER read CONTENT.md from other sections in the orchestrator
    - Context isolation is enforced by explicit file listing, not directory access
    - STATE.md checkpoint is atomic: only write after all writers in wave complete
    - Forward-only recovery: on resume, never re-execute completed plans
    - Max 4 parallel writers per wave (conservative default per research open question 1)
  </action>
