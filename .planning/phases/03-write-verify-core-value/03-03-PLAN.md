<objective>
Create the `/doc:verify-phase` command and its workflow -- the orchestrator that verifies written FDS content against phase goals (not just task completion) using a 5-level cascade, produces VERIFICATION.md, and manages the gap closure loop with max 2 iterations.

Purpose: This command ensures documentation quality by checking whether phase goals are actually achieved, not just whether files exist. The 5-level cascade (exists, substantive, complete, consistent, standards-compliant) catches progressively deeper issues. When gaps are found, the command routes to plan-phase --gaps for targeted fixes, with a hard 2-cycle limit before escalating to ENGINEER-TODO.md.

Output: 3 files -- command file (lean orchestrator), workflow file (detailed verification logic), and version-tracked copy.
</objective>

  <name>Task 2: Create verify-phase workflow file</name>
  <files>
    gsd-docs-industrial/workflows/verify-phase.md

  <action>
    Create `gsd-docs-industrial/workflows/verify-phase.md` -- the detailed execution logic for goal-backward verification with a 5-level cascade.

    Target: 400-600 lines. Wrap ALL content in `<workflow>` tags. Use numbered `## Step N:` headings.

    **Step 1: Parse Arguments and Validate Phase**
    - Parse $ARGUMENTS to get phase number
    - Read `.planning/ROADMAP.md` to find the phase entry
    - Extract: phase name, phase goal, success criteria, requirements list
    - Read `.planning/PROJECT.md` for standards settings (needed for Level 5)
    - Verify phase has written content (check for CONTENT.md files)
    - If no content found: show error box suggesting `/doc:write-phase {N}` first, abort
    - Display banner: `DOC > VERIFYING PHASE {N}`

    **Step 2: Determine Verification Cycle**
    - Check for existing VERIFICATION.md in phase directory
    - If exists and status is GAPS_FOUND:
      - Read cycle count from VERIFICATION.md header
      - Increment cycle: this is cycle {previous + 1}
      - If cycle > 2: jump to Step 7 (Max Iterations Escalation)
      - Display: `DOC > Verification Cycle {N} of 2`
    - If no VERIFICATION.md or status is PASS:
      - This is cycle 1
      - Display: `DOC > Initial Verification`

    **Step 3: Derive Must-Have Truths (Goal-Backward)**
    This is the core innovation: verify GOALS, not tasks.

    From the ROADMAP.md phase goal, derive observable truths. These are statements that must be true for the phase goal to be achieved, stated from the USER's perspective.

    **Truth derivation process:**
    1. Read the phase goal statement
    2. Ask: "What must be TRUE for this goal to be achieved?"
    3. Generate 3-7 truths that are:
       - Observable (can be checked by reading content)
       - Specific (not vague: "all equipment modules have I/O tables" not "documentation is complete")
       - User-perspective (what the engineer sees, not what files exist)

    **Example for an equipment modules phase:**
    - "All {N} equipment modules are fully documented with all required sections"
    - "All equipment modules have complete I/O tables with tag, type, range, and units"
    - "All equipment modules have state machine descriptions with entry/exit conditions"
    - "All interlocks are documented with condition, action, and priority"
    - "Cross-references between equipment modules are consistent"

    Display derived truths to engineer before proceeding.

    **Step 4: Spawn Verifier Subagent**
    Spawn a doc-verifier subagent using the Task tool with:

    Context files:
    - .planning/ROADMAP.md (phase goals)
    - .planning/phases/{NN}-{name}/{NN}-CONTEXT.md (phase decisions)
    - All {NN}-{MM}-CONTENT.md files in phase directory
    - All {NN}-{MM}-SUMMARY.md files in phase directory
    - .planning/phases/{NN}-{name}/{NN}-CROSS-REFS.md (if exists)
    - Standards files (if enabled in PROJECT.md)

    Message: "Verify phase {N} content against these must-have truths: {truths list}. For each truth, run the 5-level cascade (exists, substantive, complete, consistent, standards-compliant). Produce VERIFICATION.md per the template. Gap descriptions only -- do not suggest fixes."

    The verifier produces VERIFICATION.md in the phase directory.

    **Step 5: Process Verification Results**
    Read the generated VERIFICATION.md:
    - Parse status: PASS or GAPS_FOUND
    - Count: total truths, passed truths, gap truths
    - Extract gap descriptions for display

    **If PASS:**
    - Display banner: `DOC > PHASE {N} VERIFIED`
    - Show pass summary with all truths checked
    - Update STATE.md: phase status to VERIFIED
    - Show Next Up:
      ```
      ## Next Up
      Phase {N} verification passed.
      Continue to next phase or run /doc:review-phase {N} for optional review.
      ```

    **If GAPS_FOUND:**
    - Display banner: `DOC > GAPS FOUND IN PHASE {N}`
    - Show summary table (from VERIFICATION.md)
    - Display each gap with evidence
    - Update STATE.md: phase status to GAPS_FOUND, increment gap_closure_cycle

    **Step 6: Gap Routing (GAPS_FOUND only)**
    When gaps are found:

    - Display gap preview (per user decision: "before generating fix plans, show engineer the list of gaps"):
      ```
      Gaps to address:
      1. {Gap 1 description}
      2. {Gap 2 description}

      These gaps will be addressed by /doc:plan-phase {N} --gaps
      ```

    - Show Next Up:
      ```
      ## Next Up
      /doc:plan-phase {N} --gaps

      Then: /doc:write-phase {N} to execute fixes
      Then: /doc:verify-phase {N} to re-verify
      ```

    - Note current cycle count: "Gap closure cycle {N} of 2"

    **Step 7: Max Iterations Escalation**
    When cycle count exceeds 2 (per user decision: "max 2 cycles, escalate to ENGINEER-TODO.md AND phase completion blocked"):

    - Display banner: `DOC > MAX GAP CLOSURE CYCLES REACHED`

    - Read remaining gaps from VERIFICATION.md

    - Create `.planning/ENGINEER-TODO.md` with:
      ```markdown
      # Phase {N}: {Phase Name} - Manual Intervention Required

      **Date:** {DATE}
      **Reason:** Gap closure loop terminated after max 2 cycles

      ## Remaining Gaps

      ### Gap {N}: {Description}
      **Severity:** {High/Medium/Low}
      **Description:** {Full gap description from VERIFICATION.md}
      **Context:** {Evidence from VERIFICATION.md}
      **Attempts:** 2 automated fix cycles attempted
      **Next steps:**
      1. {Suggested manual action based on gap type}
      2. Re-run /doc:verify-phase {N} after manual fix
      ```

    - Update STATE.md: phase status to BLOCKED, blocked_reason to "Gap closure failed after 2 iterations"

    - Update VERIFICATION.md status to: `GAPS_FOUND (ESCALATED)`

    - Display:
      ```
      Phase {N} BLOCKED.
      Remaining gaps require manual resolution.
      See: .planning/ENGINEER-TODO.md
      After manual fixes: /doc:verify-phase {N}
      ```

    **Step 8: Re-Verification Scope Decision**
    When this is cycle 2+ (re-verification after gap closure):

    - Determine scope (Claude's discretion per user decision):
      - If fix plans touched cross-references (check PLAN.md gap plans for cross-ref mentions): re-verify entire phase
      - If fix plans were isolated content fixes: re-verify only fixed sections
    - Document scope decision in VERIFICATION.md header: "Re-verification scope: {full phase | sections NN-MM, NN-MM} (Reason: {rationale})"
    - Apply scope filter when spawning verifier in Step 4

    **WORKFLOW RULES:**
    - All user-facing text matches the project language (from PROJECT.md)
    - Use DOC > prefix on all banners (never GSD >)
    - Gap descriptions only -- never suggest fixes in VERIFICATION.md (user decision)
    - All 5 verification levels always run (user decision)
    - Cross-references to unwritten sections: warn-only (user decision, Claude's discretion on warn vs deferred gap)
    - Cycle count tracked in VERIFICATION.md header and STATE.md
    - ENGINEER-TODO.md + phase BLOCKED after max 2 cycles (user decision)
  </action>
