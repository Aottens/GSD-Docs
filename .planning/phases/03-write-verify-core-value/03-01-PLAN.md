    The `disallowedTools: Glob, Grep` prevents file discovery beyond explicitly loaded context. This is the enforcement mechanism for context isolation (per user decision: "strict isolation, no cross-contamination").

    Markdown body must cover these sections:

    1. **Role statement:** Technical documentation writer for FDS industrial automation. Writes in the language specified in PROJECT.md.

    2. **Context You Receive:** Explain what files are available (PROJECT.md, phase CONTEXT.md, own PLAN.md, standards if enabled) and what is NOT available (other plans, other content, main session history). State this isolation is deliberate.

    3. **Your Task (7 steps):**
       - Step 1: Read PLAN.md completely (goal, sections, context, template, standards, verification)
       - Step 2: Load the template referenced in PLAN.md to understand section structure
       - Step 3: Write CONTENT.md following the template structure:
         - Fill all required sections with substantive content (per user decision: "all required template sections must have real content")
         - Use technical detail: specific values, units, ranges
         - Mark inferred content with [VERIFY] (per user decision: "never silently guess")
         - I/O tables: generate complete rows with industry-typical signal ranges (4-20mA, 0-100%), mark inferred with [VERIFY]
         - State machines: high-level overview first (Mermaid stateDiagram-v2 + summary transition table), flag complex transitions as [DETAIL NEEDED]
         - Optional sections: fill if relevant per CONTEXT.md, leave empty with note if not relevant
         - Create cross-references where relationships are obvious (interlocks, interfaces, HMI references)
         - Use output language from PROJECT.md for headers and descriptive text
       - Step 4: Write SUMMARY.md (reference @~/.claude/gsd-docs-industrial/templates/summary.md):
         - Max 150 words, hard limit
         - 4 mandatory sections: Facts, Key Decisions, Dependencies, Cross-refs

         - No prose paragraphs
       - Step 5: Log cross-references in CROSS-REFS.md (reference @~/.claude/gsd-docs-industrial/templates/cross-refs.md):
         - Append to existing or create if first writer
         - Each reference: source section, target section, type (depends-on, related-to, see-also), context sentence, status (resolved/pending)
       - Step 6: Self-verify before completing (7 checks):
         - [ ] CONTENT.md has all required sections from PLAN.md
         - [ ] Technical content is substantive (>500 words for EM, >200 for others)
         - [ ] All tables are complete (no empty rows in required columns)
         - [ ] Inferred content marked with [VERIFY]
         - [ ] SUMMARY.md is 100-150 words
         - [ ] SUMMARY.md has all 4 sections
         - [ ] Cross-references logged in CROSS-REFS.md
       - Step 7: Return completion message with file sizes, word counts, [VERIFY] marker count and locations

       - "[VERIFY] markers are acceptable in required sections; empty required sections are not" (locked user decision)
       - Cross-reference format: use format that works best for downstream assembly resolution (Claude's discretion)
       - Proactively create cross-references based on obviousness of relationship (Claude's discretion)

    **doc-verifier.md** (~80-120 lines):

    YAML frontmatter:
    ```yaml
    ---
    name: doc-verifier
    description: Verifies FDS section content against phase goals using a 5-level cascade. Use for /doc:verify-phase verification tasks.
    tools: Read, Bash, Glob, Grep
    disallowedTools: Write
    model: sonnet
    ---
    ```

    The `disallowedTools: Write` makes the verifier read-only -- it cannot modify content, only assess it.

    Markdown body must cover:

    1. **Role statement:** Quality verifier for FDS documentation. Reads content, checks against goals, produces gap analysis. Does NOT suggest fixes (per user decision: "gap descriptions only -- don't suggest fixes").

    2. **Context You Receive:** ROADMAP.md (phase goals), all CONTENT.md files in phase, all SUMMARY.md files, CONTEXT.md, standards files if enabled.

    3. **Your Task:**
       - Step 1: Read ROADMAP.md phase entry to extract must-have truths (goal-backward)
       - Step 2: For each truth, run the 5-level verification cascade:
         - Level 1 (Exists): File present with non-zero size
         - Level 2 (Substantive): Contains structured tables, technical content, >200 words, no [TODO]/[TBD]/[PLACEHOLDER] markers (note: [VERIFY] markers are acceptable)
         - Level 3 (Complete): All required template sections present with content
         - Level 4 (Consistent): Content matches CONTEXT.md decisions (signal ranges, behaviors, decisions)
         - Level 5 (Standards): If standards enabled, state names match canonical lists, terminology correct. If no standards enabled, mark N/A.
         - STOP cascade on first failure level for each truth
       - Step 3: Check cross-references:
         - References to written sections: verify target exists
         - References to unwritten sections: warn-only, document reasoning for whether it is a warning or deferred gap (Claude's discretion)
       - Step 4: Generate VERIFICATION.md (reference @~/.claude/gsd-docs-industrial/templates/verification.md):
         - Summary table at top (quick scan)
         - Detailed findings per truth below
         - Status: PASS or GAPS_FOUND
         - Each gap: description only, no fix suggestions
         - Include evidence (file, line, context reference)
       - Step 5: Return summary with pass/gap counts and status

    4. **Rules:**
       - "All 5 verification levels always run" (locked user decision)
       - "Gap descriptions only -- describe what's missing or wrong" (locked user decision)
       - Never use imperative language in gap descriptions ("change", "update", "fix")
       - Use descriptive language ("uses X instead of Y", "missing section Z")

    Create three output format templates that writers and verifiers reference for consistent output.

    **summary.md** (~30-40 lines):

    HTML comment block at top (matching existing template pattern from context.md, baseline.md etc.):
    ```
    <!-- TEMPLATE: Section SUMMARY.md
         Generated by: doc-writer subagent after writing CONTENT.md
         Consumed by: doc-verifier (cross-reference checking without loading full CONTENT.md)
         Consumed by: write-phase orchestrator (completion reporting)

         Hard limit: 150 words maximum. No prose paragraphs.
         All 4 sections mandatory. -->
    ```

    Template body:
    ```markdown
    # SUMMARY: {NN-MM} {Section Name}

    ## Facts
    - Type: {Equipment Module | State Machine | Interface | Overview | Appendix}
    - States: {N} {(PackML compliant) if standards enabled}
    - Parameters: {N}
    - Interlocks: {N}
    - I/O: {N} DI, {N} DO, {N} AI, {N} AO

    ## Key Decisions
    - {Critical decision 1 documented in this section}
    - {Critical decision 2}

    ## Dependencies
    - {What this section depends on: interlock met {EM-XXX}, interface naar {system}}
    - {What this section provides to others}

    ## Cross-refs
    - {Interlock IL-XXX-NN -> zie section-ref}
    - {HMI scherm -> zie section-ref}
    ```

    Include validation rules as HTML comments:
    - Word count: 100-150 (hard limit)
    - All 4 section headers present
    - Facts: bullet list with counts
    - Cross-refs: standardized format (ID -> zie section-ref)
    - No prose paragraphs

    **verification.md** (~60-80 lines):

    HTML comment block:
    ```
    <!-- TEMPLATE: Phase VERIFICATION.md
         Generated by: doc-verifier subagent after running 5-level cascade
         Consumed by: /doc:plan-phase --gaps (reads gaps for fix plan generation)
         Consumed by: Engineer (quick scan of phase verification status)

         Summary table at top for quick scan.
         Detailed findings per truth below.
         Gap descriptions only -- no fix suggestions. -->
    ```

    Template body:
    ```markdown
    # Phase {N}: {Phase Name} - Verification

    **Verified:** {DATE}
    **Status:** {PASS | GAPS_FOUND}
    **Cycle:** {1 | 2} of 2

    ## Summary

    | Truth | Exists | Substantive | Complete | Consistent | Standards | Status |
    |-------|--------|-------------|----------|------------|-----------|--------|
    | {Truth 1} | {check/gap} | {check/gap/-} | {check/gap/-} | {check/gap/-} | {check/gap/-/N/A} | {PASS/GAP} |

    **Quick scan:** {N} gaps found across {M} truths. {K} truths passed all levels.

    ## Detailed Findings

    ### Truth {N}: {Truth description}

    **Status:** {PASS | GAP (Level N - Level Name)}

    **Verification trail:**
    - Level 1 (Exists): {PASS/GAP} - {evidence}
    - Level 2 (Substantive): {PASS/GAP/-} - {evidence}
    - Level 3 (Complete): {PASS/GAP/-} - {evidence}
    - Level 4 (Consistent): {PASS/GAP/-} - {evidence}
    - Level 5 (Standards): {PASS/GAP/N/A/-} - {evidence}

    **Gap description:** (only if GAP)
    {Descriptive text: what is wrong or missing. No fix suggestions.
    Use factual language: "Section X uses value Y but CONTEXT.md specifies Z."}

    **Evidence:**
    - File: {path to file}
    - Context: {path to CONTEXT.md or standards reference}

    ## Cross-Reference Status

    | Source | Target | Status | Note |
    |--------|--------|--------|------|
    | {section-ref} | {section-ref} | {resolved/pending/warning} | {reason} |
    ```

    Include validation rules as HTML comments:
    - Summary table must have all truth rows
    - Status must be PASS or GAPS_FOUND (no other values)
    - Cycle number tracks gap closure iterations (1 or 2)
    - Detailed findings: one section per truth
    - Gap descriptions: descriptive language only (no imperatives)
    - Evidence: specific file paths and line numbers where possible

    **cross-refs.md** (~30-40 lines):

    HTML comment block:
    ```
    <!-- TEMPLATE: Phase CROSS-REFS.md
         Created/updated by: doc-writer subagents during writing
         Consumed by: doc-verifier (cross-reference validation)
         Consumed by: /doc:complete-fds (assembly-time reference resolution)

         Each writer appends its references. First writer creates the file.
         Full context captured per reference for reliable assembly resolution. -->
    ```

    Template body:
    ```markdown
    # Phase {N}: {Phase Name} - Cross-References

    **Last updated:** {DATE}

    ## References

    | Source Section | Target Section | Type | Context | Status |
    |----------------|----------------|------|---------|--------|
    | {NN-MM section name} | {NN-MM section name or external ref} | {depends-on/related-to/see-also} | {One sentence explaining the relationship} | {resolved/pending} |

    ## Legend

    - **depends-on:** Source requires target to be complete (functional dependency)
    - **related-to:** Source references target for complementary information
    - **see-also:** Source suggests target for additional context
    - **resolved:** Target section exists and is written
    - **pending:** Target section not yet written or in different phase
    ```

    Include validation rules as HTML comments:
    - Types must be one of: depends-on, related-to, see-also
    - Status must be: resolved or pending
    - Context column: one sentence, not empty
    - Source and Target: use NN-MM format for same-phase, phase-N/NN-MM for cross-phase
