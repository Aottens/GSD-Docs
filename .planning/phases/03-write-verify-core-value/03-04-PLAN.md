    **Updated Step 8: Gap Closure Mode (--gaps flag)**

    This step runs ONLY if --gaps flag was detected in Step 1.

    **8a. Load VERIFICATION.md:**
    - Check for VERIFICATION.md in the phase directory: `.planning/phases/{NN}-{name}/{NN}-VERIFICATION.md`
    - If not found: show error message: "No VERIFICATION.md found. Run `/doc:verify-phase {N}` first."
    - Read VERIFICATION.md and parse:
      - Status (must be GAPS_FOUND -- if PASS, show message: "No gaps to fix. Phase already verified.")
      - Cycle count (from header)
      - Summary table rows (each truth with pass/gap status per level)
      - Detailed findings sections (gap descriptions, evidence, affected files)

    **8b. Extract and Group Gaps:**
    For each truth with status GAP in the summary table:
    - Read detailed findings section for that truth
    - Extract: truth description, failure level, gap description, evidence (file paths)
    - Group gaps by artifact (file path):
      - If multiple truths fail for the same CONTENT.md file: group into one fix plan
      - If truths fail for different files: separate fix plans
    - Also group related gaps (e.g., consistency + completeness gap for same section = one fix plan)

    **8c. Gap Preview (per user decision: "show engineer the list of gaps and let them confirm"):**
    Display the extracted gaps to the engineer:
    ```
    DOC > GAP CLOSURE FOR PHASE {N}

    Gaps to address ({count}):
    1. {Truth}: {Gap description} (Level {N}: {level name})
       File: {affected file}
    2. {Truth}: {Gap description} (Level {N}: {level name})
       File: {affected file}

    Fix plans to generate: {count}
    ```

    Note: The preview is informational. In non-interactive mode (plan-phase has no AskUserQuestion), display the preview and proceed. If the engineer wants to exclude gaps, they can delete the generated fix plans before running write-phase.

    **8d. Generate Fix Plans:**
    For each gap group, generate a targeted fix PLAN.md:

    - Find next plan number (after existing plans in phase directory)
    - File naming: `{NN}-{MM}-PLAN.md` where MM continues from last existing plan

    Fix plan format (same doc PLAN.md format as Step 6):
    ```markdown
    ---
    phase: {N}
    plan: {MM}
    name: Fix: {brief gap description}
    type: {same as original section type}
    wave: 1
    depends_on: []
    autonomous: true
    gap_closure: true
    original_plan: {NN-MM of original plan that produced the content}
    ---

    # Fix: {Gap Description}

    ## Goal
    Close the verification gap: {truth that failed}.
    Original content in {file} failed at Level {N} ({level name}).

    ## Gap Description
    {Full gap description from VERIFICATION.md -- copied verbatim}

    ## Evidence
    {Evidence from VERIFICATION.md -- file paths, line numbers, context references}

    ## Context
    {Relevant decisions from phase CONTEXT.md that apply to this fix}
    - {Decision that was violated or not implemented}

    ## Template
    @~/.claude/gsd-docs-industrial/templates/fds/{original template reference}

    ## Verification
    - [ ] {Specific check that the gap is closed}
    - [ ] {Truth that must now pass verification}
    - [ ] Content still passes all previously passing verification levels
    ```

    All gap closure plans get wave: 1 (they are independent targeted fixes).

    **8e. Self-Verify Fix Plans:**
    Run the same inline verification checks as Step 7 on generated fix plans:
    - Structure check (all required sections present)
    - Template references valid
    - No circular dependencies
    - Naming convention correct

    **8f. Completion:**
    Display banner: `DOC > GAP FIX PLANS GENERATED`

    ```
    Wave: 1 (all parallel)
    Files: {list of generated plan files}

    ```

    Create `gsd-docs-industrial/templates/engineer-todo.md` -- the template for gap closure escalation when the verification loop exceeds max 2 cycles.

    This template is referenced by verify-phase workflow (Step 7: Max Iterations Escalation) to create ENGINEER-TODO.md in the .planning/ directory.

    **Template** (~30-40 lines):

    HTML comment block at top (matching existing template pattern):
    ```
    <!-- TEMPLATE: ENGINEER-TODO.md
         Created by: /doc:verify-phase when gap closure loop exceeds max 2 cycles
         Purpose: Tracks remaining gaps that require manual engineer intervention
         Effect: Phase is BLOCKED until gaps are resolved or explicitly accepted

         This file is created automatically. Engineer resolves gaps manually,
         then re-runs /doc:verify-phase to re-check.
         To unblock without fixing: mark gaps as "accepted" with rationale. -->
    ```

    Template body:
    ```markdown
    # Phase {N}: {Phase Name} - Manual Intervention Required

    **Date:** {DATE}
    **Reason:** Gap closure loop terminated after max 2 cycles
    **Phase status:** BLOCKED

    ## Instructions

    The automated gap closure loop has run {cycle_count} times without resolving all gaps.
    Remaining gaps require manual engineer review and action.

    **To resolve:**
    1. Review each gap below
    2. Fix the content file directly or provide missing information
    3. Run `/doc:verify-phase {N}` to re-verify

    **To accept gaps (unblock without fixing):**
    1. Add `**Accepted:** {rationale}` under the gap
    2. Run `/doc:verify-phase {N}` -- accepted gaps are skipped

    ## Remaining Gaps

    ### Gap {N}: {Truth description}

    **Severity:** {High | Medium | Low}
    **Level:** {Level N - Level Name} (where verification cascade stopped)
    **Description:** {Gap description from VERIFICATION.md}
    **Evidence:**
    - File: {path}
    - Context: {CONTEXT.md reference or standards reference}
    **Attempted fixes:** {count} automated cycles
    **Suggested action:**
    1. {Actionable step for engineer}
    ```

    Include validation rules as HTML comments:
    - Severity must be High, Medium, or Low
    - Level references the 5-level cascade (1-Exists, 2-Substantive, 3-Complete, 4-Consistent, 5-Standards)
    - Description copied verbatim from VERIFICATION.md
    - "Accepted" marker pattern for unblocking
  </action>
