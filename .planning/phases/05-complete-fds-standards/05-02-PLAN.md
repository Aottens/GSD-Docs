<task type="auto">
  <name>Task 1: Create FDS structure template (fds-structure.json)</name>
  <files>gsd-docs-industrial/templates/fds-structure.json</files>
  <action>
    Create a JSON file that defines the canonical section ordering for a complete FDS document. This template is consumed by the /doc:complete-fds workflow to reorder phase content into the correct FDS document structure.

    **Structure requirements:**
    - Hierarchical section definitions matching IEC-style numbering (1, 1.1, 1.2, 2, 2.1, etc.)
    - Each section has: id, title (bilingual: en + nl), depth (1-4), required (boolean), source_type (which phase/content type maps here), placeholder_stub (text to use if section content not yet written)
    - Support all 4 project types (Type A/B/C/D) with type-conditional sections (e.g., BASELINE only for Type C/D)

    **Canonical FDS section ordering:**

    ```
    1. Introduction
       1.1 Purpose and Scope
       1.2 Document Conventions
       1.3 References
       1.4 Abbreviations
    2. System Overview
       2.1 System Description
       2.2 System Architecture
       2.3 Equipment Overview
    3. Functional Requirements
       3.1 Operating Modes
       3.2 State Machine
       3.3 General Operating Sequence
    4. Equipment Modules
       4.x [Dynamic: one subsection per equipment module from ROADMAP]
       4.x.1 Function Description
       4.x.2 Operating States
       4.x.3 Parameters and Setpoints
       4.x.4 Interlocks and Conditions
       4.x.5 I/O List
       4.x.6 HMI Visualization (optional)
       4.x.7 Alarms and Events (optional)
       4.x.8 Communication Interfaces (optional)
       4.x.9 Maintenance (optional)
    5. Interfaces
       5.1 PLC Interfaces
       5.2 SCADA/HMI Interface
       5.3 External System Interfaces
    6. Safety
       6.1 Safety Functions
       6.2 Emergency Stop
       6.3 Safety Interlocks
    7. Appendices
       7.1 Complete I/O List
       7.2 Signal Descriptions
       7.3 Mermaid Diagram Notes
    ```

    **JSON format:**
    ```json
    {
      "version": "1.0",
      "description": "Canonical FDS section ordering for document assembly",
      "sections": [
        {
          "id": "1",
          "title": { "en": "Introduction", "nl": "Introductie" },
          "depth": 1,
          "required": true,
          "source_type": "system-overview",
          "children": [
            {
              "id": "1.1",
              "title": { "en": "Purpose and Scope", "nl": "Doel en Scope" },
              "depth": 2,
              "required": true,
              "source_type": "system-overview"
            }
          ]
        }
      ],
      "dynamic_sections": {
        "equipment_modules": {
          "parent_id": "4",
          "source": "roadmap_phases",
          "per_unit": true,
          "subsections": ["function", "states", "parameters", "interlocks", "io_list", "hmi", "alarms", "communication", "maintenance"],
          "required_subsections": ["function", "states", "parameters", "interlocks", "io_list"],
          "optional_subsections": ["hmi", "alarms", "communication", "maintenance"]
        }
      },
      "type_conditional": {
        "baseline_section": {
          "applies_to": ["C", "D"],
          "parent_id": "1",
          "after": "1.3",
          "title": { "en": "Baseline System", "nl": "Bestaand Systeem" }
        }
      },
      "placeholder_stub": "[TO BE COMPLETED]"
    }
    ```

    **Key design decisions per CONTEXT.md:**
    - Section ordering follows this predefined template, NOT ROADMAP phase order (locked decision)
    - Unwritten sections appear as placeholder stubs with "[TO BE COMPLETED]" markers (locked decision)
    - Equipment modules (section 4) are dynamic -- one subsection per equipment module discovered in ROADMAP phases
    - The assembly workflow reads this template to determine: (a) which sections exist, (b) in what order, (c) which are required vs optional
    - Type-conditional sections (like Baseline) only appear for matching project types

    **Mapping logic (documented in JSON comments or separate mapping field):**
    - The `source_type` field tells the assembly workflow WHERE to find content for each section
    - For static sections: maps to a ROADMAP phase keyword or content type
    - For dynamic sections (equipment modules): maps to individual phase CONTENT.md files based on equipment module name matching
    - Mapping is by convention (phase goal text, equipment module names) -- not hardcoded file paths

    - Contains type_conditional for baseline (Type C/D only)
