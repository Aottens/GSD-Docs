<task type="auto">
  <name>Task 2: Create release version management workflow</name>
  <files>gsd-docs-industrial/workflows/release.md</files>
  <action>
    Create comprehensive version management workflow. Target: ~400-500 lines.
    Wrap in `<workflow>` tags.

    **Step 1: Parse arguments and determine version**
    - Validate --type argument: must be "client" or "internal"
    - Read STATE.md Versions section for current FDS version
    - If Versions section doesn't exist or shows "v0.0 (not started)": initialize at v0.1 for internal, error for client ("No assembled FDS exists yet. Run /doc:complete-fds first.")
    - Calculate next version:
      - Internal: minor bump (v0.3 -> v0.4, v1.2 -> v1.3)
      - Client: major bump with minor reset (v0.4 -> v1.0, v1.3 -> v2.0)
    - Display: "Current: v{current} -> Target: v{next} ({type})"
    - Per specific idea: "v0.x internal, v1.0 first customer-facing release, v1.1 internal revision, v2.0 next customer release"

    **Step 2: Verification gate (client releases only)**
    - ASBL-09/ASBL-10: Client releases require all phases to pass verification
    - Read STATE.md and ROADMAP.md to check all phase statuses
    - If any phase is NOT Verified: display error listing unverified phases
    - If --force: continue with warning "FORCED RELEASE: Some phases are not fully verified. Document will carry 'DRAFT' annotation."
    - If NOT --force: block release, instruct engineer to run /doc:verify-phase for each unverified phase
    - Internal releases: skip verification gate (drafts don't need full verification)
    - Per locked decision: "Client release gate: all phases must pass verification, but --force flag allows release with warnings"

    **Step 3: Check for assembled FDS document**
    - Look for FDS-{PROJECT_NAME}-v{current}.md in project root
    - If not found: display error "No assembled FDS document found for v{current}. Run /doc:complete-fds first."
    - If found: confirm it exists and is not a zero-byte file

    **Step 4: Archive current version**
    - Archive destination: .planning/archive/v{next}/
    - Check if archive directory already exists -- if so, abort with error "Archive v{next} already exists. Version v{next} was already released." (prevent overwrite, per research pitfall 5)
    - Copy .planning/phases/ to archive/v{next}/phases/
    - Copy .planning/ROADMAP.md to archive/v{next}/ROADMAP.md
    - Copy current FDS document to archive/v{next}/
    - Copy .planning/assembly/ reports (XREF-REPORT.md, COMPLIANCE.md, ENGINEER-TODO.md if they exist) to archive/v{next}/
    - Per locked decision: "Previous version files archived to .planning/archive/vX.Y/ on new release"

    **Step 5: Generate revision history entry**
    - Per locked decision (hybrid approach): auto-generate from git as starting draft
    - Read git log since last release tag (or all history if no tag exists)
    - Format as table rows: Version | Date | Author | Description
    - Auto-generate description from commit messages (group by phase/plan)
    - For client releases: include a marker "ENGINEER: Review and edit revision history before distribution"
    - Store generated revision history for inclusion in document front matter

    **Step 6: Rename/create versioned FDS document**
    - For internal: copy current FDS document to FDS-{PROJECT_NAME}-v{next}.md
    - For client with --force: filename becomes FDS-{PROJECT_NAME}-v{next}-DRAFT.md
    - For client without --force: filename is FDS-{PROJECT_NAME}-v{next}.md
    - If previous version document exists at old version name: keep it (archived in Step 4)
    - Update revision history in the document front matter with new entry from Step 5

    **Step 7: Update STATE.md**
    - Update Versions section: `- FDS: v{next} ({type} release, {date})`
    - SDS version unchanged (independent versioning per ASBL-12)
    - Add decision entry: "FDS v{next} released ({type}) - {date}"
    - Update Current focus: "FDS v{next} released"
    - Update Session Continuity with release information

    **Step 8: Git commit and tag**
    - Commit message: "release: FDS v{next} ({type})"
    - Include files: FDS document, STATE.md, archive directory
    - Create git tag: fds-v{next}
    - Do NOT push (per GSD convention -- user pushes explicitly)

    **Step 9: Display summary**
    - DOC > banner: "FDS v{next} Released ({type})"
    - Previous version: v{current}
    - New version: v{next}
    - Document: FDS-{PROJECT_NAME}-v{next}.md
    - Archive: .planning/archive/v{next}/
    - Revision history: generated (remind engineer to review for client releases)
    - Git tag: fds-v{next}
    - Next steps:
      - For internal: "Continue working. Run /doc:complete-fds after changes, then /doc:release --type internal for next version."
      - For client: "Review revision history in the document. Run git push && git push --tags to share the release."

    **SDS versioning note (ASBL-12):**
    - The workflow handles FDS versioning only
    - SDS versioning is handled by /doc:generate-sds (Phase 7)
    - STATE.md Versions section tracks both independently: `- FDS: vX.Y` and `- SDS: vX.Y (based on FDS vA.B)`
    - This workflow updates only the FDS line, never touches SDS
    - Per locked decision: "FDS and SDS versioned independently; SDS references source FDS version on frontpage"
