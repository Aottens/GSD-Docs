  <action>
    Create 4 reference data files in markdown format (parsed to structured data at load time by the check-standards workflow).

    **PackML STATE-MODEL.md:**
    - Valid states per ISA-TR88.00.02: IDLE, STARTING, EXECUTE, COMPLETING, COMPLETE, RESETTING, HOLDING, HELD, UNHOLDING, SUSPENDING, SUSPENDED, UNSUSPENDING, STOPPING, STOPPED, ABORTING, ABORTED, CLEARING
    - Valid state transitions as a table: From State | To State | Trigger (e.g., IDLE -> STARTING -> Start command)
    - State categories: Acting (transient), Wait (stable), Dual (can be either)
    - Include the standard state model diagram description (reference only, not Mermaid -- too complex)

    **PackML UNIT-MODES.md:**
    - Valid unit modes: PRODUCTION, MAINTENANCE, MANUAL, others per TR88.00.02
    - Mode-to-state mapping (which states are valid in which modes)
    - Mode transition rules

    **ISA-88 EQUIPMENT-HIERARCHY.md:**
    - 4-level hierarchy: Process Cell > Unit > Equipment Module > Control Module
    - Valid nesting rules (what can contain what)
    - Maximum depth: 4 levels
    - Examples of correct vs incorrect hierarchy usage

    **ISA-88 TERMINOLOGY.md:**
    - Canonical terms: Process Cell, Unit, Equipment Module, Control Module, Phase, Operation, Unit Procedure, Procedure
    - Common incorrect alternatives and their correct replacements (e.g., "machine" -> "Unit", "component" -> "Control Module")
    - Term definitions (one sentence each)

    **Format for all files:**
    - HTML comment documentation block at top (following template pattern from templates/)
    - Markdown tables for structured data (easily parsed by Claude during validation)
    - Human-readable with clear section headers
    - NO JSON or YAML -- pure markdown per research recommendation ("markdown for human readability, parsed at load time")

    **Important:** These are reference data files, not templates. They contain FIXED content based on ISA standards, not placeholders. They are the source of truth for validation checks.
  </action>

    - isa-88/EQUIPMENT-HIERARCHY.md contains all 4 hierarchy levels

  <action>
    **Command file (commands/doc/check-standards.md):**
    - Lean orchestrator following established pattern (see status.md reference)
    - name: doc:check-standards
    - description: "Run standalone standards compliance checks and generate COMPLIANCE.md report"
    - allowed-tools: Read, Write, Bash, Glob, Grep (no Task tool -- reads content directly, no subagents needed)
    - @-reference to workflows/check-standards.md
    - @-reference to CLAUDE-CONTEXT.md and ui-brand.md
    - @-reference to PROJECT.md (for standards config)
    - Optional argument: phase number (defaults to all phases)
    - Copy to gsd-docs-industrial/commands/check-standards.md (version-tracked copy, must be identical)

    **Workflow file (gsd-docs-industrial/workflows/check-standards.md):**
    - Target: ~300-400 lines (composable validator pattern from research)
    - Wrap in `<workflow>` tags (established pattern)

    **Step 1: Load PROJECT.md and check standards config**
    - Read PROJECT.md, extract `standards:` YAML block
    - If `standards.packml.enabled: false` (or absent) AND `standards.isa88.enabled: false` (or absent): display info message "No standards enabled in PROJECT.md. Enable with standards.packml.enabled: true or standards.isa88.enabled: true" and exit cleanly
    - Extract severity per standard: `standards.packml.severity: error|warning` and `standards.isa88.severity: error|warning`
    - Per user decision: configurable severity per standard, default to "warning" if not specified

    **Step 2: Load enabled reference data**
    - If PackML enabled: read STATE-MODEL.md and UNIT-MODES.md, extract valid states list, valid transitions table, valid modes list
    - If ISA-88 enabled: read EQUIPMENT-HIERARCHY.md and TERMINOLOGY.md, extract valid hierarchy levels, valid terms, incorrect alternatives
    - Lazy loading: ONLY load files for enabled standards (per pitfall 3 from research -- performance)
    - Store extracted data as structured context for Step 3

    **Step 3: Discover and read target content**
    - If phase argument provided: read all *-CONTENT.md files from that phase directory
    - If no argument: read all *-CONTENT.md files from all phase directories under .planning/phases/
    - Parse each CONTENT.md to identify: state references (state names in text and tables), equipment hierarchy terms, terminology usage
    - Track source file and line context for each finding

    **Step 4: Run PackML checks (if enabled)**
    - STND-03a: State name enforcement -- find all state references in content, compare against valid states list from STATE-MODEL.md. Flag non-standard state names. Per user decision (Claude's discretion): exact match for standard PackML states, allow common synonyms only when clearly defined in reference data.
    - STND-03b: Transition validation -- find state transition descriptions (in tables or text), verify each transition exists in valid transitions table. Flag invalid transitions.
    - STND-03c: Mode checking -- find mode references, verify against valid modes list from UNIT-MODES.md. Flag non-standard modes.
    - Each check produces: status (pass/fail), severity (from PROJECT.md config), location (file + context), message, remediation hint

    **Step 5: Run ISA-88 checks (if enabled)**
    - STND-04a: Terminology enforcement -- find equipment-related terms in content, compare against canonical terms from TERMINOLOGY.md. Flag incorrect alternatives (e.g., "machine" instead of "Unit"). Per user decision: enforce canonical ISA-88 terms.
    - STND-04b: Hierarchy depth validation -- analyze equipment hierarchy in content (from heading structure and nesting), verify maximum 4 levels. Flag deeper nesting.
    - STND-04c: Hierarchy consistency -- verify correct containment (Process Cell contains Units, Units contain Equipment Modules, EMs contain Control Modules). Flag incorrect nesting.
    - Each check produces same structure as PackML checks

    **Step 6: Generate COMPLIANCE.md report**
    - Write to `.planning/phases/{NN}-{slug}/COMPLIANCE.md` (if phase-specific) or `.planning/COMPLIANCE.md` (if all phases)
    - Use compliance-report.md template format
    - Include: header with date/scope/standards checked, summary table (standard | checks | pass | fail | score), detailed findings per standard with pass/fail per check and remediation hints
    - Overall result: COMPLIANT (all pass or only warnings), NON-COMPLIANT (any errors), PARTIALLY COMPLIANT (mix)

    **Step 7: Display summary**
    - DOC > banner with standards check result
    - Table: standard name | severity | checks run | passed | failed | result
    - If NON-COMPLIANT: "Fix issues marked as 'error' severity before /doc:complete-fds"
    - If COMPLIANT: "Standards checks passed. Ready for assembly."
    - Always: "Full report: {path to COMPLIANCE.md}"

    **COMPLIANCE.md template (gsd-docs-industrial/templates/reports/compliance-report.md):**
    - HTML comment documentation block explaining template purpose, who creates it, who consumes it
    - Header: Date, Scope (phase or all), Standards Checked, Overall Result
    - Summary table: Standard | Severity | Checks | Passed | Failed | Score
    - Per-standard detail sections: check name, status (pass/fail), severity, location (file:line), finding, remediation hint
    - Footer: "Generated by /doc:check-standards" with timestamp

    **Integration hook for verify-phase (STND-05):**
    - The existing verify-phase workflow already has Level 5 (Standards) in its 5-level cascade
    - Add a note in the workflow file (Step 4 standards level) referencing this check-standards workflow
    - When verify-phase reaches Level 5 AND standards are enabled in PROJECT.md: invoke the same validation logic from Steps 2-5 above
    - The check-standards workflow must be usable both standalone (via command) and as imported logic (via verify-phase Step 4 Level 5)
    - To enable this: structure the workflow so Steps 2-5 are the reusable core, Steps 1 and 6-7 are the standalone wrapper

    **STND-06 enforcement:**
    - Standards are NEVER pushed. The workflow checks standards.{name}.enabled in PROJECT.md before doing anything
    - If standards are not enabled, the workflow displays an informational message and exits -- it never suggests enabling them
    - Default state is disabled. Only explicit engineer action in PROJECT.md enables standards
