  <name>Task 1: Create /doc:complete-fds command and full assembly workflow</name>

  <action>
    **Command file (commands/doc/complete-fds.md):**
    - Lean orchestrator following established pattern
    - name: doc:complete-fds
    - description: "Assemble all verified phases into a single FDS document with section numbering, cross-reference resolution, and standards compliance"
    - allowed-tools: Read, Write, Bash, Glob, Grep (no Task tool -- orchestrates directly, no subagents)
    - @-reference to workflows/complete-fds.md
    - @-reference to CLAUDE-CONTEXT.md and ui-brand.md
    - @-reference to PROJECT.md, STATE.md, ROADMAP.md
    - Optional flag: --force (generate with [BROKEN REF] placeholders and DRAFT suffix when broken references exist)
    - Copy to gsd-docs-industrial/commands/complete-fds.md (version-tracked, must be identical)

    **Workflow file (gsd-docs-industrial/workflows/complete-fds.md):**
    - Target: ~700-900 lines (this is the most complex workflow in the project)
    - Wrap in `<workflow>` tags (established pattern)

    **Step 1: Pre-flight verification**
    - Read STATE.md and ROADMAP.md to determine all phases
    - For each phase: check if VERIFICATION.md exists with status PASS
    - If any phase does NOT have PASS status: display error with list of unverified phases and instruction to run /doc:verify-phase for each. Block assembly unless --force flag is provided.
    - ASBL-01: "verifies all phases PASS before assembly"
    - Check for command conflict (same pattern as verify-phase Step 1a): if STATE.md shows IN_PROGRESS for a different command, warn and ask for confirmation

    **Step 2: Load project configuration**
    - Read PROJECT.md for: project name, client name, language (en/nl), project type (A/B/C/D), version (from Versions section in STATE.md), standards config
    - Read fds-structure.json (from Plan 02) for canonical section ordering
    - Read front matter templates (from Plan 02)
    - Determine output version from STATE.md Versions section (FDS version)

    **Step 3: Discover and collect all CONTENT.md files**
    - Scan all phase directories under .planning/phases/ for *-CONTENT.md files
    - For each CONTENT.md: read content, extract metadata (phase number, plan number, heading structure)
    - Also collect CROSS-REFS.md from each phase directory for cross-reference data
    - Also collect any existing COMPLIANCE.md files
    - Track: file path, phase, plan, content, headings, cross-references

    **Step 4: Apply FDS structure template ordering**
    - Read fds-structure.json sections hierarchy
    - Map each collected CONTENT.md to a section in the structure template based on source_type matching
    - For equipment modules (dynamic section): match CONTENT.md files to equipment module subsections by name/heading
    - Reorder collected content to match canonical FDS ordering
    - For sections in fds-structure.json with no matching CONTENT.md: create placeholder stub with "[TO BE COMPLETED]" marker and the section title
    - Per locked decision: "Section ordering follows a predefined FDS structure template, not ROADMAP phase order"
    - Per locked decision: "Unwritten sections appear as placeholder stubs with [TO BE COMPLETED] markers"
    - Pitfall 1 mitigation: ordering MUST happen BEFORE numbering (never number then reorder)

    **Step 5: Apply hierarchical section numbering**
    - Walk through ordered sections and apply IEC-style numbering: 1, 1.1, 1.2, 2, 2.1, 2.1.1, etc.
    - Numbering based on heading depth in each section's content
    - Heading depth mapping: # = top level (1, 2, 3...), ## = second level (1.1, 1.2...), ### = third level (1.1.1...), #### = fourth level
    - Prepend section number to each heading text (e.g., "## Function Description" becomes "## 4.1.1 Function Description")
    - Store section number mappings for cross-reference resolution in Step 6
    - Build symbol table: map symbolic IDs (plan ID like "03-02", equipment module name like "EM-200") to their assigned section numbers
    - Per locked decision: "Hierarchical (IEC-style) section numbering: 1.1, 1.2, 2.1.1, etc."

    **Step 6: Resolve cross-references**
    - Read all CROSS-REFS.md files collected in Step 3
    - For each symbolic reference in the assembled content (patterns: "see NN-MM", "see phase-N/NN-MM", "zie NN-MM", "zie §NN-MM"):
      - Look up in symbol table from Step 5
      - If found: replace with "see Section X.Y.Z" (or "zie Paragraaf X.Y.Z" for Dutch)
      - If NOT found: replace with "see {original} [BROKEN REF]" (or leave unchanged in non-force mode)
    - Track all references: source section, target, resolution status (resolved/broken)
    - Detect orphan sections: sections that exist in the document but are never referenced by any other section
    - Per locked decision: "Cross-references render as plain text section numbers, not clickable links"
    - Per locked decision: "Unresolved references render as [BROKEN REF] placeholders"
    - Pitfall 2 mitigation: build symbol table AFTER numbering is finalized (Step 5 complete)

    **Step 7: Cross-reference validation**
    - Count broken references from Step 6
    - ASBL-04: If broken references exist AND --force NOT provided: display error listing all broken references with source and target, block assembly, suggest running /doc:verify-phase or fixing content
    - ASBL-06: If broken references exist AND --force IS provided: continue assembly with [BROKEN REF] placeholders, append "DRAFT" suffix to output filename
    - ASBL-05: Report orphan sections in XREF-REPORT.md with severity assessment (Claude's discretion based on section type -- equipment modules are more concerning than appendices)

    **Step 8: Run standards compliance checks (if enabled)**
    - Check PROJECT.md for enabled standards
    - If any standards enabled: invoke the standards validation logic from check-standards workflow (Steps 2-5 from Plan 01's workflow)
    - If standards check finds errors (severity: error) AND --force NOT provided: block assembly, display issues, suggest running /doc:check-standards for full report
    - If standards check finds errors AND --force IS provided: continue with warnings in output
    - If standards check finds only warnings or passes: continue normally
    - Results feed into COMPLIANCE.md generation

    **Step 9: Generate front matter**
    - Read front matter templates from gsd-docs-industrial/templates/frontmatter/
    - Generate title page: fill {PROJECT_NAME}, {CLIENT}, {VERSION}, {DATE}, {LANGUAGE} placeholders from PROJECT.md
    - Generate table of contents: walk through all numbered sections, create hierarchical TOC with section numbers and titles
    - Generate revision history: read git log for recent commits, format as revision table entries (placeholder for engineer edits)
    - Generate abbreviations list: scan assembled content for uppercase 2+ letter abbreviation patterns, look up in pre-populated abbreviation dictionary (from abbreviations.md template), add any manually-defined abbreviations from PROJECT.md
    - Select language variant based on PROJECT.md language setting
    - Per locked decision: "Full auto-generated front matter: title page, revision history, table of contents, abbreviations list"

    **Step 10: Assemble final document**
    - Concatenate in order: front matter (title page, TOC, revision history, abbreviations) + page break + all numbered sections
    - Add page break markers between major sections (# headings)
    - Write to output file: `FDS-{PROJECT_NAME}-v{VERSION}.md` in project root
    - If --force was used with broken refs: filename becomes `FDS-{PROJECT_NAME}-v{VERSION}-DRAFT.md`

    **Step 11: Generate ENGINEER-TODO.md for complex diagrams**
    - Scan assembled content for Mermaid diagram blocks (```mermaid ... ```)
    - Apply complexity heuristic: >20 nodes OR >4 nesting levels OR >50 lines = exceeds Mermaid complexity
    - For each exceeding diagram: record section reference (number + title), diagram type (state/flowchart/sequence), description (first comment or context), priority (High for state machines, Medium for flowcharts, Low for sequence diagrams)
    - Write to .planning/assembly/v{VERSION}/ENGINEER-TODO.md using format: section reference, diagram type, description, priority
    - ASBL-07 + KNOW-04: "ENGINEER-TODO.md lists diagrams too complex for Mermaid with section reference, type, description, priority"
    - Note: this is a DIFFERENT use of ENGINEER-TODO.md than the gap-closure version from Phase 3 -- this is assembly-specific, located in .planning/assembly/v{VERSION}/ not in phase directories

    **Step 12: Generate reports**
    - Write XREF-REPORT.md to .planning/assembly/v{VERSION}/XREF-REPORT.md using template from this plan's Task 2
    - Content: all references with source section, target, resolution status, orphan sections
    - If standards were checked: write COMPLIANCE.md to .planning/assembly/v{VERSION}/COMPLIANCE.md (reuse template from Plan 01)

    **Step 13: Archive phase files**
    - ASBL-08: Copy .planning/phases/ to .planning/archive/v{VERSION}/phases/
    - Copy .planning/ROADMAP.md to .planning/archive/v{VERSION}/ROADMAP.md
    - Check if archive directory already exists -- if so, create timestamped backup (pitfall 5 from research)
    - Archive is a COPY, not a move -- original phase files remain for continued work

    **Step 14: Update STATE.md**
    - Update Current Operation section: command = complete-fds, status = COMPLETE
    - Update Versions section with current FDS version
    - Update Current focus with assembly completion note

    **Step 15: Display summary**
    - DOC > banner with assembly result
    - Output file path and size
    - Section count (numbered sections)
    - Cross-reference stats: total references, resolved, broken, orphan sections
    - Standards compliance status (if checked)
    - ENGINEER-TODO.md count (complex diagrams requiring manual work)
    - Archive location
    - Next steps: "/doc:release --type internal" to bump version or "/doc:release --type client" to promote to client release
  </action>

  <action>
    Create the XREF-REPORT.md template used by /doc:complete-fds to report cross-reference resolution results.

    **Template structure:**
    - HTML comment documentation block (who creates, who consumes, validation rules)
    - Header: Date, FDS Version, Total References, Resolved, Broken, Orphan Sections
    - Summary table: quick overview of resolution stats

    **Sections:**

    1. **Resolution Summary**
       - Table: Status | Count | Percentage
       - Resolved, Broken, Total

    2. **Resolved References**
       - Table: Source Section | Target Section | Symbolic ID | Final Number | Context
       - Lists all successfully resolved cross-references

    3. **Broken References**
       - Table: Source Section | Symbolic ID | Reason | Suggested Fix
       - Lists all unresolved references with explanation (target not found, ambiguous match, etc.)
       - If --force was used: note that these appear as [BROKEN REF] in the assembled document

    4. **Orphan Sections**
       - Table: Section Number | Section Title | Severity | Notes
       - Lists sections that are never referenced by any other section
       - Severity: Claude's discretion based on section type (per CONTEXT.md)
       - Equipment modules: HIGH (should always be referenced from System Overview)
       - Appendices: LOW (standalone by nature)
       - Introduction/Safety: MEDIUM (typically self-contained but should be referenced)

    5. **Statistics**
       - Total sections in document
       - Total cross-references found
       - Resolution rate (percentage)
       - Most-referenced section
       - Most-referencing section (section with most outgoing references)

    **Format:** Follow existing template pattern (cross-refs.md, verification.md) with HTML comment block and structured markdown.

    Per locked decision: "Always generate XREF-REPORT.md listing all references, resolution status, and orphan sections"
    Per locked decision: "Orphan handling: Claude's discretion on severity based on section type"
  </action>

    - Contains all 5 sections (Resolution Summary, Resolved References, Broken References, Orphan Sections, Statistics)
    - Orphan severity column present with type-based classification guidance
    - Broken references section includes "Suggested Fix" column
  </verify>

The template is consumed by /doc:complete-fds to generate assembly-time cross-reference reports.
