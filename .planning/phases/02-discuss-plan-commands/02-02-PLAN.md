<task type="auto">
  <name>Task 2: Create discuss-phase workflow file</name>
  <files>
    gsd-docs-industrial/workflows/discuss-phase.md

  <action>
    Create `gsd-docs-industrial/workflows/discuss-phase.md` -- the detailed execution logic for phase discussion. Adapt from the GSD discuss-phase workflow (C:/Users/Aotte/.claude/get-shit-done/workflows/discuss-phase.md, 422 lines) for the FDS domain.

    Target: 400-500 lines. Wrap ALL content in `<workflow>` tags. Use numbered `## Step N:` headings.

    **Step 1: Validate Phase**
    - Parse $ARGUMENTS to get phase number
    - Read `.planning/ROADMAP.md` to find the phase entry
    - Extract: phase name, phase goal, phase dependencies, phase requirements list
    - Verify dependencies are met (check for SUMMARY.md files in dependent phase directories -- SUMMARY.md = completion proof pattern)
    - If phase not found or dependencies unmet: show error box (from ui-brand.md) and abort
    - Display banner: `DOC > DISCUSSING PHASE {N}`

    **Step 2: Detect Content Type**
    - Read `.planning/PROJECT.md` for project type (A/B/C/D), language, and standards settings
    - Analyze phase goal text to determine content type(s). Mapping guidance:
      - Keywords "equipment", "module", "EM-", "unit" -> Equipment Modules content
      - Keywords "architecture", "system overview", "hierarchy" -> System Architecture content
      - Keywords "interface", "communication", "protocol", "bus" -> Interfaces content
      - Keywords "HMI", "visualization", "screen", "display" -> HMI content
      - Keywords "safety", "interlock", "SIL", "E-stop" -> Safety content
      - Keywords "foundation", "scope", "terminology" -> Foundation content
      - Keywords "appendix", "signal list", "parameter list" -> Appendices content
    - A phase can have multiple content types (e.g., an equipment module phase has both equipment modules AND their state machines)
    - For Type C/D projects: also read `.planning/BASELINE.md` to understand existing system scope

    **Step 3: Identify Gray Areas**
    This is the core of the adaptation from GSD to FDS. The GSD discuss-phase analyzes what kind of software is being built (visual, CLI, API). The doc version analyzes what kind of FDS content is being written and probes at FULL functional spec depth.

    For each detected content type, generate gray area topics. Do NOT use a fixed list -- derive topics from the phase goal and content type. But use these probing patterns as guidance:

    **Equipment Modules gray areas (probe FULL depth):**
    - Operating parameters: ranges, tolerances, timing, capacities
    - Operating states: state model, entry/exit conditions, transitions
    - Interlocks: conditions, actions, priorities, cross-EM interlocks
    - Failure modes: sensor failure behavior, actuator failure, recovery (automatic vs manual)
    - Manual overrides: which operations, conditions, safety constraints
    - Startup/shutdown sequences: step order, timing, conditions
    - Maintenance mode: behavior, restrictions, access conditions

    **Interfaces gray areas:**
    - Protocol selection: which protocol, why
    - Signal list: direction, data types, update rates
    - Error handling: timeout behavior, retry logic, fallback values
    - Handshake: acknowledgment patterns, data validation
    - Connection failure: detection, alarm, recovery

    **HMI gray areas:**
    - Screen hierarchy: navigation structure, access levels
    - Alarm presentation: grouping, priorities, acknowledgment
    - Trend displays: which signals, retention, scales
    - Manual controls: which operations from HMI, safety constraints

    **Safety gray areas:**
    - Risk categories: which interlocks are safety-critical
    - E-stop behavior: scope (local/zone/plant), recovery sequence
    - SIL levels: if applicable, which functions
    - Fail-safe states: per equipment module

    **Foundation/Architecture gray areas:**
    - Scope boundaries: what is in, what is out
    - Equipment grouping: how EMs are organized
    - Operating modes: plant-level modes and their effect on equipment
    - Terminology: project-specific terms

    **For Type C/D projects:** Frame gray areas as deltas: "The existing system does X (from BASELINE.md). Are you changing this? If so, how?" Focus questions on what is NEW or DIFFERENT, not on re-specifying the existing system.

    **Step 4: Present Gray Areas**
    Display gray areas grouped by topic. Use AskUserQuestion to let the engineer:
    1. See the proposed gray area topics (numbered list, grouped by content type)
    2. Select which topics to discuss (multiSelect or "all")
    3. Mark any topics as "Claude's Discretion" (Claude will make reasonable decisions and document them without asking)

    Present 3-5 gray area topics per content type (not 8-10). Keep it manageable (Pitfall 7 mitigation). If the phase has many areas, present the most impactful ones and offer "any others you want to discuss?"

    **Step 5: Deep-Dive Discussion**
    For each selected gray area topic, conduct a focused discussion:

    1. State what needs to be decided (be specific: "What is the temperature range for sensor TT-101?" not "Tell me about temperatures")
    2. Ask 3-5 targeted questions at functional spec depth
    3. After each question batch, summarize decisions and ask "Anything else for this topic?"
    4. Move to next topic

    **CRITICAL: Probe at FULL functional spec depth.** Do not ask generic questions. Examples:
    - BAD: "What parameters does this pump have?"
    - GOOD: "What is the flow rate range? What happens when flow drops below minimum -- automatic shutdown or alarm only? Is there a startup ramp time?"

    For cross-references to equipment not yet documented: capture the decision and include a flag note in CONTEXT.md: "Verify when [target module] is documented in Phase {X}."

    **Step 6: Capture in CONTEXT.md**
    Read the CONTEXT.md template from `@~/.claude/gsd-docs-industrial/templates/context.md` for structure.

    Generate `CONTEXT.md` in the phase directory (`.planning/phases/{NN}-{name}/{NN}-CONTEXT.md`):
    - `<domain>` section: phase boundary from ROADMAP.md goal. For Type C/D, include delta scope statement.
    - `<decisions>` section: organized by topic (matching gray area topics). Each decision is a concrete, actionable statement. Include "Claude's Discretion" subsection for delegated areas.
    - `<specifics>` section: exact values, preferred approaches, references to existing documentation
    - `<deferred>` section: ideas that belong in other phases

    **Size control (Pitfall 7):** CONTEXT.md MUST stay under 100 lines. If decisions exceed this, prioritize:
    1. Decisions that change what gets written (highest priority)
    2. Specific technical values (capacities, ranges, timing)
    3. General approach notes (lowest priority -- omit or compress if over budget)

    **Step 7: Completion**
    Display banner: `DOC > PHASE {N} DISCUSSED`

    Show summary:
    - Topics discussed (count)
    - Decisions captured (count)
    - Items delegated to Claude's Discretion (count)
    - Deferred items (count)
    - CONTEXT.md location

    Show Next Up block:
    ```
    ## Next Up
    /doc:plan-phase {N}
    ```

    **WORKFLOW RULES:**
    - All user-facing text matches the project language (from PROJECT.md)
    - Use DOC > prefix on all banners (never GSD >)
    - Use AskUserQuestion for structured choices (topic selection, confirmation)
    - Use inline conversation for technical deep-dives (more natural for specification discussions)
    - Never ask about items marked "Claude's Discretion" -- document them with reasonable defaults and note the delegation
    - Cross-references to undocumented equipment: capture and flag, do not block
