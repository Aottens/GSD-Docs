<task type="auto">
  <name>Task 2: Create plan-phase workflow file</name>
  <files>
    gsd-docs-industrial/workflows/plan-phase.md
  </files>
  <action>
    Create `gsd-docs-industrial/workflows/plan-phase.md` -- the detailed execution logic for phase planning. This is the most complex workflow in Phase 2 because it must generate well-structured doc PLAN.md files.

    Target: 400-600 lines. Wrap ALL content in `<workflow>` tags. Use numbered `## Step N:` headings.

    **IMPORTANT DISTINCTION:** This workflow generates DOC PLAN.md files (markdown sections: ## Goal, ## Sections, ## Context, ## Template, ## Standards, ## Verification). It does NOT generate GSD-format plans with `<task type="auto">` XML elements. The doc PLAN.md format is defined in the RESEARCH.md (Example 7) and uses YAML frontmatter (phase, plan, name, type, wave, depends_on, autonomous).

    **Step 1: Parse Arguments and Validate Phase**
    - Parse $ARGUMENTS: extract phase number, detect --gaps flag
    - If --gaps flag detected: jump to Step 8 (Gap Closure Mode)
    - Read `.planning/ROADMAP.md` to find the phase entry
    - Extract: phase name, phase goal, phase requirements, dependencies
    - Verify dependencies met (check for SUMMARY.md files or VERIFICATION.md in dependent phases)
    - If phase not found: show error box and abort
    - Display banner: `DOC > PLANNING PHASE {N}`

    **Step 2: Load Phase Context**
    - Read `.planning/PROJECT.md` for project type, language, standards settings
    - Read `.planning/phases/{NN}-{name}/{NN}-CONTEXT.md` for phase decisions
    - If CONTEXT.md does not exist: show error box suggesting `/doc:discuss-phase {N}` first, then abort
    - Parse CONTEXT.md sections: extract `<decisions>`, `<specifics>`, `<domain>`
    - Identify "Claude's Discretion" items from CONTEXT.md -- these inform plan generation without asking

    **Step 3: Analyze Sections**
    Based on phase goal, CONTEXT.md, and ROADMAP.md, determine what sections need plans:

    **For Equipment Module phases:**
    - Identify each equipment module from CONTEXT.md decisions or ROADMAP phase description
    - Each EM gets one plan (one PLAN.md per equipment module)
    - Determine which subsections apply per EM. Use the configurable subsection list from the equipment module template:
      - Required always: description, operating-states, parameters, interlocks, io-table
      - Include manual-controls if: CONTEXT.md mentions manual operation for this EM
      - Include alarm-list if: CONTEXT.md mentions alarms for this EM
      - Include maintenance-mode if: CONTEXT.md mentions maintenance procedures
      - Include startup-shutdown if: CONTEXT.md mentions complex startup/shutdown sequences
    - If the EM has a state machine: the state machine is part of the EM plan (not a separate plan), referencing the state machine template as an additional template
    - If the phase has an overview/summary section: create a separate plan for it

    **For Interface phases:**
    - One plan per interface (or per interface group if interfaces are closely related)
    - Template: section-interface.md

    **For Foundation/Architecture/HMI/Safety/Appendix phases:**
    - One plan per logical section group
    - Template: varies (or no specific FDS template -- plan describes structure directly)

    **Granularity (Claude's discretion from CONTEXT.md):**
    - If a phase has 1-3 sections: one plan per section
    - If a phase has 4-8 sections: one plan per section or per small group
    - If a phase has 8+ sections: group related sections (e.g., similar equipment types)

    **Step 4: Build Dependency Graph**
    For each identified section/plan:
    1. Record what it cross-references (other EMs, interfaces, shared interlocks)
    2. Record what references IT
    3. Identify overview/summary sections that reference multiple other sections

    **Step 5: Assign Waves**
    Apply dependency-based wave assignment (Claude's discretion from CONTEXT.md):

    - Wave 1: Sections with no cross-references to other sections in this phase (independent)
    - Wave 2: Sections that reference Wave 1 sections (e.g., EM with cross-EM interlocks to Wave 1 EM)
    - Wave 3+: Sections that reference Wave 2 sections (cascading dependencies)
    - Last wave: Overview/summary sections that reference multiple sections

    Rules:
    - If no dependencies exist between sections, put ALL in wave 1
    - Sections within the same EM are NEVER split across plans
    - Cross-references to sections in OTHER phases are flagged for review, not blocked

    **Step 6: Generate PLAN.md Files**
    For each identified section, generate a PLAN.md file in the phase directory.

    File naming: `{NN}-{MM}-PLAN.md` where NN is the zero-padded phase number and MM is the zero-padded plan number within the phase (01, 02, 03...).

    Each PLAN.md follows this format (NOT GSD format -- this is the doc plan format):

    ```markdown
    ---
    phase: {N}
    plan: {MM}
    name: {Section Name}
    type: {equipment-module | state-machine | interface | overview | appendix}
    wave: {W}
    depends_on: [{list of NN-MM plan IDs this plan depends on}]
    autonomous: true
    ---

    # {Section Name}

    ## Goal
    {What this documentation section must achieve -- derived from ROADMAP phase goal
    and CONTEXT.md decisions. Specific to THIS section, not the whole phase.}

    ## Sections
    {Numbered list of subsections to write. For equipment modules, this is the
    selected subsections from the configurable list.}
    1. {Subsection 1}
    2. {Subsection 2}
    ...

    ## Context
    {Relevant decisions extracted from CONTEXT.md for THIS section ONLY.
    Do not dump the entire CONTEXT.md -- extract only what the writer needs.}
    - {Decision 1 relevant to this section}
    - {Decision 2 relevant to this section}

    ## Template
    @~/.claude/gsd-docs-industrial/templates/fds/{template-file}.md

    ## Standards
    {Only if standards are enabled in PROJECT.md. Reference by name, not inline.}
    - PackML: @~/.claude/gsd-docs-industrial/references/standards/packml/
    - ISA-88: @~/.claude/gsd-docs-industrial/references/standards/isa-88/

    ## Writing Rules
    @~/.claude/gsd-docs-industrial/references/writing-guidelines.md

    ## Verification
    - [ ] {Content completeness check}
    - [ ] {Consistency check with CONTEXT.md decisions}
    - [ ] {Table completeness check (all required columns filled)}
    - [ ] {Standards compliance check if applicable}
    - [ ] {Cross-reference accuracy check}
    - [ ] {Language correctness check}
    ```

    **Verification checklist generation (full FDS quality level from CONTEXT.md):**
    - Content completeness: all sections listed in ## Sections are present with substantive content
    - CONTEXT.md consistency: decisions from ## Context are reflected in content (not contradicted)
    - Table completeness: all required table columns have values, no empty cells (except optional columns)
    - Standards compliance (if enabled): PackML state names are exact, ISA-88 terminology is correct
    - Signal naming: follows project tag naming convention from PROJECT.md
    - Cross-references: symbolic references to other sections use correct format
    - Language: configured language used for descriptive text, English for technical terms per CONTEXT.md decisions

    **Step 7: Self-Verify Plans**
    Before completing, run inline verification checks on all generated plans:

    1. **Structure check:** Every plan has all required sections (Goal, Sections, Context, Template, Verification)
    2. **Wave consistency:** No plan depends on a plan in the same or later wave
    3. **No circular dependencies:** Follow depends_on chains -- they must terminate
    4. **CONTEXT.md coverage:** Every decision in CONTEXT.md `<decisions>` section is reflected in at least one plan's ## Context
    5. **Template references valid:** Every ## Template path points to an existing template file (check with Glob)
    6. **Standards conditional:** ## Standards section only present if PROJECT.md has standards enabled
    7. **Naming convention:** All files follow NN-MM-PLAN.md naming

    If any check fails: fix the issue and re-verify. Do not present broken plans to the engineer.

    Display verification results:
    ```
    Self-Verification:
    [PASS] Structure: all plans have required sections
    [PASS] Waves: no dependency violations
    [PASS] Coverage: all CONTEXT.md decisions mapped
    [PASS] Templates: all references valid
    [PASS] Standards: conditional on PROJECT.md settings
    [PASS] Naming: all files follow NN-MM-PLAN.md format
    ```

    **Step 8: Gap Closure Mode (--gaps flag)**
    This step runs ONLY if --gaps flag was detected in Step 1.

    - Check for VERIFICATION.md in the phase directory: `.planning/phases/{NN}-{name}/{NN}-VERIFICATION.md`
    - If not found: show error message: "No VERIFICATION.md found. Run `/doc:verify-phase {N}` first to identify gaps. (Gap closure requires Phase 3's verify-phase command.)"
    - If found: parse gaps from VERIFICATION.md
      - Each gap has: truth (what failed), reason (why), artifacts (which files), missing (what to add/fix)
      - Group related gaps by section/artifact
      - Find next plan number (after existing plans)
      - Generate targeted fix plans with:
        - `## Goal` focused on closing the specific gap
        - `## Context` including the gap description and what the original plan missed
        - `## Verification` focused on the specific truth that must pass
      - Wave 1 for all gap closure plans (they are independent fixes)

    **Step 9: Completion**
    Display banner: `DOC > PHASE {N} PLANNED`

    Show wave summary table:
    ```
    Wave Structure:
    | Wave | Plans | Sections |
    |------|-------|----------|
    | 1    | 01, 02, 03 | EM-100, EM-200, EM-300 |
    | 2    | 04 | EM-400 (depends on EM-100 interlocks) |
    | 3    | 05 | Phase Overview |

    Plans: 5 in 3 waves
    Parallel: Wave 1 (3 plans), Wave 2 (1 plan)
    Sequential: Wave 3 depends on all prior
    ```

    Show Next Up block:
    ```
    ## Next Up
    /doc:write-phase {N}
    ```

    **WORKFLOW RULES:**
    - All user-facing text matches the project language (from PROJECT.md)
    - Use DOC > prefix on all banners
    - Plan files use the DOC PLAN.md format (## Goal, ## Sections, etc.) NOT the GSD format (<task> XML)
    - Standards are ALWAYS referenced by name/path, never inline content
    - Verification checklists are at FULL FDS quality level
    - Each plan extracts ONLY relevant CONTEXT.md decisions (not the full CONTEXT.md)
    - Equipment module subsections are selected based on CONTEXT.md decisions, not always the full set
  </action>
