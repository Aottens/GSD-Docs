  <action>
    Create three FDS section templates in `gsd-docs-industrial/templates/fds/`. These are STRUCTURAL templates -- they define table formats, section headings, and placeholders. They do NOT contain content. The plan-phase command references these templates via @-paths in generated doc PLAN.md files, and the write-phase command (Phase 3) uses them to produce actual FDS content.

    All templates use bilingual column headers with the pattern `{English / Nederlands}` so the writer can select based on the project language setting in PROJECT.md.

    **1. section-equipment-module.md (~80-120 lines)**

    YAML frontmatter:
    ```yaml
    ---
    type: equipment-module
    language: "{LANGUAGE}"
    standards: [packml, isa88]
    subsections:
      required: [description, operating-states, parameters, interlocks, io-table]
      optional: [manual-controls, alarm-list, maintenance-mode, startup-shutdown]
    ---
    ```

    Sections (each with table structure and placeholder values):

    1. `{Description / Beschrijving}` -- free text describing equipment function and operation. Always included.

    2. `Operating States` -- table with columns: State, ID, {Description / Beschrijving}, {Entry Condition / Startconditie}, {Exit Condition / Stopconditie}. Include a note that PackML state names (IDLE, STARTING, EXECUTE, etc.) are always in English per industry convention. Always included.

    3. `{Parameters / Parameters}` -- table with columns: Parameter, {Range / Bereik}, {Unit / Eenheid}, Default, {Description / Beschrijving}. Always included.

    4. `Interlocks` -- table with columns: ID, {Condition / Conditie}, {Action / Actie}, {Priority / Prioriteit}. ID format: IL-{EM}-NN. Always included.

    5. `I/O` -- table with ALL 9 columns from CONTEXT.md: Tag, {Description / Beschrijving}, Type (DI/DO/AI/AO), {Signal Range / Signaalbereik}, {Eng. Unit / Eenheid}, {PLC Address / PLC Adres}, {Fail-safe State / Veilige Stand}, {Alarm Limits / Alarmgrenzen}, {Scaling / Schaling}. Always included.

    6. `{Manual Controls / Handmatige Bediening}` -- table: {Control / Bediening}, {Function / Functie}, {Condition / Voorwaarde}. OPTIONAL: marked with HTML comment `<!-- OPTIONAL: include if equipment has manual operation -->`.

    7. `{Alarm List / Alarmlijst}` -- table: ID, {Description / Beschrijving}, {Category / Categorie}, {Action / Actie}. ID format: AL-{EM}-NN. OPTIONAL.

    8. `{Maintenance Mode / Onderhoudsmodus}` -- free text block for maintenance mode behavior. OPTIONAL.

    9. `{Startup/Shutdown Sequence / Opstart/Afsluitprocedure}` -- step-by-step sequence description. OPTIONAL.

    Each section uses heading pattern `#### [N.M.X]` where N.M is the parent section number (filled by writer) and X is the subsection index.

    **2. section-state-machine.md (~50-70 lines)**

    YAML frontmatter:
    ```yaml
    ---
    type: state-machine
    language: "{LANGUAGE}"
    standards: [packml]
    ---
    ```

    Sections:
    1. `State Diagram` -- contains a Mermaid `stateDiagram-v2` code block with a standard PackML state machine as the default template (IDLE -> STARTING -> EXECUTE -> COMPLETING -> COMPLETE -> RESETTING -> IDLE, plus HOLDING/HELD/UNHOLDING branch, plus STOPPING/STOPPED and ABORTING/ABORTED/CLEARING from any state). Include `SC` abbreviation note (State Complete = internal transition trigger). The writer adapts this to the specific unit.

    2. `{State Descriptions / Toestandsbeschrijvingen}` -- table: State, {Description / Beschrijving}, {Actions / Acties}

    3. `{Transitions / Overgangen}` -- table: {From / Van}, {To / Naar}, {Trigger / Trigger}, {Condition / Conditie}, {Action / Actie}

    **3. section-interface.md (~50-70 lines)**

    YAML frontmatter:
    ```yaml
    ---
    type: interface
    language: "{LANGUAGE}"
    ---
    ```

    Sections:
    1. `{Overview / Overzicht}` -- key-value table: Type (Modbus TCP / Profinet / OPC-UA / Hardwired / ...), {Direction / Richting}, {Counterpart / Tegenpartij}, {Update Rate / Verversingssnelheid}

    2. `{Signals / Signalen}` -- table: #, {Name / Naam}, Type (BOOL/INT/REAL), {Description / Beschrijving}, {Direction / Richting} (IN/OUT), {Range / Bereik}

    3. `{Protocol Details / Protocoldetails}` -- bullet list format: Polling interval, Timeout, {Error handling / Foutafhandeling}, {Handshake / Handshake}. Note in template that Claude structures this differently based on interface type (fieldbus vs Ethernet vs hardwired) per CONTEXT.md "Claude's discretion" decision.

    **IMPORTANT: These templates are structure definitions, not content.** Every data value is a `{PLACEHOLDER}` that the writer fills. Table rows show the format with one example row of placeholders. The writer generates actual rows from CONTEXT.md and PLAN.md decisions.

  <action>
    Create the CONTEXT.md template in `gsd-docs-industrial/templates/context.md`. This template defines the structure that `/doc:discuss-phase` uses to capture phase discussion decisions. It is adapted from the GSD context.md template (C:/Users/Aotte/.claude/get-shit-done/templates/context.md) for the FDS domain.

    Structure (~60-80 lines):

    ```markdown
    # Phase {N}: {Phase Name} - Context

    **Gathered:** {date}
    **Status:** Ready for planning

    <domain>
    ## Phase Boundary

    {What this phase delivers -- extracted from ROADMAP.md phase goal. This is the fixed boundary.}
    {For Type C/D: "Delta from BASELINE.md -- only changes are in scope."}

    </domain>

    <decisions>
    ## Implementation Decisions

    ### {Topic 1 from gray area discussion}
    - {Specific technical decision with concrete values}
    - {Failure mode or edge case handling}
    - {Cross-reference to other equipment if applicable -- flag for review}

    ### {Topic 2}
    - {Decision}

    ### Claude's Discretion
    - {Areas explicitly delegated to Claude -- documented but not asked}

    </decisions>

    <specifics>
    ## Specific Ideas

    {Technical specifics from discussion: exact values, preferred approaches,
    references to existing documentation or equipment.
    Keep focused -- specifics that change what gets written.}

    </specifics>

    <deferred>
    ## Deferred Ideas

    {Ideas that belong in other phases -- captured so they are not lost,
    but explicitly out of scope for this phase.}

    </deferred>

    ---
    *Phase: {NN}-{name}*
    *Context gathered: {date}*
    ```

    Key adaptations from GSD template:
    1. Phase Boundary includes explicit Type C/D delta instruction
    2. Decisions section uses topic headings from gray area discussion (equipment names, interface names, etc.) rather than generic categories
    3. "Claude's Discretion" subsection is mandatory -- always present to document what was delegated
    4. Specifics section emphasizes concrete values (temperatures, capacities, timing) not vague requirements
    5. XML section tags (`<domain>`, `<decisions>`, `<specifics>`, `<deferred>`) for semantic grouping -- matching the pattern established in CLAUDE-CONTEXT.md

    Add a comment block at the top of the template (before the markdown content) explaining:
    - This template is used by the discuss-phase workflow
    - The workflow reads this template for structure, then fills it with actual discussion content
    - Size guideline: CONTEXT.md should stay under 100 lines to avoid overwhelming writer subagents (Pitfall 7 mitigation)
    - Priority: decisions that change implementation > specific values > general approach notes
