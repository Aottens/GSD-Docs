<task type="auto">
  <name>Task 2: Create new-fds workflow file (full execution logic)</name>
  <files>~/.claude/gsd-docs-industrial/workflows/new-fds.md</files>
  <action>
    Create `~/.claude/gsd-docs-industrial/workflows/new-fds.md` — the detailed execution logic.

    This is the LARGEST file in Phase 1. It contains the complete step-by-step process that Claude follows when the user runs `/doc:new-fds`. Structure it as follows:

    **Step 1: Prerequisites Check**
    - Check git repo exists: `[ -d .git ] || [ -f .git ]`. If not, run `git init`.
    - Check .planning/ does not exist: `[ -d .planning ]`. If it exists, show error box (from ui-brand.md pattern) and suggest `/doc:status` instead. Abort.
    - Check for optional tools: `pandoc --version 2>/dev/null` and `mmdc --version 2>/dev/null`. If missing, show info (not error) that these are needed only for /doc:export later.

    **Step 2: Language Selection**
    ALWAYS ask language FIRST (from CONTEXT.md decision: "Always ask during project creation, no default assumed").
    Use AskUserQuestion:
    - header: "Language / Taal"
    - question: "In which language should the FDS documentation be written? / In welke taal moet de FDS documentatie geschreven worden?"
    - options:
      - "Nederlands (Dutch)" — FDS output and prompts in Dutch
      - "English" — FDS output and prompts in English

    Store the choice. From this point forward, ALL user-facing text (prompts, questions, banners, summaries) must match the chosen language. Include bilingual instruction for this: "If language is 'nl', use Dutch for all AskUserQuestion prompts, banners, and output text. If 'en', use English."

    **Step 3: Classification — Stage 1 (Type Determination)**
    Display stage banner: `DOC > CLASSIFYING` (or Dutch equivalent if nl)

    Ask classification questions using AskUserQuestion. The flow from SPECIFICATION section 3.1:

    Question 1: "New system or modification of existing?" / "Nieuw systeem of modificatie van bestaand?"
    - Options: "New / Nieuw", "Modification / Modificatie"

    If NEW:
    Question 2: "Are formal standards required (PackML, ISA-88)?" / "Zijn formele standaarden vereist (PackML, ISA-88)?"
    - Options: "Yes, formal standards / Ja, formele standaarden" -> Type A, "No, flexible approach / Nee, flexibele aanpak" -> Type B

    If MODIFICATION:
    Question 2: "What is the scope of the modification?" / "Wat is de omvang van de modificatie?"
    - Options: "Substantial (multiple EMs, new functionality) / Substantieel (meerdere EMs, nieuwe functionaliteit)" -> Type C, "Limited (single change, TWN) / Beperkt (enkele wijziging, TWN)" -> Type D

    **Present suggested type with confirm/override:**
    Show the suggested type with description. Use AskUserQuestion:
    - header: "Project Classification / Projectclassificatie"
    - question: "Based on your answers, this is a Type {X} project: {description}. Confirm or select a different type."
    - options: "Type A — {desc}", "Type B — {desc}", "Type C — {desc}", "Type D — {desc}"
    - Pre-select the suggested type

    **Override warning:** If the user picks a type that differs from the suggestion, show a brief warning using the checkpoint box pattern from ui-brand.md: "Warning: Your answers suggest Type {X} but you selected Type {Y}. This may result in {consequence}. Proceeding with Type {Y}." Then continue.

    **Step 4: Classification — Stage 2 (Metadata Gathering)**
    Ask metadata in a focused sequence. Use AskUserQuestion for structured questions, inline for free-text:

    1. Project name (inline question, free-text)
    2. Client name (inline question, free-text)
    3. Location/site (inline question, free-text, optional)
    4. Estimated equipment modules count (inline question — helps calibrate scope, informs dynamic ROADMAP evolution later)

    For Type A only:
    5. Standards configuration — Use AskUserQuestion with multiSelect:
       - "PackML (state model, operating modes)"
       - "ISA-88 (equipment hierarchy, terminology)"
       - Both are enabled by default for Type A, but user can deselect

    For Type C/D only:
    6. Existing system information — Ask about:
       - Existing system name/version (inline)
       - Do existing documents exist? (AskUserQuestion: "Yes, I have existing FDS/documentation" / "No, we'll describe from scratch")
       - If yes: "Please place existing documents in the intake/ folder. We'll reference them for the baseline."
       - If no: Proceed (BASELINE.md will be populated through structured interview during scaffolding)

    **Step 5: Scaffold Workspace**
    Display stage banner: `DOC > SCAFFOLDING`

    Create directory structure:
    ```bash
    mkdir -p .planning/phases
    mkdir -p intake
    mkdir -p output
    mkdir -p diagrams/mermaid diagrams/rendered
    mkdir -p export
    ```

    **Generate artifacts (this is where templates are consumed):**

    For each artifact, READ the template from ~/.claude/gsd-docs-industrial/templates/ and FILL placeholders with the gathered metadata. Do NOT just copy templates — produce populated, project-specific content.

    1. **PROJECT.md** — Fill from templates/project.md:
       - Replace all {PLACEHOLDERS} with actual values
       - YAML config block with actual project_type, standards, language
       - What This Is: synthesize from project name + type
       - Scope: derive from type (new vs modification, full vs delta)

    2. **ROADMAP.md** — Load the correct roadmap template based on type:
       - Type A: templates/roadmap/type-a-nieuwbouw-standaard.md
       - Type B: templates/roadmap/type-b-nieuwbouw-flex.md
       - Type C: templates/roadmap/type-c-modificatie.md
       - Type D: templates/roadmap/type-d-twn.md
       - Replace {PLACEHOLDERS} with actual values
       - Add phase directory names (01-foundation, 02-system-architecture, etc.)
       - Include success criteria per phase (2-3 observable outcomes)
       - Include dependency chain between phases

    3. **REQUIREMENTS.md** — Fill from templates/requirements.md:
       - Category structure based on project type:
         - Type A: Foundation, System Architecture, Equipment Modules, Control & HMI, Interfaces & Safety, Appendices
         - Type B: Foundation, System Overview, Functional Units, HMI & Interfaces, Appendices
         - Type C: Scope & Baseline, Delta Functional, Delta HMI, Verification
         - Type D: Change, Implementation
       - Each category gets a placeholder: "Requirements to be identified during /doc:discuss-phase"
       - Traceability table linking categories to ROADMAP phases

    4. **STATE.md** — Fill from templates/state.md:
       - Phase: 1, Status: pending, Progress: 0%
       - Phase table from ROADMAP (all pending)
       - Decisions: include classification decision (Type {X} selected)
       - Next step: /doc:discuss-phase 1

    5. **config.json** — Create with:
       ```json
       {
         "project_type": "{TYPE}",
         "language": "{LANGUAGE}",
         "git_integration": true,
         "standards": {
           "packml": {PACKML_BOOL},
           "isa88": {ISA88_BOOL}
         }
       }
       ```

    6. **BASELINE.md** (Type C/D only) — Fill from templates/baseline.md:
       - If user said existing documents exist: include import instructions
       - If user said no documents: populate with structured interview results from Step 4
       - The INSTRUCTIE block is always present (bilingual)

    **Step 6: Auto-Commit**
    Stage and commit all scaffolded files with descriptive message:
    ```bash
    git add .planning/ intake/ output/ diagrams/ export/
    git commit -m "docs: initialize {PROJECT_NAME} (Type {TYPE})

    Project type: Type {TYPE} - {type_description}
    Language: {LANGUAGE}
    Phases: {PHASE_COUNT}
    Standards: {standards_summary}

    Artifacts: PROJECT.md, REQUIREMENTS.md, ROADMAP.md, STATE.md{, BASELINE.md}"
    ```

    **Step 7: Completion Summary**
    Display banner: `DOC > PROJECT INITIALIZED`

    Show completion summary matching the language:
    - Project name, type, client, language
    - Table of created artifacts with locations
    - Phase overview table (from ROADMAP)
    - File tree of .planning/ directory

    Show Next Up block:
    ```
    ## Next Up
    **Phase 1: {Phase Name}** — {Goal from ROADMAP}
    `/doc:discuss-phase 1`
    ```

    **CRITICAL WORKFLOW RULES:**
    - The workflow file wraps ALL instructions in `<workflow>` tags
    - Each step is a numbered `## Step N:` heading
    - Include bilingual instructions where the language affects user-facing text
    - Include the SUMMARY.md = completion proof pattern: document that SUMMARY.md existence (not STATE.md) is the truth for downstream commands. This is established here (Pitfall 4 mitigation) even though the pattern is used in later phases.
    - Include note about template composition: the workflow READS templates and FILLS them, it does not copy them verbatim
  </action>
